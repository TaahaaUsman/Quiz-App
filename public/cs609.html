Windows Operating System Concepts

Topic 1: Windows Operating System

Software can be categorized into two main types: System Software and Application Software. System Software is concerned with the system or its resources while Application Software is related to some application. An Operating System (OS) is one of the prime examples of System software that acts as a manager for the System's resources. It arbitrates and schedules the resources among the processes to avoid any kind of conflicts.

There are varieties of Operating Systems available in the market, but Windows, developed by Microsoft, is one of the most important. It is widely used inside PCs, Laptops, enterprise servers, handheld devices, and cell phones etc.

Core Features of Windows Operating System

  1. Memory Management: Memory Management is a vital or key feature of the Windows Operating System. The OS is responsible for managing primary as well as secondary memory. Virtual address space on secondary memory is also managed by the OS to transfer data between primary and secondary storage and accommodate large processes in small memory spaces. All kinds of services and related data structures are supported by Operating Systems to efficiently manage the Virtual memory space.
  2. File systems: Windows OS manages all kinds of files and folders on disk in hierarchical form. A large file naming space of 255 characters is supported by the OS. A number of APIs are provided by the OS to access files in both Sequential and Random modes.
  3. Processors: Windows Operating System provides support for multiprocessors and multi-core systems. It also arbitrates among the cores or processors to efficiently divide and allocate computational tasks to them.
  4. Resource naming and location: In Windows OS, resources like processes or devices are treated as objects, and each object is assigned a unique name to identify, locate, and access it.
  5. Multitasking: Windows OS supports multitasking. It manages processes, threads, and other independent units, and their asynchronous execution. Tasks can be preempted and scheduled according to dynamically calculated priorities.
  6. Communication and Synchronization: The Windows OS provides constructs to manage inter-process communication and synchronization within a computer or networked computers.
  7. Security and Protection: The Windows OS has a strong security mechanism to protect resources from illegal and accidental access. A user cannot access other user data without assigning privileges.

Topic 2: Windows Evolution

Windows exists in several versions. New versions of Windows are introduced from time to time. New APIs are often included in new versions to improve or extend functionalities. The following major themes or features are considered while developing a new version:

  • Scalability: A new version of Windows OS runs on different platforms including PCs, enterprise servers, multiprocessing systems, mobiles, and systems having large memory space.
  • Performance: A newer version of Windows typically improves performance compared to previous versions.
  • Integration: A newer version must integrate with new technologies like web services, .NET technologies, multimedia etc.
  • Ease of use: Certain new APIs and improved Graphical Interfaces (GI) in the new version can ensure ease of use.
  • Enhanced API: Introducing new APIs or enhancements in existing APIs is often a main theme of new versions.

Disk Operating System (DOS)

In the 1980s, Microsoft Disk Operating System (MS-DOS) was used inside IBM PCs incorporating Intel Processors. It was a text-based and command-line operating system. It was a single-user OS. Its filing system was based on FAT (File Allocation Table) and was able to access up to a maximum of 4GB files.

Windows History

Keeping the demand for a graphical user interface (GUI), Microsoft developed its first widely popular version, Windows 3.1. In this version, the DOS Kernel and FAT based file systems were used.

After that, in the 1990s, certain new versions of Windows named Windows 95, 97 (Windows 95 OSR2), and 98 were introduced, supporting the 32-bit architecture of Intel's processors.

Later on, Windows NT versions were introduced, supporting a file system based on new technology called NTFS (New Technology File System). Its security and file system were better than the previous versions.

Windows Server 2008 OS was developed for professional use to manage enterprise and server applications. Support for multi-core technology and 64-bit applications was provided in this OS. Other Windows versions supporting 32-bit, 64-bit architecture, multi-core, and multiprocessing were also introduced, including Windows XP, Windows Vista, Windows 7, Windows 8, and Windows 10, and later versions.

Topic 3: Windows Market Role

Several competitors of Windows OS, like UNIX, Linux, etc., exist in the market; however, Windows has its own unique status. It holds several significant advantages over other operating systems, particularly in the desktop market.

Above 90% of PCs are based on Intel's processors (or compatible architectures), and Windows is the most common OS for these PCs. In the world of desktops, the most dominant OS is Microsoft Windows, enjoying a market share often cited above 80%. Windows is not confined to the desktop; it also has support for diverse platforms including multi-core, multiprocessing, servers, and mobiles etc.

Due to its dominant role, a vast ecosystem of applications and software development tools are available that can easily integrate with Windows OS and develop Windows applications ranging from small scale to enterprise level.

One of the key features of Windows OS is its rich GUI that makes its use very convenient. This interface can be easily customized according to the local setup. The size, color, and visibility of graphical interface objects can also be changed by the user.

Compared to other operating systems, certain modern features exist in Windows, which encourages many developers to create applications targeting the huge market of Windows users.

Topic 4: Windows, Standards and Open Systems

Open Source Software is software that is publicly available with its source code to use, modify, and distribute with original rights. It is typically developed by a community rather than a single company or vendor. In contrast, proprietary software is copyrighted and only available to use under a license.

Windows Operating system is proprietary and copyrighted software of Microsoft corporation. It is provided for use only under a License agreement. Without purchasing a license, its use is illegal and is an act of copyright infringement.

Being a closed system, Windows has the following strengths:

  • As Windows components are provided and updated only by a single vendor, its implementation remains relatively uniform throughout the world.
  • Extensions in Window components or APIs are generally vendor-specific, limiting non-standard extensions except for platform differences.

Windows also supports various types of hardware platforms, similar to open systems in terms of hardware compatibility.

Interoperability of Windows

Windows provides interoperability with non-Windows components:

  • Windows OS provides support for the Standard C and C++ libraries. We can install and use standard C/C++ compilers on Windows systems.
  • Socket is a resource required for interfacing when two computers are interconnected. Windows supports sockets to communicate among devices having different computer architectures and access to TCP/IP and other networking protocols.
  • It also supports the Remote Procedure Calls (RPCs) architecture to call remote functions in distributed client-server based applications.
  • Windows also supports the X Window System, which is open-source, cross-platform software providing a GUI in a distributed network environment (often used via third-party implementations).

Topic 5: Windows Principles

  • In Windows OS, all system resources including processes, threads, memory, pipes, DLLs, etc., are represented by objects which are identified and referenced by a handle. These objects cannot be directly accessed. If an application attempts to access these objects directly, Windows throws an appropriate exception. The only way to access and operate on these objects is through a set of Application Programming Interfaces (APIs) provided by Windows. Several APIs can be related to a single object to manipulate it differently.
  • A long list of parameters is associated with each API, where each parameter has its own significance, but often only a few parameters need to be specified for a particular operation.
  • To perform the tasks of multitasking and multi-threading efficiently, Windows provides a number of synchronization constructs to arbitrate among resources.
  • The names of Windows APIs are often long and descriptive for proper and convenient use.
  • Some pre-defined data types required for Windows APIs include:
    • BOOL (for storing a single logical value)
    • HANDLE (a handle for an object)
    • LPTSTR (a pointer to a string, generic character type)
    • DWORD (a 32-bit unsigned integer)
  • Windows Data types often abstract away the pointer operator (*) in their definitions, although the underlying mechanism involves pointers.
  • Some lowercase prefix letters with variable names are used to identify the type of variable. This notation is called Hungarian notation. For example, in the variable name lpszFilename, 'lpsz' is Hungarian notation representing a long pointer to a zero-terminated string. (Note: Use of Hungarian notation is less common in modern C++ development but prevalent in the core Win32 API).
  • windows.h is a header file including prototypes and data types for most core Windows APIs.

Topic 6: 32-Bit and 64-Bit Source Code Portability

Windows keeps two versions of many APIs, one for 32-bit and another for 64-bit. A 32-bit code can run on 64-bit hardware (through WoW64 - Windows on Windows 64-bit subsystem) but will be unable to exploit some features of 64-bit, like accessing large disk space (> 2TB with certain APIs), using large pointers (> 4GB memory address space), or performing native 64-bit operations efficiently.

Latest versions of Windows support both 32 and 64-bit architectures by keeping two versions of each relevant API.

Interoperability of 32 and 64-bit:

A single source code can often be built for both 32-bit and 64-bit versions using generic data types and conditional compilation. The decision of whether the executable code generated by the compiler is 32 or 64-bit depends on the compiler settings and configuration. Furthermore, deciding which version of an API is used (e.g., the 'A' ANSI version or 'W' Unicode version, and implicitly the 32/64-bit variant) is also based on the compiler's configuration and preprocessor definitions (like UNICODE).

A 32-bit application can run on 64-bit hardware successfully but will be limited by the 32-bit address space and potentially other 32-bit constraints.

Source code developed specifically for a 64-bit architecture (e.g., assuming pointer sizes) might not run correctly or compile on a 32-bit machine without modification. Re-compilation with appropriate settings is required to generate a 32-bit executable code from portable source.

Topic 7: When to use Standard C Library for File Operations

Windows provides a set of built-in APIs (like CreateFile, ReadFile, WriteFile) to perform I/O operations. A specific API with relevant parameters is invoked for the desired resource and I/O operation.

Similarly, standard C/C++ library functions are available to perform I/O operations. For example, fopen(), fclose(), fread(), fwrite() etc., are standard C functions that can be used to perform I/O operations related to files.

Differences between Windows APIs and Standard Functions

Standard C functions can be used inside source code to run on the Windows platform because Windows provides low-level system calls that support the C/C++ runtime libraries for I/O operations.

If it is required to run a program on multiple platforms (cross-platform), then it is generally preferred to use the standard C/C++ library functions inside the source code for better portability. However, in this case, advanced Windows-specific features like file locking types, synchronization modes, asynchronous I/O, inter-process communication specifics tied to file handles, etc., might not be directly accessible or achievable using only standard functions.

In cases where portability is not the primary focus and it's required to avail the advanced Windows features, then it is preferred to use the Windows APIs directly.

Topic 8: A Simple File Copy Program Using Standard C Library

This program demonstrates copying a file using standard C library functions.

// cpC. Basic File Copy Program: C Library Implementation
// Copy File1 to File2
#include 
#include 

#define BUF_SIZE 256

int main(int argc, char *argv[]) {
    FILE *inFile, *outFile;
    char rec[BUF_SIZE];
    size_t bytesIn, bytesOut;

    if(argc != 3) {
        printf("Usage: cp file1 file2\n");
        return 1;
    }

    inFile = fopen(argv[1], "rb"); // Open source file in read binary mode
    if(inFile == NULL) {
        perror(argv[1]); // Report error related to source file
        return 2;
    }

    outFile = fopen(argv[2], "wb"); // Open destination file in write binary mode
    if(outFile == NULL) {
        perror(argv[2]); // Report error related to destination file
        fclose(inFile); // Close the source file before exiting
        return 3;
    }

    /* Process the input file a record at a time */
    while((bytesIn = fread(rec, 1, BUF_SIZE, inFile)) > 0) {
        bytesOut = fwrite(rec, 1, bytesIn, outFile);
        if(bytesOut != bytesIn) {
            perror("Fatal write error");
            fclose(inFile);
            fclose(outFile);
            return 4;
        }
    }

    fclose(inFile);
    fclose(outFile);
    return 0; // Success
}

This program copies one file to another using C standard functions. A buffer of size 256 bytes is used, and chunks of the file are copied one by one. The source file is opened in read binary mode ("rb") and the destination file in write binary mode ("wb") using the C fopen() function. If both are successfully opened, the file is read inside a loop chunk by chunk using fread() and written to the destination file using fwrite(). After the loop finishes (or an error occurs), both files are closed using fclose().

Topic 9: A Simple File Copy Program Using Windows APIs

This program achieves the same file copy task but uses the native Windows API functions.

// cpW. Basic File Copy Program: Windows Implementation
// Copy File1 to File2
#include 
#include 
// #include  // Often included via windows.h

#define BUF_SIZE 16384 // Larger buffer size often more efficient

int main(int argc, char *argv[]) {
    HANDLE hIn, hOut;
    DWORD nIn, nOut;
    CHAR buffer[BUF_SIZE];
    WCHAR wszFile1[MAX_PATH]; // Use WCHAR for Unicode filenames
    WCHAR wszFile2[MAX_PATH];
    INT iLen1, iLen2;

    if(argc != 3) {
        fprintf(stderr, "Usage: cp file1 file2\n");
        return 1;
    }

    // Convert command line arguments (ANSI) to Unicode (WCHAR) for API calls
    iLen1 = MultiByteToWideChar(CP_ACP, 0, argv[1], -1, wszFile1, MAX_PATH);
    iLen2 = MultiByteToWideChar(CP_ACP, 0, argv[2], -1, wszFile2, MAX_PATH);
    if (iLen1 == 0 || iLen2 == 0) {
         fprintf(stderr, "Filename conversion error: %x\n", GetLastError());
         return 1; // Indicate error
    }


    hIn = CreateFileW(wszFile1,        // Use CreateFileW for Unicode
                       GENERIC_READ,
                       FILE_SHARE_READ, // Allow others to read while open
                       NULL,            // Default security attributes
                       OPEN_EXISTING,   // Open only if it exists
                       FILE_ATTRIBUTE_NORMAL,
                       NULL);           // No template file

    if (hIn == INVALID_HANDLE_VALUE) {
        fprintf(stderr, "Cannot open input file. Error: %x\n", GetLastError());
        return 2;
    }

    hOut = CreateFileW(wszFile2,       // Use CreateFileW
                        GENERIC_WRITE,
                        0,                 // No sharing for write
                        NULL,
                        CREATE_ALWAYS,   // Create new or overwrite existing
                        FILE_ATTRIBUTE_NORMAL,
                        NULL);

    if (hOut == INVALID_HANDLE_VALUE) {
        fprintf(stderr, "Cannot open output file. Error: %x\n", GetLastError());
        CloseHandle(hIn); // Close the input handle
        return 3;
    }

    // Read from input and write to output
    while(ReadFile(hIn, buffer, BUF_SIZE, &nIn, NULL) && nIn > 0) {
        WriteFile(hOut, buffer, nIn, &nOut, NULL);
        if (nIn != nOut) {
            fprintf(stderr, "Fatal write error: %x\n", GetLastError());
            CloseHandle(hIn);
            CloseHandle(hOut);
            return 4;
        }
    }
     // Check for ReadFile error after the loop
    if (GetLastError() != ERROR_SUCCESS && GetLastError() != ERROR_HANDLE_EOF) {
         fprintf(stderr, "Fatal read error: %x\n", GetLastError());
         CloseHandle(hIn);
         CloseHandle(hOut);
         return 5;
    }


    CloseHandle(hIn);
    CloseHandle(hOut);
    return 0; // Success
}

In this program, Windows APIs are used instead of C standard functions. In the main program, capitalized words like HANDLE, DWORD, CHAR (though WCHAR is used for filenames), LPWSTR (implicitly via WCHAR*), and INT are Windows data types. As the file paths given in command line parameters are typically in ASCII/Multibyte format, they are first converted to Unicode (WCHAR strings) using the MultiByteToWideChar() function, as modern Windows APIs primarily operate with Unicode strings (the 'W' versions like CreateFileW).

After that, a file is opened for reading purposes using CreateFileW, and its handle is stored in hIn. Similarly, another file is created (or overwritten) for writing purposes, and its handle is stored in hOut. If files are successfully opened, the source file is read in a loop using ReadFile and written to the destination file using WriteFile. At the end, both handles for the files are closed using CloseHandle.

Topic 10: A Simple File Copy Program Using Windows Convenience Function

Numerous Windows functions are used to perform various tasks at a low level. However, Windows also has a set of Convenience functions that combine several functions to perform a common task. In most cases, these functions can improve performance or simplify code because several steps are performed internally by a single function call.

For example, CopyFile() (or its Unicode variant CopyFileW) is a convenience function that replaces the algorithms used for creating, opening, reading, and writing one file to another.

        using convenience function CopyFile() */

        fprintf(stderr, "Usage: cp file1 file2\n");
        return 1;
    }

    iLen1 = MultiByteToWideChar(CP_ACP, 0, argv[1], -1, wszFile1, MAX_PATH);
    iLen2 = MultiByteToWideChar(CP_ACP, 0, argv[2], -1, wszFile2, MAX_PATH);
     if (iLen1 == 0 || iLen2 == 0) {
         fprintf(stderr, "Filename conversion error: %x\n", GetLastError());
         return 1;
    }


    // Use the CopyFileW function (Unicode version)
    // The last parameter (BOOL bFailIfExists) is FALSE, meaning overwrite if exists.
    if (!CopyFileW(wszFile1, wszFile2, FALSE)) {
        fprintf(stderr, "CopyFile Error: %x\n", GetLastError());
        return 2; // Return non-zero on error
    }

    printf("File copied successfully.\n"); // Optional success message
    return 0; // Success
}

This version is significantly simpler. After converting filenames to Unicode, it makes a single call to CopyFileW. The third parameter, FALSE, indicates that if the destination file (wszFile2) already exists, it should be overwritten. If it were TRUE, the function would fail if the destination file existed. Error checking is done by inspecting the return value of CopyFileW and using GetLastError() if it fails.

Lecture 11: Windows File System

Windows supports various file systems.

NT File System (NTFS)

NTFS is the standard modern file system supported by Windows. Its main features include:

  • Security: Access Control Lists (ACLs) allow fine-grained permissions; one user generally cannot access another user's data without privileges.
  • Fault tolerance: Uses journaling to maintain file system integrity. Can recover from certain types of disk errors. (Note: Does not inherently mean file content copies are maintained without features like Shadow Copies or user backups).
  • Encryption: Encrypting File System (EFS) allows transparent encryption/decryption of files tied to user accounts. (BitLocker provides full disk encryption).
  • Compression: Supports transparent file compression to save disk space.
  • Large File Sizes: Supports very large files and volumes (theoretically up to 2^64 bytes, practically limited by OS).
  • Features like sparse files, disk quotas, hard links, symbolic links, volume mount points.

File Allocation Tables (FAT)

Older file systems also supported by Windows for compatibility (especially with removable media):

  • FAT12, FAT16: Early versions with significant limitations on file size and volume size.
  • FAT32: An improvement, supporting larger volumes (up to 2TB officially by Windows tools, technically larger) and files up to 4GB. Lacks the security and robustness features of NTFS.
  • exFAT: An evolution of FAT, designed for flash drives, supporting larger files and volumes than FAT32, but still lacking features like journaling and granular permissions.

Compact Disk File System (CDFS)

  • Supports reading from CD-ROMs, following the ISO 9660 standard.

Universal Disk Format (UDF)

  • An optical media file system standard, commonly used for DVDs and Blu-rays. Windows supports reading and sometimes writing UDF.

(Note: LFS - Live File System was a specific implementation related to writing to optical media in packets, often using UDF).

Lecture 12: File Naming Conventions

Windows OS supports a number of File Systems. Each file system has its own mechanism and potential limitations for naming files.

Certain Limitations for File Naming (Primarily NTFS/FAT on Windows)

  • Drive letters like A:, B:, C: etc., are used to represent local drives. Network drives are often mapped to higher letters like N:, K:, L: etc.
  • A double slash (\\) at the start of a path represents a remote resource (UNC path, e.g., \\server\share).
  • Forward slash (/) or backslash (\) is used as a path name separator (Windows primarily uses backslash, but often accepts forward slash in APIs).
  • The first 31 ASCII characters (control characters, ASCII 0-31) cannot be used in file names.
  • Special symbols like \, /, :, *, ?, ", <, >, | cannot be used in filenames.
  • File and directory names are generally case insensitive but case preserving (e.g., MyFile.txt and myfile.txt refer to the same file, but the original casing is stored).
  • File name and extensions are typically separated by a dot (.).
  • The maximum size for a file name component (e.g., between backslashes) is typically 255 characters (Unicode characters). The maximum path length is traditionally around 260 characters (MAX_PATH), although newer Windows versions support much longer paths (up to ~32,767 characters) if enabled and using specific API conventions (e.g., prefixing with \\?\).
  • File extensions (part after the last dot) are conventionally 2-4 characters but can be longer.
  • A single dot (.) represents the current directory. A double dot (..) represents the parent directory (one step back up).

Lecture 13: Creating and Opening Files (Using Windows API)

The CreateFile() API

The CreateFile() API (or its Unicode variant CreateFileW and ANSI variant CreateFileA) is used to create a new file or open an existing file or device. Its return type is a HANDLE to the opened object in case of success, or INVALID_HANDLE_VALUE on failure. The parameters are:

  • lpFileName: A pointer to a string that specifies the name of the file or device to be created or opened.
  • dwDesiredAccess: A DWORD value that specifies the type of access to the object. Common values include:
    • GENERIC_READ: Read access.
    • GENERIC_WRITE: Write access.
    • Can be combined (e.g., GENERIC_READ | GENERIC_WRITE).
  • dwShareMode: Specifies how the file or device can be shared by other processes.
    • 0: Prevents subsequent open operations on the object that request conflicting access.
    • FILE_SHARE_READ: Allows subsequent open operations for reading.
    • FILE_SHARE_WRITE: Allows subsequent open operations for writing.
    • FILE_SHARE_DELETE: Allows subsequent open operations for deleting. (Can be combined).
  • lpSecurityAttributes: A pointer to a SECURITY_ATTRIBUTES structure that controls whether the returned handle can be inherited by child processes and specifies security descriptor. Often NULL for default attributes.
  • dwCreationDisposition: Specifies the action to take on files that exist and do not exist.
    • CREATE_NEW: Creates a new file, only if it does not already exist. Fails if it exists.
    • CREATE_ALWAYS: Creates a new file, always. Overwrites the file if it exists.
    • OPEN_EXISTING: Opens a file or device, only if it exists. Fails if it does not exist.
    • OPEN_ALWAYS: Opens a file, always. Creates the file if it does not exist.
    • TRUNCATE_EXISTING: Opens a file and truncates it so that its size is zero bytes, only if it exists. Fails if it does not exist. Requires GENERIC_WRITE access.
  • dwFlagsAndAttributes: Specifies the file attributes and flags. Common flags include FILE_ATTRIBUTE_NORMAL, FILE_ATTRIBUTE_READONLY, FILE_FLAG_OVERLAPPED (for asynchronous I/O), FILE_FLAG_WRITE_THROUGH, FILE_FLAG_DELETE_ON_CLOSE.
  • hTemplateFile: A handle to a template file with the GENERIC_READ access right. Specifies attributes and extended attributes for the file being created. Often NULL.

Lecture 14: Reading/Writing a File

The ReadFile() API

The ReadFile() API is used to read data from a file (or other input device like a pipe) into a buffer.

Syntax:

BOOL ReadFile(
  HANDLE       hFile,                // Handle to the file/device
  LPVOID       lpBuffer,             // Pointer to the buffer that receives the data
  DWORD        nNumberOfBytesToRead, // Maximum number of bytes to read
  LPDWORD      lpNumberOfBytesRead,  // Pointer to variable receiving number of bytes read
  LPOVERLAPPED lpOverlapped          // Pointer to OVERLAPPED struct for async I/O (or NULL for sync)
);
  • If the file is not opened in overlapped (asynchronous) mode, ReadFile() starts reading from the current file pointer position and updates it after the read.
  • If the current file pointer location is at the End of File (EOF), ReadFile() returns TRUE, but sets the value pointed to by *lpNumberOfBytesRead to zero. No error occurs.
  • The function returns FALSE if it fails (e.g., invalid handle, invalid parameters, read error). Use GetLastError() to get extended error information.

Parameters:

  • HANDLE hFile is the file handle obtained from CreateFile.
  • LPVOID lpBuffer is the address of the buffer (e.g., a character array) that stores the data read from the file.
  • DWORD nNumberOfBytesToRead is the maximum number of bytes to be read from the file into the buffer.
  • LPDWORD lpNumberOfBytesRead points to a DWORD variable that receives the number of bytes actually read. This can be less than requested if EOF is reached or an error occurs during synchronous reads, or if reading from a pipe/device with less data available. Must not be NULL for synchronous reads.
  • LPOVERLAPPED lpOverlapped is used for asynchronous (overlapped) I/O operations. Set to NULL for synchronous operations.

The WriteFile() API

The WriteFile() API is used to write data from a buffer to a file (or other output device).

Syntax:

BOOL WriteFile(
  HANDLE       hFile,                   // Handle to the file/device
  LPCVOID      lpBuffer,                // Pointer to the buffer containing data to write
  DWORD        nNumberOfBytesToWrite,   // Number of bytes to write
  LPDWORD      lpNumberOfBytesWritten,  // Pointer to variable receiving number of bytes written
  LPOVERLAPPED lpOverlapped             // Pointer to OVERLAPPED struct for async I/O (or NULL for sync)
);
  • Similar to ReadFile, performs synchronous write from the current file pointer if lpOverlapped is NULL.
  • Returns TRUE on success, FALSE on failure. Use GetLastError() for details.
  • *lpNumberOfBytesWritten receives the actual number of bytes written. In synchronous mode, failure usually means 0 bytes were written. A short write (fewer bytes than requested) usually indicates an error condition (e.g., disk full).
  • To ensure data is physically written to disk and not just cached, the file can be opened with the FILE_FLAG_WRITE_THROUGH flag in CreateFile, or FlushFileBuffers() can be called after writing.

Lecture 15: Closing a File

After opening and using a file (or other kernel object represented by a handle), it is required to close the handle to release the system resources associated with it and invalidate the handle variable.

The CloseHandle() API is used to close an open object handle (including file handles). A valid handle is passed as a parameter to this API. If the operation is successful, the function returns TRUE. In case the handle is already invalid or the close operation fails, it will return FALSE. Always close handles when you are finished with them to prevent resource leaks.

Lecture 16: Generic Characters

  • Windows supports both ASCII (usually represented as 8-bit single-byte characters, or more accurately, a specific code page like Windows-1252) and Unicode (usually UTF-16LE, using 16-bit code units) characters.
  • Windows has different API function variants to deal with both Unicode ('W' suffix, e.g., CreateFileW) and legacy ANSI/MBCS ('A' suffix, e.g., CreateFileA) character strings.
  • Unicode is used to represent characters in virtually all languages, including Arabic, Chinese, Urdu, etc., avoiding the limitations and ambiguities of code pages.
  • The structure of a program might differ slightly when handling Unicode (e.g., using wchar_t) versus ASCII/MBCS (using char).
  • A "Generic" program or code aims to work correctly whether compiled for ASCII/MBCS or Unicode.
  • Windows provides a set of generic data types (like TCHAR, LPTSTR, LPCTSTR) and macros that resolve to either the ASCII/MBCS or Unicode equivalents based on whether the UNICODE preprocessor symbol is defined during compilation.
  • For Unicode compilation, the following two macros should typically be defined before including :
    • #define UNICODE: Selects the Unicode ('W') versions of APIs.
    • #define _UNICODE: Selects the Unicode versions of C runtime library functions (like those in tchar.h) and affects types like TCHAR.
    (These macros force the generic types and functions to resolve to their Unicode counterparts).
  • The sizeof() operator should be used when calculating buffer sizes instead of hardcoding based on character type (e.g., sizeof(TCHAR) instead of assuming 1 or 2).
  • Use generic C Library functions provided in like _tcslen(), _stprintf(), _tcscpy() etc., which adapt based on the _UNICODE definition.
  • Use the _TEXT("string") or T("string") macro for representing string literals generically. This macro adds the L prefix for Unicode builds (e.g., L"string") and leaves it as is for non-Unicode builds.
  • The header file should be included for using generic C runtime functions and macros.
  • Most modern APIs in the latest versions of Windows primarily expect parameters in Unicode. Therefore, writing generic code compiled with UNICODE defined is the recommended practice for new Windows development.

Lecture 17: Generic Functions

  • Besides Generic data types, Windows and the C runtime library (via tchar.h) also support generic functions.
  • Examples of generic C runtime functions (defined in tchar.h):
    • _tcscmp() instead of strcmp() (ANSI) or wcscmp() (Unicode). Resolves based on _UNICODE.
    • _tcscmpi() or _tcsicmp() instead of _stricmp() or _wcsicmp() (case-insensitive compare).
    • Many others exist for string manipulation, formatting, character classification, etc.
  • Some Windows API functions that deal with Unicode characters and strings transparently work with locale settings:
    • CharUpper() / CharLower(): Convert strings to upper/lower case respecting locale (use CharUpperBuff for safety).

Lecture 18: Unicode Strategies

While writing a new code or enhancing an existing one, a programmer can adapt one of the following strategies regarding Unicode, based on requirements:

Ignore Unicode Policy (Not Recommended for New Code)

  • Use 8-bit char only.
  • Do not define UNICODE or _UNICODE.
  • Use ANSI ('A') versions of Windows APIs explicitly or implicitly.
  • Use standard library functions like printf(), scanf(), atoi() etc.
  • This approach severely limits the application's ability to handle international characters and file paths correctly on modern systems. Most Windows functionality cannot be fully utilized ignoring Unicode.

Using Generic Code Policy (Recommended for Portability/Maintainability)

  • Define both UNICODE and _UNICODE (usually for Unicode builds) or neither (for ANSI builds).
  • Use generic data types (TCHAR, LPTSTR, etc.).
  • Use generic functions (_tmain, _tcscpy, API macros like TextOut).
  • Use the _TEXT() or T() macro for string literals.
  • Allows compiling the same codebase for either Unicode or ANSI (though ANSI builds are rarely needed now).

Using Unicode Policy (Recommended for New Windows Development)

  • Define UNICODE and _UNICODE.
  • Use Unicode data types directly (WCHAR, LPWSTR) or use generic types (which resolve to Unicode types).
  • Use Unicode functions explicitly (wmain, wcscpy, API functions with 'W' suffix like CreateFileW) or use generic functions/macros.
  • Use wide character string literals (L"string") or the _TEXT() macro.
  • Ensures correct handling of international text and system objects like file paths.

Unicode and 8-bit Policy (Mixed Mode - Complex)

  • Write code to handle both Unicode and 8-bit (ANSI/UTF-8) data internally.
  • May involve converting between encodings frequently (e.g., using MultiByteToWideChar and WideCharToMultiByte).
  • Decision regarding which path to take might be made at runtime.
  • Can make the code more complex but might be necessary when interfacing with different systems or libraries that use different encodings.
  • Use of generic code often makes the codebase more flexible for different build configurations. The use of a Unicode-only strategy is increasingly popular and often simpler for Windows-centric development.

Topic 21: Standard IO devices

Operating systems typically provide three standard I/O devices (or streams) for processes:

  • Standard Input (stdin): Default source for input, usually the keyboard.
  • Standard Output (stdout): Default destination for output, usually the console window/screen.
  • Standard Error (stderr): Default destination for error messages, usually the console window/screen.

Standard C functions like printf write to standard output by default, and getc or scanf read from standard input by default. In Windows console applications, these standard devices are manipulated through handles.

Acquiring Handles to Standard IO Devices

The primary API to get handles for these standard devices is GetStdHandle():

HANDLE GetStdHandle(DWORD nStdHandle);
  • It returns a valid handle if the function succeeds.
  • In case of failure, it returns INVALID_HANDLE_VALUE.
  • Successive calls to the function for the same standard handle typically return the same handle value.
  • Important: Do not close standard handles using CloseHandle(), as multiple processes might share them, and closing them can cause unexpected behavior.

The nStdHandle parameter specifies which standard handle to retrieve, using one of the following values:

  1. STD_INPUT_HANDLE (-10): The standard input device.
  2. STD_OUTPUT_HANDLE (-11): The standard output device.
  3. STD_ERROR_HANDLE (-12): The standard error device.

These handles correspond to the console by default (CONIN$ for input, CONOUT$ for output/error) but can be redirected (e.g., using <, >, | in the command line or programmatically).

Redirecting Standard IO

The operating system allows redirection. You can also programmatically set the standard handles for a process using the SetStdHandle() API:

BOOL SetStdHandle(DWORD nStdHandle, HANDLE hHandle);
  • This function replaces the handle for the specified standard device (nStdHandle) with the provided handle (hHandle).
  • It returns TRUE if successful and FALSE in case of failure.
  • This is often used in parent processes to set up redirected I/O for child processes they create.

  • Copies the file from lpExistingFileName to lpNewFileName.

Topic 29: Dealing with 64-bit File Sizes

Historically, file systems like FAT16/FAT32 had limitations, notably a maximum file size of 4GB (232 bytes) for FAT32. Modern file systems like NTFS support vastly larger files (theoretically up to 264 bytes).

Files exceeding the 4GB limit are sometimes called "huge files". While 32-bit file sizes are sufficient for many applications, increasing disk capacities and large media files make handling 64-bit file sizes important. Windows provides APIs that support 64-bit file system operations.

Topic 30: File Pointer APIs

Whenever a file is opened using CreateFile(), the system maintains a file pointer indicating the current position for subsequent read/write operations. By default, it's placed at the start of the file (offset 0). ReadFile() and WriteFile() operations (when used synchronously without an explicit offset) advance this file pointer.

To perform random access (reading or writing at specific positions), the file pointer position needs to be changed. Windows provides functions for this:

`SetFilePointer()` (Legacy / 32-bit focused)

This function was traditionally used but is clumsy for 64-bit file offsets.

DWORD SetFilePointer(
          HANDLE hFile,              // Handle to the file
          LONG   lDistanceToMove,    // Low 32 bits of the signed distance to move
          PLONG  lpDistanceToMoveHigh, // Pointer to high 32 bits of distance (can be NULL for offsets < 2GB)
          DWORD  dwMoveMethod        // Starting point for the move
        );
  • hFile: Handle to the file.
  • lDistanceToMove: The low 32 bits of the signed byte offset.
  • lpDistanceToMoveHigh: A pointer to a LONG variable containing the high 32 bits of the signed offset. If the offset fits within a signed 32-bit value (~ +/- 2GB), this can be NULL. For larger offsets, this pointer must be valid. The function also *returns* the high 32 bits of the *new* file pointer position via this parameter if it's not NULL.
  • dwMoveMethod: The starting point for the move:
    • FILE_BEGIN: Move relative to the start of the file.
    • FILE_CURRENT: Move relative to the current file pointer position.
    • FILE_END: Move relative to the end of the file (lDistanceToMove is often negative or zero here).
  • **Return Value:** Returns the low 32 bits of the new file pointer position. If lpDistanceToMoveHigh is non-NULL, it receives the high 32 bits of the new position. If the function fails, it returns INVALID_SET_FILE_POINTER (0xFFFFFFFF) and GetLastError() provides the error code. Because 0xFFFFFFFF can be a valid low 32-bit position, you *must* check GetLastError() if the return is INVALID_SET_FILE_POINTER.
  • Using SetFilePointer for 64-bit offsets is complex due to the input/output nature of lpDistanceToMoveHigh.

`SetFilePointerEx()` (Recommended for 64-bit)

This function is preferred for its cleaner handling of 64-bit offsets.

BOOL SetFilePointerEx(
          HANDLE         hFile,             // Handle to the file
          LARGE_INTEGER  liDistanceToMove,  // Signed 64-bit distance to move
          PLARGE_INTEGER lpNewFilePointer,  // Pointer to variable receiving the new file pointer (optional)
          DWORD          dwMoveMethod       // Starting point (FILE_BEGIN, FILE_CURRENT, FILE_END)
        );
  • hFile: Handle to the file.
  • liDistanceToMove: A LARGE_INTEGER structure or 64-bit integer specifying the signed distance to move.
  • lpNewFilePointer: An optional pointer to a LARGE_INTEGER variable that will receive the new absolute file pointer position after the move. Can be NULL if the new position isn't needed.
  • dwMoveMethod: Same as for SetFilePointer (FILE_BEGIN, FILE_CURRENT, FILE_END).
  • **Return Value:** Returns TRUE on success, FALSE on failure. Much simpler error checking.

Topic 31: 64-bit Arithmetic and LARGE_INTEGER

To handle large file sizes (potentially > 232 bytes) and 64-bit file offsets, understanding 64-bit arithmetic is necessary. Windows provides the LARGE_INTEGER union to facilitate working with 64-bit signed integers.

Definition of LARGE_INTEGER Union

typedef union _LARGE_INTEGER {
          struct {
            DWORD LowPart;  // Low-order 32 bits
            LONG HighPart; // High-order 32 bits (signed)
          } DUMMYSTRUCTNAME; // Anonymous struct (often used)
          struct {
            DWORD LowPart;
            LONG HighPart;
          } u;              // Named struct member 'u'
          LONGLONG QuadPart; // The entire 64-bit signed value
        } LARGE_INTEGER;
  • This union allows accessing a 64-bit signed integer (LONGLONG or __int64) as a single QuadPart or as two 32-bit parts: LowPart (unsigned DWORD) and HighPart (signed LONG).
  • This structure is used by APIs like SetFilePointerEx and GetFileSizeEx.
  • You can manipulate the QuadPart directly using 64-bit arithmetic or work with the LowPart and HighPart separately, although direct QuadPart manipulation is usually simpler if your compiler supports 64-bit integers well.

(There is also a ULARGE_INTEGER for unsigned 64-bit integers).

Topic 32: Overlapped I/O Structure

The OVERLAPPED structure is defined in windows.h and is primarily used for asynchronous (overlapped) file I/O operations with ReadFile and WriteFile. However, it can also be used to specify the starting offset for a synchronous read or write operation, bypassing the need to call SetFilePointerEx separately.

Definition of OVERLAPPED Structure

typedef struct _OVERLAPPED {
          ULONG_PTR Internal;     // Reserved for operating system use.
          ULONG_PTR InternalHigh; // Reserved for operating system use.
          union {
            struct {
              DWORD Offset;     // Low-order 32 bits of the file position
              DWORD OffsetHigh; // High-order 32 bits of the file position
            } DUMMYSTRUCTNAME;
            PVOID Pointer;      // Alternate member (used in some contexts)
          } DUMMYUNIONNAME;
          HANDLE    hEvent;       // Handle to an event object signaled when the operation completes (for async)
        } OVERLAPPED, *LPOVERLAPPED;
  • Internal, InternalHigh: Reserved for system use. Do not modify directly.
  • Offset, OffsetHigh: These members specify the 64-bit starting offset within the file where the I/O operation should begin. This provides an alternative to using SetFilePointerEx before calling ReadFile/WriteFile, even for synchronous operations (when the file is NOT opened with FILE_FLAG_OVERLAPPED).
  • Pointer: An alternative union member, sometimes used for specific system purposes.
  • hEvent: Crucial for asynchronous I/O. It holds a handle to an event object that the system signals when the I/O operation completes. For synchronous use specifying an offset, this should typically be NULL or a valid handle that you manage appropriately (though it's not strictly necessary if not waiting on it).

Using OVERLAPPED for File Position

To read or write at a specific 64-bit offset `filePos` (a `LARGE_INTEGER`) using a synchronous handle:

OVERLAPPED ov = {0}; // IMPORTANT: Always zero-initialize the structure
        LARGE_INTEGER filePos;
        filePos.QuadPart = desired_64bit_offset; // Your target offset
        
        // Set the offset in the OVERLAPPED structure
        ov.Offset = filePos.LowPart;
        ov.OffsetHigh = filePos.HighPart;
        ov.hEvent = NULL; // Or a valid handle if needed for other reasons
        
        // Now call ReadFile or WriteFile with &ov as the last parameter
        BOOL result = ReadFile(hFile, buffer, bytesToRead, &bytesRead, &ov);
        // or WriteFile(hFile, buffer, bytesToWrite, &bytesWritten, &ov);
        
        // Note: When using OVERLAPPED, even synchronously, the file pointer
        // maintained by the system handle is NOT updated by the ReadFile/WriteFile call.
        // You must manage the position explicitly via the Offset/OffsetHigh fields.

To append to a file using this method, you can specify 0xFFFFFFFF in both Offset and OffsetHigh. This tells the system to perform the write at the current end of the file.

Topic 33: Getting and Setting File Size

One way to get the file size is to open the file, use SetFilePointerEx to move the file pointer to the end (offset 0 from FILE_END), and record the resulting file pointer position. However, Windows also provides a direct API:

`GetFileSizeEx()`

This function retrieves the size of the specified file.

          HANDLE         hFile,       // Handle to the file
          PLARGE_INTEGER lpFileSize   // Pointer to LARGE_INTEGER receiving the file size

  • The SetEndOfFile() function sets the end-of-file position for the specified file to the current position of the file pointer.
  • Syntax: BOOL SetEndOfFile(HANDLE hFile);
  • You must first use SetFilePointerEx to move the file pointer to the desired new end-of-file position. Then, calling SetEndOfFile marks that position as the end.
  • Requires GENERIC_WRITE access to the file.
  • Returns TRUE on success, FALSE on failure.

Topic 35: Searching for Files

Windows provides a set of APIs to search for files or folders within the directory structure. These allow iterating through directory contents based on a search pattern.

The main APIs involved are:

  • FindFirstFile() / FindFirstFileEx(): Start a search.
  • FindNextFile(): Continue the search, finding the next matching item.
  • FindClose(): End the search and release resources.

`FindFirstFile()` API

Initiates a directory search based on a filename pattern.

HANDLE FindFirstFile(
          LPCTSTR            lpFileName,      // Directory/path and pattern (e.g., "C:\\Users\\*.txt")

        );
  • lpFileName: Represents the directory or path and the filename pattern. Can include wildcard characters like * (matches zero or more characters) and ? (matches a single character).
  • lpFindFileData: A pointer to a WIN32_FIND_DATA structure that receives information about the first file or directory found matching the pattern.
  • **Return Value:** Returns a search handle (HANDLE) if successful, which is used in subsequent calls to FindNextFile and FindClose. Returns INVALID_HANDLE_VALUE if no matching files are found or an error occurs (check GetLastError).

(FindFirstFileEx provides more options for controlling the search, like filtering by attributes or using larger buffers).

This structure receives information about a found file or directory.

Topic 40: File Processing Strategies

When performing file processing in C/C++ on Windows, developers can choose between using the standard C library functions (like fopen, fread, fprintf) or the native Windows API functions (like CreateFile, ReadFile, WriteFile). Each approach has distinct features and trade-offs:

Standard C Library Features:

  • Portability: Code using standard C functions is generally portable to operating systems other than Windows with minimal or no changes.
  • Convenience: Provides convenient line-oriented (fgets, fputs) and character-oriented (fgetc, fputc), as well as formatted I/O (fprintf, fscanf) functions, often simplifying tasks like text file processing.
  • Abstraction Level: Functions are generally higher-level and conceptually simpler to use than the more granular Windows API functions.
  • Generic Migration: Line and stream character-oriented functions can often be migrated to use generic character types (TCHAR via tchar.h) for builds supporting both ANSI and Unicode, though care is needed.

Limitations and Advantages:

  • C Library Limitations:
    • Does not expose Windows-specific features like advanced file locking, asynchronous I/O (overlapped), security attributes management, or directory change notifications directly.
    • Buffering performed by the C runtime library can sometimes add latency or complexity compared to direct OS calls.
  • C Library Performance: Can sometimes offer performance advantages due to optimized buffering within the runtime library, especially for certain types of I/O patterns (e.g., formatted text output).
  • Windows API Advantages:
    • Provides full access to all underlying OS file system capabilities (security, locking, overlapped I/O, sparse files, etc.).
    • Offers fine-grained control over operations like caching behavior (FILE_FLAG_NO_BUFFERING, FILE_FLAG_WRITE_THROUGH).

The choice depends on the project's requirements: prioritize portability and convenience (Standard C Library) or low-level control and Windows-specific features (Windows API).

Topic 41: File Locking

Windows OS provides mechanisms to lock files, either entirely or specific byte ranges within a file. This prevents conflicting access by different processes or even different threads within the same process, ensuring data integrity.

The primary functions available are LockFileEx and the older LockFile.

`LockFileEx()` Syntax and Parameters

LockFileEx provides more control and is generally preferred.

BOOL LockFileEx(
  HANDLE         hFile,              // Handle to the open file
  DWORD          dwFlags,            // Lock options (LOCKFILE_EXCLUSIVE_LOCK, LOCKFILE_FAIL_IMMEDIATELY)
  DWORD          dwReserved,         // Reserved, must be 0
  DWORD          nNumberOfBytesToLockLow,  // Low 32 bits of the length of the byte range to lock
  DWORD          nNumberOfBytesToLockHigh, // High 32 bits of the length of the byte range to lock
  LPOVERLAPPED   lpOverlapped       // Pointer to OVERLAPPED struct containing the starting offset
);
  • hFile: Handle to the file, opened with appropriate access (e.g., GENERIC_READ or GENERIC_WRITE).
  • dwFlags: Specifies lock behavior:
    • LOCKFILE_EXCLUSIVE_LOCK (0x00000002): Requests an exclusive lock. Denies all other processes read and write access to the specified region. If omitted (0), requests a shared lock (allowing others to read but not write the locked region).
    • LOCKFILE_FAIL_IMMEDIATELY (0x00000001): If the lock cannot be granted immediately (because of a conflicting lock), the function fails instead of waiting.
  • dwReserved: Must be zero.
  • nNumberOfBytesToLockLow / nNumberOfBytesToLockHigh: Specify the 64-bit length of the byte range to lock. To lock from the specified offset to the end of the file, set both to 0xFFFFFFFF.
  • lpOverlapped: Pointer to an OVERLAPPED structure. The Offset and OffsetHigh members of this structure specify the 64-bit starting offset of the range to lock. The hEvent member is not used by LockFileEx.
  • Returns TRUE if the lock was granted, FALSE otherwise (check GetLastError). Common errors include ERROR_LOCK_VIOLATION if the lock conflicts or cannot be granted immediately (with LOCKFILE_FAIL_IMMEDIATELY).

(The older LockFile function has a similar purpose but works only with 32-bit offsets and lengths and always waits if the lock isn't immediately available).

Unlocking Files

To unlock a previously locked region, the UnlockFileEx() function is used. Its parameters list is highly similar to LockFileEx, specifying the same file handle and the same byte range (offset and length) that was originally locked.

BOOL UnlockFileEx(
  HANDLE       hFile,
  DWORD        dwReserved,
  DWORD        nNumberOfBytesToUnlockLow,
  DWORD        nNumberOfBytesToUnlockHigh,
  LPOVERLAPPED lpOverlapped
);

Topic 42: Releasing file locks

It is crucial that programs release file locks promptly after they are no longer needed. If a program does not release a lock (e.g., due to an error or crash before calling UnlockFileEx) or holds a lock for an unnecessarily long time, other programs attempting to access the locked region will be blocked.

This blocking can negatively impact the performance and responsiveness of other applications and the system overall. Therefore, programs using file locking should be designed carefully:

  • Acquire locks only when necessary and for the shortest duration possible.
  • Ensure locks are always released using appropriate functions (UnlockFileEx) in all code paths, including error handling paths (e.g., using `try...finally` or RAII techniques in C++).
  • Locks are automatically released when the file handle used to acquire them is closed or when the process terminates. However, relying solely on process termination is poor practice.

Topic 43: Lock Logic consequences

Improper use of file locking can lead to several adverse consequences:

  • Deadlocks: File locking can lead to deadlocks, similar to those caused by other mutual exclusion mechanisms (like mutexes or critical sections). A deadlock can occur if, for example, Process A locks Region 1 and needs Region 2, while Process B locks Region 2 and needs Region 1. Neither process can proceed.
  • Blocking: A process requesting a lock (shared or exclusive) on a region that conflicts with an existing lock held by another process will block (wait) unless LOCKFILE_FAIL_IMMEDIATELY was specified.
  • Partial Reads/Writes: If a process attempts to read or write a range of bytes that overlaps with a region locked by another process (in a conflicting mode), the read or write operation may fail (return FALSE). The associated byte transfer count (returned by ReadFile/WriteFile) might be less than the number requested, indicating that the operation stopped at the boundary of the locked region.
  • Shared Lock Issues: Shared locks allow multiple processes to read the locked region simultaneously but prevent any process from writing to it. If used improperly (e.g., if a process assumes it can eventually write after getting a shared lock, which might never happen if other shared locks persist), they can lead to unexpected program behavior or starvation.

The Windows Registry is a central, hierarchical database used by the Windows operating system to store low-level configuration settings and options. It contains information and settings for hardware, operating system software, most non-operating system software, user preferences, and application settings.

Many components of Windows rely on the registry, including:

  • The OS kernel
  • Device drivers
  • Services
  • The Security Accounts Manager (SAM)
  • User interface components
  • Applications (though modern applications increasingly use configuration files or other methods)

When programs or hardware are installed or configured, information such as settings, options, versions, and file paths may be added to or updated in the registry.

Topic 45: Registry Keys

The registry's structure consists of two basic elements:

  • Keys: These are container objects, analogous to folders or directories in a file system. Keys contain values and can also contain other keys (sub-keys), forming a hierarchical tree structure.
  • Values: These are non-container objects, analogous to files. They store the actual configuration data associated with a key. Each value has a name, a data type, and the data itself.

Keys are referenced using a syntax similar to Windows file paths, with backslashes (\) separating levels in the hierarchy (e.g., HKEY_LOCAL_MACHINE\Software\Microsoft). Key names are case-insensitive and cannot contain backslashes.

Access to the registry hierarchy begins at one of several predefined root keys (also known as hives). These root keys are represented by constant numeric handles in the Windows API. Examples include:

  • HKEY_CLASSES_ROOT (HKCR)
  • HKEY_CURRENT_USER (HKCU)
  • HKEY_LOCAL_MACHINE (HKLM)
  • HKEY_USERS (HKU)
  • HKEY_CURRENT_CONFIG (HKCC)
  • HKEY_PERFORMANCE_DATA (Windows NT only, special virtual key)
  • HKEY_DYN_DATA (Windows 9x only, special virtual key)

These root keys are mapped by the kernel to the content of stored "hive" files (e.g., SOFTWARE, SYSTEM, SAM, NTUSER.DAT) or to dynamically generated system information.

Topic 48: Exception and their Handlers

Structured Exception Handling (SEH) is a Microsoft-specific extension to the C language (and also supported in Microsoft C++) designed to gracefully handle exceptional code situations, particularly hardware-related faults (like access violations or divide-by-zero errors).

SEH provides mechanisms (like __try, __except, __finally) to intercept these low-level exceptions and execute specific handler code.

While Windows and Microsoft C++ support SEH, the general recommendation for C++ development is to use ISO-standard C++ exception handling (using try, catch, throw). Standard C++ exceptions offer several advantages:

  • Portability: Standard C++ code is portable across different compilers and operating systems. SEH is specific to Windows and Microsoft's compiler extensions.
  • Flexibility: Standard C++ exceptions are type-safe and integrate well with object-oriented programming principles (e.g., RAII - Resource Acquisition Is Initialization for cleanup).

However, SEH might still be necessary when:

  • Maintaining existing legacy code that uses SEH.
  • Writing specific kinds of programs where direct handling of hardware exceptions is required (though C++ exceptions can sometimes wrap SEH exceptions).
  • Working in pure C code where standard C++ exceptions are unavailable.

Topic 49: Try and Except Blocks

Different programming languages implement exception handling with varying syntax. The source material uses Python as an example to illustrate the common concepts:

  • Try Block: Contains the code that might potentially raise an exception.
  • Except Block (like `catch` in C++): Contains the code to handle a specific type of exception if it occurs in the `try` block.
  • Else Block (Python specific): Contains code that executes only if no exceptions occurred in the `try` block.
  • Finally Block: Contains cleanup code that executes regardless of whether an exception occurred or was handled in the `try` or `except` blocks.

In the context of Windows SEH (Topic 48), the relevant constructs are:

  • __try: Encloses the code block to be protected.
  • __except: Follows a __try block and contains an exception filter expression and a handler block. The handler block executes only if the filter expression evaluates to allow handling.
  • __finally: Follows a __try block and contains cleanup code that executes whether or not an exception occurred, during the "unwinding" process.

Topic 50: Filter Expressions

A key part of Windows Structured Exception Handling (SEH) is the filter expression used within the __except block. It determines how an exception is handled.

__try {
    // Protected code that might raise an exception
}
__except ( filter_expression ) {
    // Exception handler code block
}

The filter_expression is evaluated when an exception occurs within the preceding __try block. It must evaluate to one of three values:

  1. EXCEPTION_EXECUTE_HANDLER (1): Indicates that this handler recognizes the exception and should execute. The code block following the __except statement is executed, and then execution typically continues after the __except block (stack unwinding occurs before the handler runs).
  2. EXCEPTION_CONTINUE_SEARCH (0): Indicates that this handler does not handle this specific exception. The system continues searching up the call stack for another handler (__except or C++ catch).
  3. EXCEPTION_CONTINUE_EXECUTION (-1): Dismisses the exception. Execution resumes at the point where the exception originally occurred. This should be used with extreme caution, only if the filter expression has fully resolved the condition that caused the exception. Otherwise, the exception will likely reoccur immediately.

Instead of complex logic directly in the filter expression, it's common practice to call a function from the filter expression. This function can analyze the exception details (using GetExceptionInformation()), perform logging or cleanup, and then return one of the three values above.

int MyExceptionFilter(LPEXCEPTION_POINTERS pExInfo) {
   // Analyze pExInfo->ExceptionRecord->ExceptionCode, etc.
   // Perform logging or specific fixes if possible.
   if (/* should handle */) {
       return EXCEPTION_EXECUTE_HANDLER;
   } else if (/* can safely continue */) {
       // Maybe fix the cause of the exception here.
       return EXCEPTION_CONTINUE_EXECUTION;
   } else {
       return EXCEPTION_CONTINUE_SEARCH;
   }
}

__try {
    // ... code ...
}
__except (MyExceptionFilter(GetExceptionInformation())) {
    // Handler code...
}

The source text mentions an example where a filter function (`Eval_Exception`) *never* returns EXCEPTION_EXECUTE_HANDLER (1), meaning the __except block itself is never entered. This might be used if the filter's only purpose is logging or attempting to fix the issue and continue execution, or simply passing the exception up the chain.

Other C language features useful within complex filter expressions include the conditional (ternary) operator (? :) and the comma operator (,) for sequencing operations, although calling a dedicated filter function is usually clearer.

Topic 52: Exceptions Handling Sequence

The sequence of execution for exception handling varies between languages. The example provided uses Python's structure:

try:
        # Operations that might raise an exception
        do_operations_here()
    except ExceptionI:
        # Block executed if an exception of type ExceptionI occurs
        handle_exception_I()
    except ExceptionII:
        # Block executed if an exception of type ExceptionII occurs
        handle_exception_II()
    # ... more except blocks for different exception types ...
    else:
        # Block executed ONLY if NO exception occurred in the try block
        execute_if_no_exception()
    finally:

        perform_cleanup()

This sequence allows for specific handling based on the type of exception, optional execution when no exception occurs, and guaranteed cleanup.

(Contrast this with C++ `try`/`catch` which lacks a direct `else` block, and SEH `__try`/`__except`/`__finally` which has different execution flow rules based on filter expressions and termination handling).

Topic 53: Floating-Point Exceptions

The Institute of Electrical and Electronics Engineers (IEEE) defines standards for floating-point arithmetic, notably the IEEE 754 standard (IEEE Standard for Binary Floating-Point Arithmetic). This standard defines how floating-point numbers are represented and how operations are performed, including specifying conditions that raise floating-point exceptions.

The five standard types of floating-point exceptions defined by IEEE 754 are:

  • Invalid operation: Occurs for operations like square root of a negative number, 0/0, infinity - infinity, or operations involving NaN (Not a Number).
  • Division by zero: Occurs when dividing a non-zero finite number by zero.
  • Overflow: Occurs when the result of an operation is too large to be represented in the destination format (e.g., exceeds the maximum representable positive or negative value).
  • Underflow: Occurs when the result of an operation is too small (close to zero) to be represented accurately as a normalized number, potentially resulting in a denormalized number or zero.
  • Inexact calculation: Occurs when the result of an operation requires rounding because it cannot be represented exactly in the destination format. This is very common and often ignored.

Operating systems and programming environments provide mechanisms (like SEH filters or specific runtime functions, e.g., _controlfp_s in C/C++) to detect and potentially handle these floating-point exceptions.

Topic 54: Errors and Exceptions

While often used interchangeably in casual conversation, "errors" and "exceptions" can represent different concepts in programming, particularly in contexts like Java or systems programming:

Feature Errors Exceptions
Origin Usually raised by the runtime environment (e.g., JVM, OS) due to severe problems outside the application's direct control (e.g., out of memory, stack overflow). Typically caused by the application's code logic (e.g., invalid input, null pointer, file not found, division by zero).
Recovery Generally considered unrecoverable. Attempting to handle them is usually not recommended as the system state might be unstable. Often recoverable. Exception handling mechanisms (try-catch, __try-__except) are designed to catch exceptions and allow the program to recover or terminate gracefully.
Detection Time Occur at runtime and are typically unknown to the compiler. Can occur at runtime. Some types (checked exceptions in Java) *must* be declared or handled, making the compiler aware; others (unchecked/runtime exceptions) may not be caught by the compiler.

Note: In the context of Windows C/C++ programming and SEH, the distinction is less strict. SEH handles both software-generated exceptions (like dividing by zero in code) and hardware/system exceptions (like memory access violations), blurring the lines compared to the typical Java definition of Error vs. Exception.

Topic 55: Treating Errors as Exceptions

In Windows programming, return codes from API functions often indicate errors (e.g., CreateFile returning INVALID_HANDLE_VALUE). Programmers typically check these return codes explicitly.

The ReportError function (discussed in Topic 20) provides a way to handle these API errors: report the error message (using GetLastError and FormatMessage) and potentially terminate the process.

However, simply terminating the entire process via ReportError (calling ExitProcess) has limitations:

  • Process vs. Thread Termination: A fatal error reported by ReportError terminates the entire process, even if the error occurred in a specific thread that could potentially be terminated or recovered independently.
  • Continuation Requirement: Sometimes, the programmer might want the application to attempt recovery or continue execution with reduced functionality after an error, rather than terminating immediately.
  • Resource Cleanup: Abrupt termination via ExitProcess might prevent proper cleanup. Synchronization resources (like mutexes, semaphores, critical sections) held by the terminating process might not be released correctly, potentially impacting other processes. File handles might not be flushed or closed cleanly. (Termination handlers, Topic 56, address this).

Therefore, while convenient for simple cases or truly fatal errors, relying solely on a function like ReportError for all error handling might not be robust enough for complex applications.

Topic 56: Termination Handlers

A Termination Handler provides a mechanism to guarantee the execution of cleanup code when the flow of control leaves a specific block of code, regardless of how it leaves (normal execution, exceptions, jumps like return, goto, break, continue).

While similar in structure to an exception handler, its purpose is different: it focuses on cleanup rather than diagnosing or handling the cause of leaving the block. A termination handler cannot diagnose an exception directly.

A termination handler construct consists of:

  • A guarded body of code: The block of code whose exit triggers the handler.
  • A termination block: The block of cleanup code that is guaranteed to execute when control leaves the guarded body.

In Microsoft C/C++, termination handlers are implemented using the SEH keywords __try and __finally:

ResourceType* resource = NULL;
    __try {
        // Guarded body of code
        resource = AcquireResource(); // Acquire resource
        if (resource == NULL) { /* handle error */ }
    
        // Perform operations using the resource...
        if (SomeCondition()) {
            return; // Leaving via return
        }
        DoSomethingElse(resource);
        // Normal completion falls through
    
    } __finally {
        // Termination block (cleanup code)
        // This code executes regardless of how the __try block was exited.
        if (resource != NULL) {
            ReleaseResource(resource); // Guaranteed resource release
        }
    }
    // Execution continues here after __finally block (unless __try block exited via return/goto/exception not handled)

The termination block (__finally) is executed when control leaves the guarded body (__try), whether normally (falling through the end), via return, goto, break, continue, or due to an exception propagating out of the __try block.

Termination handlers simplify recovery and cleanup by guaranteeing that critical cleanup actions, such as releasing resources (memory, file handles, locks, network connections), occur regardless of errors or exceptions.

Topic 59: Console Control Handles

Console Control Handlers provide a mechanism similar to exception handlers but specifically for console-related events, most notably the Ctrl+C signal.

While normal exception handling (SEH or C++) responds to events like division by zero or access violations, it doesn't typically catch console events like Ctrl+C or Ctrl+Break by default.

Console control handlers allow a console application to detect and respond to these events, often for graceful shutdown or cleanup.

The API used to register or unregister a handler is SetConsoleCtrlHandler():

BOOL SetConsoleCtrlHandler(
      PHANDLER_ROUTINE HandlerRoutine, // Pointer to the handler function
      BOOL             Add             // TRUE to add/register, FALSE to remove
    );
  • HandlerRoutine: A pointer to the application-defined callback function (the "handler routine") that will be invoked when a console control event occurs.
  • Add: If TRUE, the handler is added to the list of handlers for the process. If FALSE, the handler is removed from the list.
  • Multiple handlers can be registered for a process. They form a chain.
  • If HandlerRoutine is NULL and Add is TRUE, the process essentially ignores Ctrl+C events (this is generally discouraged).
  • Returns TRUE on success, FALSE on failure.

Handler Routine Function

The handler routine itself must have the following signature:

BOOL WINAPI HandlerRoutine (DWORD dwCtrlType);
  • dwCtrlType: Identifies the type of control event that occurred. Common values include:
    • CTRL_C_EVENT (0): Ctrl+C signal received.
    • CTRL_BREAK_EVENT (1): Ctrl+Break signal received.
    • CTRL_CLOSE_EVENT (2): Console window is being closed.
    • CTRL_LOGOFF_EVENT (5): User is logging off.
    • CTRL_SHUTDOWN_EVENT (6): System is shutting down.
  • Return Value: The handler should return TRUE if it handled the signal (preventing subsequent handlers in the chain from being called). It should return FALSE if it did not handle the signal, allowing the system to pass the signal to the next registered handler.

Topic 60: Console Control Handler Example

This topic describes a simple program demonstrating the use of a console control handler.

Program Logic:

  1. Setup: Includes necessary headers (e.g., "Everything.h"). Declares a handler function prototype (e.g., `Handler`) and a global flag (e.g., `exitFlag`).
  2. `_tmain` Function:
    • Registers the `Handler` function using `SetConsoleCtrlHandler(Handler, TRUE)`. Reports an error if registration fails.
    • Enters a loop (`while (!exitFlag)`).
    • Inside the loop, it periodically performs an action (e.g., calls `Sleep` and then `Beep`).
    • The loop continues until `exitFlag` becomes TRUE.
    • Prints a "Stopping" message after the loop.
    • Returns 0.
  3. `Handler` Function (`BOOL WINAPI Handler(DWORD dwCtrlType)`):
    • This function is invoked by the system in a separate thread when a console event occurs.
    • Uses a `switch` statement on `dwCtrlType`.
    • For relevant events (e.g., `CTRL_C_EVENT`, `CTRL_CLOSE_EVENT`), it:
      • Prints a message indicating the event was received.
      • Sets the global `exitFlag` to TRUE. This signals the main loop in `_tmain` to terminate.
      • May include a `Sleep` to simulate work or observe timing.
      • Prints a message indicating the handler is exiting.
      • Returns TRUE to indicate the signal was handled and should not be passed to other handlers (including the default system handler which would terminate the process).
    • Might have a `default` case to handle unexpected event types, possibly also setting `exitFlag` and returning `TRUE`.

Execution Flow:

  • The program starts, registers the handler, and begins beeping in the main loop.
  • If the user presses Ctrl+C or closes the console window, the system invokes the `Handler` function in a separate thread.
  • The `Handler` function identifies the event, prints messages, sets `exitFlag = TRUE`, and returns `TRUE`.
  • The main loop in `_tmain`, upon checking `exitFlag` after its `Sleep`, finds it TRUE, exits the loop, prints the "Stopping" message, and terminates gracefully.

Topic 61: Vectored Exceptions Handling (VEH)

Vectored Exception Handling (VEH) provides another mechanism in Windows for handling exceptions, distinct from Structured Exception Handling (SEH - `__try`/`__except`).

Key differences and features:

  • Invocation Order: When an exception occurs, Windows first looks for registered Vectored Exception Handlers *before* starting the stack unwinding process required for SEH (`__except`) or C++ (`catch`) handlers.
  • No Keywords Required: VEH does not use language keywords like `__try` or `__except`. Handlers are registered programmatically using Windows APIs.
  • Scope: VEH handlers are associated with the process, not a specific frame on the call stack like SEH handlers.
  • Analogy: Similar in concept to Console Control Handlers, where handlers are registered and called back by the system.

VEH Management APIs

  • AddVectoredExceptionHandler(): Registers a VEH handler for the current process. 
    PVOID WINAPI AddVectoredExceptionHandler(
      ULONG                       FirstHandler,    // Non-zero for first call, zero for last call
      PVECTORED_EXCEPTION_HANDLER VectoredHandler  // Pointer to the handler function
    );
    • FirstHandler: If non-zero, the handler is added at the beginning of the list (called first). If zero, it's added at the end (called last).
    • VectoredHandler: Pointer to the application-defined handler function.
    • Returns a handle to the registration on success, or `NULL` on failure. This handle is used for removal.
  • RemoveVectoredExceptionHandler(): Removes a previously registered VEH handler. 
    ULONG WINAPI RemoveVectoredExceptionHandler(
      PVOID HandlerHandle // Handle returned by AddVectoredExceptionHandler
    );
    • Returns non-zero on success, zero on failure.

VEH Handler Function Structure

LONG WINAPI VectoredHandler(
      PEXCEPTION_POINTERS ExceptionInfo // Pointer to exception information
    );
  • ExceptionInfo: A pointer to an EXCEPTION_POINTERS structure, the same type retrieved by GetExceptionInformation() in SEH filters. It contains pointers to the exception record and context record.
  • Return Value: The handler function returns one of the same disposition values used by SEH filters:
    • EXCEPTION_CONTINUE_EXECUTION (-1): Dismisses the exception, execution continues from the point of the exception (use with caution).
    • EXCEPTION_CONTINUE_SEARCH (0): This handler does not handle the exception. The system continues searching for other handlers (next VEH handler, then SEH/C++ handlers via stack unwinding).
    • EXCEPTION_EXECUTE_HANDLER (1): (Not typically returned by VEH). Returning this value from VEH is treated like EXCEPTION_CONTINUE_SEARCH. VEH primarily decides whether to continue the search or resume execution. The actual handling (like terminating or unwinding) is usually left to SEH/C++ handlers if the search continues.

Pitfalls of Vectored Exceptions

  • Speed: Handler functions should execute quickly and return promptly, as they might delay the handling of critical exceptions.
  • Restrictions: Handlers should avoid blocking operations (like Sleep, waiting on synchronization objects) or complex processing that could interfere with exception processing.
  • Purpose: Typically used for monitoring/logging exceptions, making dynamic decisions about handling (e.g., attaching a debugger), or performing minimal, quick fixes before potentially continuing execution or letting SEH handle the rest.

Topic 62: Memory Management

Dynamic Memory

Dynamic memory allocation is essential when the exact memory requirements of a program are not known at compile time, or when data structures need to grow and shrink during execution (e.g., search tables, trees, linked lists).

Windows provides a comprehensive set of APIs for managing dynamic memory, primarily through:

  • Heap Allocation: Managing blocks of memory within heap objects (discussed further in Topic 64+).
  • Virtual Memory Management: Lower-level control over reserving and committing pages in the process's virtual address space.
  • Memory Mapped Files: Allows mapping a file directly into the process's address space, treating file data as if it were in memory. This facilitates direct access without explicit file I/O calls, helps manage large files, and provides a mechanism for inter-process memory sharing.

Topic 63: Windows Memory Management Overview

Dynamic Memory & Memory Mapped Files (Reiteration)

As mentioned in Topic 62, dynamic memory allocation is needed for data structures with sizes unknown at compile time. Windows provides APIs for heap allocation.

Memory-mapped files allow direct mapping of file contents into a process's virtual address space, bypassing traditional file I/O APIs. This can improve performance for file handling and allows memory treatment of file data, also enabling memory sharing between processes by mapping the same file.

Windows Memory Management Platforms (Win32 vs. Win64)

Windows operates primarily on two API platforms regarding memory addressing:

  • Win32 (32-bit):
    • Uses 32-bit pointers.
    • Each process has a virtual address space of 232 bytes (4 Gigabytes).
    • Typically, 2GB (or 3GB with /3GB switch) is accessible to the user process, with the remaining space reserved for the kernel (shared data, code, drivers).
    • Data types and structures were historically optimized for 32-bit alignment.
  • Win64 (64-bit):
    • Uses 64-bit pointers.
    • Virtual address space is vastly larger, 264 bytes (16 Exabytes), though practically limited by current hardware and OS implementations (e.g., 256 Terabytes user space on current Windows).
    • Provides significantly more virtual memory for user programs.

A good design strategy aims for code portability, allowing an application to be compiled and run correctly in both 32-bit and 64-bit modes with minimal changes (e.g., using appropriate data types like SIZE_T, ULONG_PTR, and avoiding assumptions about pointer sizes).

The large virtual address space provided by Windows (especially Win64) requires sophisticated memory management by the OS.

System Information

Information about the system's memory architecture (page size, allocation granularity, address space limits) can be obtained using the GetSystemInfo() API:

VOID GetSystemInfo(LPSYSTEM_INFO lpSystemInfo);

This function fills a SYSTEM_INFO structure with details about the underlying hardware and memory configuration.

Topic 64: Introduction to Heaps

In programming, dynamic memory is typically allocated from a region called the heap. Windows maintains pools of memory for this purpose.

While standard C library functions like malloc(), calloc(), realloc(), and free() provide heap allocation, they usually operate on a single, default heap managed by the C runtime library.

Windows provides its own, more granular heap management APIs. A Windows heap is an object accessed via a HANDLE. Every process has a default process heap automatically created by the system.

Default Process Heap

The handle to the default process heap can be obtained using:

HANDLE GetProcessHeap(VOID);

This API returns the handle to the calling process's default heap, or NULL on failure (though failure is rare). The C runtime's malloc often uses this heap internally.

Separate Heaps

While a single heap is often sufficient, creating multiple, separate heaps within a process using Windows APIs can offer several advantages, making a program potentially more efficient:

  1. Thread Isolation: Assigning a distinct heap to each thread (or group of related threads) can prevent threads from interfering with each other's memory allocations if access isn't serialized (see point 2).
  2. Reduced Memory Contention: When multiple threads allocate/free memory frequently from the same heap, they contend for the heap's internal locks, which serializes access and can become a bottleneck. Separate heaps reduce this contention.
  3. Improved Locality / Reduced Fragmentation: Allocating objects of a specific, fixed size from their own dedicated heap can reduce memory fragmentation and improve locality of reference, potentially reducing page faults.
  4. Simplified Synchronization: If a heap is used exclusively by a single thread, allocations/deallocations on that heap don't require explicit locking (using the HEAP_NO_SERIALIZE flag, see Topic 65).
  5. Bulk De-allocation: Complex data structures allocated entirely within a single, dedicated heap can be deallocated all at once by simply destroying the entire heap (HeapDestroy), which is much faster than freeing each element individually.
  6. Component Isolation: Different software components or libraries within a process can use separate heaps to avoid interfering with each other's allocation patterns or potentially corrupting a shared heap.

Topic 65: Creating Heaps

New, private heaps (separate from the default process heap) can be created using the HeapCreate() API.

HANDLE HeapCreate(
      DWORD   flOptions,     // Heap allocation options
      SIZE_T  dwInitialSize, // Initial size of the heap (committed)
      SIZE_T  dwMaximumSize  // Maximum size the heap can grow to
    );
  • flOptions: A set of flags controlling heap behavior. Common flags include:
    • HEAP_GENERATE_EXCEPTIONS (0x00000004): If memory cannot be allocated (e.g., heap limit reached), the system raises a structured exception (STATUS_NO_MEMORY or STATUS_ACCESS_VIOLATION) instead of returning NULL from allocation functions like HeapAlloc. Note: This flag applies to subsequent allocations, not HeapCreate itself.
    • HEAP_NO_SERIALIZE (0x00000001): Disables automatic mutual exclusion locking for heap access. This provides a performance improvement *only if* access to the heap is manually serialized by the application (e.g., the heap is used by only one thread, or access is protected by a critical section). Using this incorrectly in a multi-threaded environment will lead to heap corruption.
    • HEAP_CREATE_ENABLE_EXECUTE (0x00040000): Allows code to be executed from memory allocated in this heap. By default, Data Execution Prevention (DEP) usually prevents code execution from heap memory. Use with caution.
    • Can be 0 if no special options are needed (implies serialization is enabled, exceptions are not automatically generated).
  • dwInitialSize: The initial amount of memory (in bytes) to commit for the heap. The system rounds this up to a multiple of the page size. Memory is committed upfront.
  • dwMaximumSize:
    • If non-zero, specifies the maximum total size (in bytes) the heap can grow to. Allocations will fail if this limit is exceeded.
    • If zero, the heap is "growable". It can expand up to the limits of available virtual memory.
  • SIZE_T: A data type that represents sizes (unsigned integer). It's 32 bits wide on Win32 platforms and 64 bits wide on Win64 platforms, ensuring portability for size parameters.
  • **Return Value:** Returns a handle to the newly created heap on success, or `NULL` on failure (check `GetLastError`).

Deferring memory commitment by setting `dwInitialSize` relatively small (or zero) can be good practice if the required memory isn't known upfront, as heaps consume system resources.

Destroying Heaps

When a private heap is no longer needed, it should be destroyed to release all associated memory and resources back to the system. Using CloseHandle() is incorrect for heap handles.

BOOL HeapDestroy(HANDLE hHeap);
  • hHeap: The handle to the heap previously created by HeapCreate.
  • **Important:** Do NOT use this function on the handle obtained from GetProcessHeap(). The default process heap is managed by the system and destroyed automatically on process termination.
  • Destroying a heap invalidates all pointers to memory previously allocated from that heap.
  • Returns TRUE on success, FALSE on failure.

HeapDestroy provides an efficient way to free all memory associated with a heap, including complex data structures allocated within it.

Topic 66: Managing Heap Memory

Once a heap is created (using HeapCreate or obtained via GetProcessHeap), memory blocks are allocated, reallocated, and freed within that heap using specific APIs.

Allocating Memory (`HeapAlloc`)

LPVOID HeapAlloc(
      HANDLE hHeap,   // Handle to the heap
      DWORD  dwFlags, // Allocation options
      SIZE_T dwBytes  // Number of bytes to allocate
    );
  • hHeap: The handle of the heap from which to allocate.
  • dwFlags: Similar options to HeapCreate, controlling behavior for *this specific allocation*:
    • HEAP_GENERATE_EXCEPTIONS: Raise an exception on allocation failure (if also set during HeapCreate, or if used with the default process heap).
    • HEAP_NO_SERIALIZE: Perform this allocation without acquiring the heap lock (only safe if access is already serialized externally and the heap was created with this flag or it's the only thread accessing).
    • HEAP_ZERO_MEMORY (0x00000008): Initializes the allocated memory block to zeros.
  • dwBytes: The number of bytes to allocate. (There's an upper limit, mentioned as ~0.5MB (0x7FFF8) for non-growable heaps in the OCR, but typically limited by available memory/address space for growable heaps).
  • **Return Value:** On success, returns a pointer (LPVOID) to the allocated memory block. On failure:
    • If HEAP_GENERATE_EXCEPTIONS was specified (and effective), an exception is raised (STATUS_NO_MEMORY or STATUS_ACCESS_VIOLATION).
    • Otherwise, returns NULL. Note: GetLastError() does *not* provide meaningful error information for HeapAlloc failure when exceptions are not generated. You must check for NULL.

Freeing Memory (`HeapFree`)

BOOL HeapFree(
      HANDLE  hHeap,   // Handle to the heap
      DWORD   dwFlags, // Free options (HEAP_NO_SERIALIZE)
      LPVOID _Frees_ptr_opt_ lpMem // Pointer to the memory block to free
    );
  • hHeap: Handle to the heap where the memory was allocated.
  • dwFlags: Usually 0. Can include HEAP_NO_SERIALIZE if access is externally serialized.
  • lpMem: The pointer to the memory block previously returned by HeapAlloc or HeapReAlloc. Must be a pointer allocated from the specified `hHeap`. Freeing a null pointer or an invalid pointer can corrupt the heap. The `_Frees_ptr_opt_` annotation indicates the pointer `lpMem` is freed by this call.
  • Returns TRUE on success, FALSE on failure (e.g., invalid handle, invalid pointer, heap corruption). GetLastError() can provide error details on failure.

Reallocating Memory (`HeapReAlloc`)

LPVOID HeapReAlloc(
      HANDLE  hHeap,    // Handle to the heap
      DWORD   dwFlags,  // Reallocation options
      LPVOID _Frees_ptr_opt_ lpMem, // Original block to reallocate
      SIZE_T  dwBytes   // New desired size in bytes
    );
  • Attempts to resize a previously allocated block (`lpMem`).
  • dwFlags: Can include HEAP_GENERATE_EXCEPTIONS, HEAP_NO_SERIALIZE, HEAP_ZERO_MEMORY (only zeroes the *additional* memory if expanding), and:
    • HEAP_REALLOC_IN_PLACE_ONLY (0x00000010): Fails if the block cannot be resized without moving it to a new location in memory.
  • lpMem: Pointer to the block allocated from `hHeap`.
  • dwBytes: The new desired size. Can be larger or smaller than the original.
  • **Return Value:** On success, returns a pointer to the reallocated block (which *may* be different from `lpMem`). On failure, returns NULL (if exceptions disabled) or raises an exception. If `NULL` is returned, the original block at `lpMem` is *not* freed.

Topic 67: Heap Size and Serialization

Sometimes it's necessary to determine the actual size of a memory block previously allocated from a heap.

Getting Allocated Block Size (`HeapSize`)

SIZE_T HeapSize(
      HANDLE  hHeap,   // Handle to the heap
      DWORD   dwFlags, // Options (HEAP_NO_SERIALIZE)
      LPCVOID lpMem    // Pointer to the allocated block
    );
  • hHeap: Handle to the heap where `lpMem` was allocated.
  • dwFlags: The only valid flag is HEAP_NO_SERIALIZE.
  • lpMem: Pointer to the allocated block.
  • Returns the size (in bytes) of the allocated block on success. This size might be slightly larger than the requested size due to heap management overhead and alignment.
  • Returns (SIZE_T)-1 on failure (check GetLastError).

Serialization (`HEAP_NO_SERIALIZE`)

Serialization ensures that only one thread can access the heap's internal data structures at a time, preventing corruption in multi-threaded environments. This is the default behavior for heaps created without HEAP_NO_SERIALIZE and for the default process heap.

Using the HEAP_NO_SERIALIZE flag (in HeapCreate, HeapAlloc, HeapFree, HeapReAlloc, HeapSize) bypasses the heap's internal locking mechanism. This can provide a performance boost (estimated around 16% in the source material) by eliminating lock contention.

However, HEAP_NO_SERIALIZE is **only safe** to use if heap access is guaranteed to be serialized by other means. This applies if:

  1. The entire program is single-threaded.
  2. The specific heap being accessed is private to a single thread (no other thread ever uses its handle).
  3. All access to the heap from multiple threads is explicitly protected by an external synchronization mechanism (like a critical section, mutex, etc.) ensuring only one thread operates on the heap at any given moment.

Using HEAP_NO_SERIALIZE without proper external serialization will inevitably lead to heap corruption and crashes.

Heap Exceptions (`HEAP_GENERATE_EXCEPTIONS`)

Enabling heap exceptions with this flag changes the behavior of allocation functions (HeapAlloc, HeapReAlloc) upon failure. Instead of returning NULL, they raise a structured exception.

This allows using SEH (__try/__except) or VEH to catch memory allocation failures centrally, which can be useful for ensuring critical cleanup (like closing handles) occurs before the program terminates due to lack of memory.

The two primary exceptions raised are:

  • STATUS_NO_MEMORY (0xC0000017): Indicates allocation failed because the heap reached its maximum size limit or the system is out of available virtual memory/commit charge.
  • STATUS_ACCESS_VIOLATION (0xC0000005): Can indicate heap corruption (e.g., attempting to allocate within a corrupted heap) or possibly accessing the heap handle incorrectly. The source also suggests it might occur if trying to execute code from a non-executable heap (related to DEP).

Other Heap Functions

Windows provides other functions for advanced heap management:

  • HeapSetInformation(): Configures heap features, such as enabling the Low Fragmentation Heap (LFH) for specific heaps (often done automatically by the system) or setting policies for handling heap corruption.
  • HeapCompact(): Attempts to coalesce free blocks within the heap. Returns the size of the largest committed free block (useful for estimating if a large allocation might succeed).
  • HeapValidate(): Checks the integrity of the specified heap or a specific block within it. Useful for debugging heap corruption.
  • HeapWalk(): Enumerates the memory blocks (both allocated and free) within a heap.
  • HeapLock() / HeapUnlock(): Explicitly acquire or release the heap's internal lock when serialization is enabled. Allows performing multiple heap operations atomically without interruption from other threads using the same heap. Generally less efficient than using HEAP_NO_SERIALIZE with external locking if many operations are needed.

Topic 68: Using Heaps

This topic summarizes the typical methodology for using Windows heap APIs for dynamic memory management.

  1. Get Heap Handle: Obtain a handle to the heap you want to use.
    • Use GetProcessHeap() to get the handle for the default process heap.
    • Use HeapCreate() to create a new, private heap with specific options and sizes.
  2. Allocate Memory: Use HeapAlloc() with the obtained handle to allocate memory blocks from that heap.
  3. Reallocate Memory (if needed): Use HeapReAlloc() to resize previously allocated blocks.
  4. Get Block Size (if needed): Use HeapSize() to determine the actual size of an allocated block.
  5. Free Memory: Use HeapFree() to deallocate memory blocks when they are no longer needed. Pass the correct heap handle and the pointer returned by HeapAlloc/HeapReAlloc.
  6. Destroy Heap (for private heaps): When a private heap created with HeapCreate is entirely finished with, use HeapDestroy() to release all its resources. Do *not* call this on the default process heap handle.

It is generally advised not to mix Windows heap API calls (HeapAlloc, etc.) with standard C runtime library functions (malloc, free) for the same memory block. Memory allocated with HeapAlloc should be freed with HeapFree; memory allocated with malloc should be freed with free.

Topic 69: Working with Heaps

This topic illustrates the use of multiple heaps with a conceptual example: sorting data from a file.

The example uses three heaps (as shown in the diagram "Figure 5-2 Memory Management in Multiple Heaps"):

  1. ProcHeap (Process Heap): The default process heap obtained via GetProcessHeap(). Used to store the root pointer (pRoot) of the data structure (e.g., a tree or index).
  2. RecHeap (Record Heap): A private heap created using HeapCreate(). Used specifically to store the actual data records (keys) read from the file.
  3. NodeHeap (Node Heap): Another private heap created using HeapCreate(). Used specifically to store the nodes of the data structure (e.g., tree nodes) that contain pointers or references to the records stored in `RecHeap`.

Rationale:

  • Storing the relatively small tree nodes in `NodeHeap` and the potentially larger data records in `RecHeap` separates allocations based on size and type, which can improve locality and reduce fragmentation compared to allocating everything from one heap.
  • Sorting or indexing operations might primarily manipulate the `NodeHeap` (e.g., rearranging pointers in tree nodes). Accessing the actual data in `RecHeap` might be less frequent during the sorting process itself.
  • At the end of the operation, the private heaps `RecHeap` and `NodeHeap` can be efficiently destroyed using `HeapDestroy()`, freeing all associated records and nodes instantly without needing to traverse the data structure and free each element individually.
  • The `ProcHeap` (default process heap) is not destroyed, as it's managed by the system. Only the small root pointer allocated from it would need separate handling if necessary (though often it's just a local variable or part of a larger structure managed elsewhere).

This demonstrates how using multiple heaps tailored to specific data structures or allocation patterns can optimize performance and simplify cleanup in certain application scenarios.

Topic 70: Binary Search Using Heaps

This topic describes an example program (`sortBT`) that sorts records from input files using a binary search tree built in memory, leveraging multiple heaps for memory management.

Concept:

  • Uses two private heaps: one for tree nodes (`hNode`) and one for the actual data records (`hData`).
  • Reads input files record by record. Each record is assumed to have a fixed-length key (e.g., 8 characters) at the beginning, followed by variable-length data.
  • Builds a binary search tree based on the keys. Each node in the tree (allocated from `hNode`) contains the key and a pointer to the full record data (allocated from `hData`).
  • Uses recursive functions (`InsertTree`, `Scan`) to build and traverse the tree.
  • After building the tree for a file, performs an in-order traversal (`Scan`) to print the records in sorted key order.
  • Uses Structured Exception Handling (SEH) to catch potential memory allocation errors from the heaps.
  • Uses `HeapDestroy` to efficiently free all nodes and records associated with a file after processing.

Program Structure (`sortBT` Example)

Key definitions and functions from the example:

// Function Prototypes

    HANDLE hIn = INVALID_HANDLE_VALUE, hNode = NULL, hData = NULL;
    LPTNODE pRoot = NULL;
    BOOL noPrint = FALSE; // Option flag
    // CHAR errorMessage[256]; // Used in _except block
    int iFirstFile;

    // Parse options (e.g., -n for no print)
    iFirstFile = Options (argc, argv, _T ("n"), &noPrint, NULL);
    if (argc <= iFirstFile) {
        ReportError (_T ("Usage: sortBT [options] file1 [file2 ...]"), 1, FALSE);
    }

    /* Process all files on the command line. */
    for (iFile = iFirstFile; iFile < argc; iFile++) {
         // Use SEH for file errors and heap errors within this loop iteration
        __try { // Outer try for file open / general errors per file
            /* Open the input file. */

                // Raise custom exception for file open error
                RaiseException (STATUS_FILE_ERROR, 0, 0, NULL);

            // Inner try/finally to ensure heaps/handles are cleaned up
            __try {
                /* Allocate the two growable heaps. Enable exceptions. */
                hNode = HeapCreate (HEAP_GENERATE_EXCEPTIONS | HEAP_NO_SERIALIZE, NODE_HEAP_ISIZE, 0);
                hData = HeapCreate (HEAP_GENERATE_EXCEPTIONS | HEAP_NO_SERIALIZE, DATA_HEAP_ISIZE, 0);
                 // Check if HeapCreate failed (though unlikely with HEAP_GENERATE_EXCEPTIONS)

                /* Process the input file, creating the tree. */
                pRoot = FillTree (hIn, hNode, hData);

                /* Display the tree in key order. */
                if (!noPrint) {
                    _tprintf (_T("Sorted file: %s\n"), argv[iFile]);
                    Scan (pRoot); // In-order traversal
                }
            } __finally { /* Heaps and file handle are always closed/destroyed */
                /* Destroy the two private heaps and their data structures. */
                if (hNode != NULL) HeapDestroy (hNode);
                if (hData != NULL) HeapDestroy (hData);
                hNode = NULL; hData = NULL; // Reset handles after destroying
                pRoot = NULL; // Root is invalid now

                if (hIn != INVALID_HANDLE_VALUE) CloseHandle (hIn);
                hIn = INVALID_HANDLE_VALUE; // Reset handle after closing
            } // End inner __try/__finally

        } __except ( (GetExceptionCode() == STATUS_FILE_ERROR || GetExceptionCode() == STATUS_NO_MEMORY)
                     ? EXCEPTION_EXECUTE_HANDLER : EXCEPTION_CONTINUE_SEARCH)
        { // Handle expected exceptions (file open, out of memory)
            TCHAR errorMessage[256];
            _stprintf_s (errorMessage, 256, _T("\nError processing file: %s"), argv[iFile]);
            ReportError (errorMessage, 0, TRUE); // Report error, print GetLastError, don't exit loop
        } // End outer __try/__except
    } // End loop over files

    return 0;
}

Topic 71: Memory-Mapped Files

Memory Mapping Concept

Dynamic memory (like the heap) is typically allocated from the system's paging file, with the OS managing the mapping between virtual addresses used by the process and physical RAM or the paging file on disk.

Memory-Mapped Files provide a different mechanism: they allow a process to map a portion or the entirety of a normal file directly into its virtual address space. The OS then manages the loading of data from the file into physical memory pages as the process accesses the corresponding virtual addresses.

Advantages of Memory Mapped I/O

  • Direct Memory Access: Eliminates the need for explicit file I/O operations like ReadFile and WriteFile. The file contents can be accessed directly via pointers as if they were already in memory.
  • Performance: Can be significantly faster than traditional buffered I/O for certain access patterns (especially random access or processing large files) because it avoids extra buffer copying between user space and kernel space. The OS handles paging directly from the file.
  • Simplified Algorithms: Allows using standard in-memory algorithms (for sorting, searching, data structure manipulation) directly on file data without manually reading the file into separate memory buffers.
  • Large File Handling: Provides a way to access files larger than available physical RAM, as only the accessed portions (pages) need to be loaded into memory at any time.
  • Inter-Process Communication (IPC): Multiple processes can map the same file mapping object (often identified by a name) into their address spaces, providing an efficient mechanism for sharing large amounts of data.
  • Reduced Paging File Usage: Data pages come directly from the mapped file, not the system paging file (unless using `PAGE_WRITECOPY`).

Other Considerations

  • Memory mapping is used internally by Windows for loading Dynamic Link Libraries (DLLs) and executable (EXE) files into process memory.
  • When working with memory-mapped files, it's crucial to handle potential exceptions, especially EXCEPTION_IN_PAGE_ERROR (status code 0xC0000006). This SEH exception occurs if the system encounters an I/O error while trying to read a required page from the underlying file into memory when the mapped address is accessed. Robust code should use `__try`/`__except` to catch these errors.

Topic 72: File Mapping Objects

To perform memory-mapped file I/O, you first need to create a file mapping object (also known as a section object in kernel terms). This object represents the connection between a file on disk (or the paging file) and a potential mapping in memory.

The primary API for this is CreateFileMapping():

HANDLE CreateFileMapping(
  HANDLE                hFile,                   // Handle to the file to map
  LPSECURITY_ATTRIBUTES lpFileMappingAttributes, // Security attributes (optional)
  DWORD                 flProtect,               // Protection flags for the mapping
  DWORD                 dwMaximumSizeHigh,       // High 32 bits of mapping size
  DWORD                 dwMaximumSizeLow,        // Low 32 bits of mapping size
  LPCTSTR               lpName                   // Name for sharing (optional)
);
  • hFile: A handle to the open file on disk that you want to map. This handle must have been opened with access rights compatible with the `flProtect` flags requested. To map the system paging file instead of a specific disk file (for anonymous shared memory), set this to INVALID_HANDLE_VALUE.
  • lpFileMappingAttributes: Optional pointer to a SECURITY_ATTRIBUTES structure, allowing control over handle inheritance and security descriptor. Usually NULL for default attributes.
  • flProtect: Specifies the memory protection for the mapping when a view is created. It also implies required access rights on the `hFile`. Common values:
    • PAGE_READONLY: Allows views to be mapped with read-only access. `hFile` must have at least GENERIC_READ access.
    • PAGE_READWRITE: Allows views to be mapped with read/write access. `hFile` must have both GENERIC_READ and GENERIC_WRITE access. Changes written to the view are eventually written back to the disk file by the memory manager.
    • PAGE_WRITECOPY: Allows views to be mapped with copy-on-write access. `hFile` must have GENERIC_READ and GENERIC_WRITE access. When a process writes to the view, a private copy of the page is created using the paging file; the original disk file is not modified.
    • Can be combined with section attributes like SEC_COMMIT, SEC_IMAGE, SEC_NOCACHE, SEC_RESERVE.
  • dwMaximumSizeHigh / dwMaximumSizeLow: Specify the 64-bit maximum size (in bytes) of the file mapping object.
    • If these are both 0, the mapping object's size is set to the current size of the file specified by `hFile`. The file must not be empty.
    • If mapping the paging file (`hFile` is `INVALID_HANDLE_VALUE`), these *must* specify a non-zero size for the memory region.
    • If mapping a disk file but specifying a size larger than the current file, the disk file is automatically extended to this size (requires `GENERIC_WRITE` on `hFile`).
    • **Important:** A file mapping object cannot be resized after creation. If the underlying file grows beyond the specified maximum size, only the initially specified size can be mapped.
  • lpName: An optional null-terminated string specifying a name for the file mapping object. If named, other processes can open this same mapping object using OpenFileMapping(). If NULL, the object is unnamed and cannot be shared by name (though it could potentially be shared via handle inheritance).
  • **Return Value:** Returns a handle to the newly created file mapping object on success, or `NULL` on failure (check `GetLastError`). If the object already exists with the specified name, it returns a handle to the existing object, and `GetLastError()` returns `ERROR_ALREADY_EXISTS`.

Topic 73: Open Existing File Mapping Objects

If a file mapping object was created with a name (using the `lpName` parameter in CreateFileMapping), other processes can gain access to that same mapping object using the OpenFileMapping() API.

HANDLE OpenFileMapping(
  DWORD   dwDesiredAccess, // Access desired for the mapping view
  BOOL    bInheritHandle,  // Should child processes inherit handle?
  LPCTSTR lpName           // Name of the mapping object to open
);
  • dwDesiredAccess: Specifies the access rights required for the subsequent call to MapViewOfFile. Common values are FILE_MAP_READ, FILE_MAP_WRITE, FILE_MAP_COPY, FILE_MAP_ALL_ACCESS. This requested access must be compatible with the protection specified when the mapping object was created with CreateFileMapping.
  • bInheritHandle: If TRUE, handles created by child processes can inherit this handle.
  • lpName: The null-terminated string name of the file mapping object to open (must match the name used in CreateFileMapping).
  • **Return Value:** Returns an open handle to the existing file mapping object on success, or `NULL` on failure (e.g., if the name does not exist or access is denied). Check `GetLastError`.

Topic 74: Mapping Objects to Process Address Space

Once you have a handle to a file mapping object (from either CreateFileMapping or OpenFileMapping), the next step is to map all or part of that object into your process's virtual address space. This creates a "view" of the file mapping.

The API used is MapViewOfFile():

LPVOID MapViewOfFile(
  HANDLE hFileMappingObject,    // Handle to the file mapping object
  DWORD  dwDesiredAccess,       // Type of access to the mapped view
  DWORD  dwFileOffsetHigh,      // High 32 bits of file offset where view starts
  DWORD  dwFileOffsetLow,       // Low 32 bits of file offset where view starts
  SIZE_T dwNumberOfBytesToMap   // Number of bytes to map (0 = entire mapping)
);
  • hFileMappingObject: The handle returned by CreateFileMapping or OpenFileMapping.
  • dwDesiredAccess: Specifies the access type for the mapped view. Must be compatible with the `flProtect` flags used when creating the mapping object. Common flags:
    • FILE_MAP_READ: Read access. Requires mapping created with PAGE_READONLY or PAGE_READWRITE.
    • FILE_MAP_WRITE: Read and write access. Requires mapping created with PAGE_READWRITE.
    • FILE_MAP_COPY: Copy-on-write access. Requires mapping created with PAGE_WRITECOPY.
    • FILE_MAP_ALL_ACCESS: Equivalent to the access specified in `flProtect` during creation.
    • FILE_MAP_EXECUTE: Execute access (requires `PAGE_EXECUTE_` protection flags).
  • dwFileOffsetHigh / dwFileOffsetLow: Specify the 64-bit offset within the file mapping object where the view should begin. This offset **must** be a multiple of the system's memory allocation granularity (obtainable via GetSystemInfo, usually 64KB). Setting both to 0 starts the view at the beginning of the mapping object.
  • dwNumberOfBytesToMap: Specifies the number of bytes of the mapping object to map into the view.
    • If 0, the view extends from the specified offset to the end of the file mapping object.
    • If non-zero, maps the specified number of bytes.
  • **Return Value:** On success, returns the starting virtual address (LPVOID pointer) where the file view is mapped in the process's address space. On failure, returns NULL (check GetLastError).

Unmapping the View (`UnmapViewOfFile`)

When access to the mapped view is no longer needed, it must be unmapped to release the portion of the process's address space it occupies.

BOOL UnmapViewOfFile(LPCVOID lpBaseAddress);
  • lpBaseAddress: The base address pointer returned by a previous successful call to MapViewOfFile.
  • Returns TRUE on success, FALSE on failure.
  • Unmapping the view does not close the file mapping object handle or the original file handle.

Closing Handles

After unmapping all views, the file mapping object handle (from CreateFileMapping/OpenFileMapping) should be closed using CloseHandle(). Finally, the original file handle (from CreateFile) should also be closed using CloseHandle().

Topic 75: More About File Mapping

File Mapping Limitations

  • Win32 Size Limit: On 32-bit Windows, the available user-mode virtual address space (typically 2GB or 3GB) limits the total size of all memory mappings a process can create. Mapping very large files (> 2-3 GB) might fail or be impossible. Win64 alleviates this significantly.
  • Mapping Size Fixed: The maximum size of a file mapping object is fixed when created by CreateFileMapping. It cannot be extended later. If the underlying file grows beyond this size, the extended portion cannot be accessed via the existing mapping object.
  • Advance Size Knowledge: You generally need to know the maximum size required for the mapping in advance, especially if the file might grow or if mapping the paging file.

File Mapping Procedure Summary

  1. Open the target file using CreateFile with appropriate access rights (at least GENERIC_READ, potentially GENERIC_WRITE depending on desired mapping protection).
  2. (If creating a new file or ensuring minimum size) Optionally set the file size using SetFilePointerEx followed by SetEndOfFile.
  3. Create the file mapping object using CreateFileMapping, specifying compatible protection flags, the maximum size, and an optional name for sharing. OR Open an existing mapping using OpenFileMapping.
  4. Create one or more views of the mapping object in the process address space using MapViewOfFile, specifying desired access, the starting offset (multiple of allocation granularity), and the number of bytes to map.
  5. Access the file data directly through the pointer returned by MapViewOfFile as if it were an array in memory.
  6. (Optional) To change mapped regions or apply changes, unmap the current view using UnmapViewOfFile, potentially flush changes using FlushViewOfFile (see Topic 74 image context), and then map a new view using MapViewOfFile with different offset/size parameters.
  7. Use SEH (__try/__except) to catch potential EXCEPTION_IN_PAGE_ERROR exceptions that can occur during memory access to the mapped view if the OS encounters disk I/O errors.
  8. When finished, unmap all views using UnmapViewOfFile.
  9. Close the file mapping object handle using CloseHandle.
  10. Close the original file handle using CloseHandle.

Topic 76: Sequential File Access Using file mapping

Accessing file data sequentially using memory mapping can offer significant performance advantages over conventional file I/O (ReadFile/WriteFile) for certain tasks, potentially reducing processing time considerably (the OCR suggests a 3-fold reduction, though this is highly dependent on the specific task and system).

The example described (`cci_fMM.c`) implements the simple Caesar cipher (byte-wise shift) using memory mapping for both input and output files.

Caesar Cipher with Memory Mapping (`cci_fMM.c`):

  1. Open Input File: Open the source file (`fIn`) with GENERIC_READ access using CreateFile.
  2. Get Input File Size: Get the exact size using GetFileSizeEx. Check for errors and potentially for sizes too large for mapping on Win32.
  3. Create Input Mapping: Create a read-only file mapping object (`hInMap`) for the input file using CreateFileMapping with PAGE_READONLY protection and the obtained file size.
  4. Map Input View: Map a view (`pInFile`) of the entire input mapping object using MapViewOfFile with FILE_MAP_READ access.
  5. Create/Open Output File: Create or open the destination file (`fOut`) with GENERIC_READ | GENERIC_WRITE access using CreateFile (e.g., with CREATE_ALWAYS).
  6. Create Output Mapping: Create a read-write file mapping object (`hOutMap`) for the output file using CreateFileMapping with PAGE_READWRITE protection. **Crucially**, specify the same size as the input file. This ensures the output file is large enough and potentially extends it if necessary.
  7. Map Output View: Map a view (`pOutFile`) of the entire output mapping object using MapViewOfFile with FILE_MAP_WRITE access.
  8. Process Data: Use pointers (`pIn`, `pOut` initialized to `pInFile`, `pOutFile`) to iterate through the mapped input view byte by byte. For each byte, calculate the ciphered byte (`*pIn + cShift`) and write it to the corresponding position in the mapped output view (`*pOut = ...`). Increment both pointers. This loop runs entirely in memory. Use SEH (`__try`/`__except`) around this loop to catch potential `EXCEPTION_IN_PAGE_ERROR`.
  9. Cleanup: Regardless of success or failure (using `__finally` or careful error handling):
    • Unmap the output view (`UnmapViewOfFile(pOutFile)`).
    • Unmap the input view (`UnmapViewOfFile(pInFile)`).
    • Close the output mapping handle (`CloseHandle(hOutMap)`).
    • Close the input mapping handle (`CloseHandle(hInMap)`).
    • Close the output file handle (`CloseHandle(hOut)`).
    • Close the input file handle (`CloseHandle(hIn)`).
    • Optionally delete the output file if the operation did not complete successfully.

This approach replaces file read/write loops with direct memory manipulation, leveraging the OS's paging mechanism for efficient data transfer.

Topic 77: Sorting a Memory-Mapped File

Another significant advantage of memory mapping is the ability to apply standard memory-based algorithms directly to file contents. Sorting a file's records in memory is generally much simpler and often faster than implementing disk-based sorting algorithms.

The example program (`sortFL`) sorts a file containing fixed-length records, assuming an 8-byte key at the beginning of each record.

Sorting with Memory Mapping (`sortFL`):

  1. Create Temporary Copy: Copy the original input file to a temporary file (e.g., using CopyFile). This is done because `qsort` modifies the data in place, and we want to leave the original file unchanged.
  2. Open Temporary File: Open the temporary file with read-write access (`GENERIC_READ | GENERIC_WRITE`).
  3. Get File Size: Get the size of the temporary file using GetFileSizeEx.
  4. Create Mapping: Create a read-write file mapping object (`hMap`) for the temporary file using CreateFileMapping with PAGE_READWRITE and the file size.
  5. Map View: Map a view (`pFile`) of the entire temporary file using MapViewOfFile with FILE_MAP_ALL_ACCESS (or `FILE_MAP_WRITE`).
  6. Sort In Memory: Call the standard C library function qsort():
    • Pass the base address of the mapped view (`pFile`) as the array base.
    • Pass the number of records (calculated as `fileSize / recordSize`).
    • Pass the size of each record (`recordSize`).
    • Pass a pointer to a custom comparison function (`KeyCompare`) that knows how to compare two records based on the key field.
    qsort rearranges the records directly within the mapped memory view.
  7. Output Sorted Data: Iterate through the now-sorted mapped view (`pFile`) and print each record to standard output (e.g., using `_tprintf`). (The OCR notes a null character might be appended to the mapped view in memory before printing, which requires mapping slightly larger than the file size – handle with care).
  8. Cleanup:
    • Unmap the view (`UnmapViewOfFile(pFile)`).
    • Close the mapping handle (`CloseHandle(hMap)`).
    • Close the temporary file handle (`CloseHandle(hFile)`).
    • Delete the temporary file (`DeleteFile(tempFile)`).

This technique leverages memory mapping and standard library functions (`qsort`) to perform efficient in-memory sorting on file data.

Topic 78: Using Base Pointer

A significant challenge arises when storing data structures containing pointers within memory-mapped files. If the file is unmapped and then re-mapped later (possibly in a different process or after a system reboot), the virtual address returned by MapViewOfFile for the base of the mapping may be different from the original address.

This means any absolute virtual address pointers stored within the mapped data structure become invalid ("meaningless") because they point to incorrect locations relative to the new mapping base address.

The solution is to avoid storing absolute pointers and instead store offsets relative to a known base address (typically the start of the mapped view). Microsoft C/C++ provides a language extension (also found in some other compilers) called based pointers to facilitate this.

Based Pointer Syntax (Microsoft C/C++):

type __based( base_variable ) * declarator;
  • type: The data type the pointer points to.
  • __based(base_variable): Specifies that the pointer `declarator` is an offset relative to the address stored in `base_variable`. `base_variable` itself must be a pointer (or resolve to one).
  • * declarator: The name of the based pointer being declared.

Examples from OCR:

// Assume pInFile is the LPVOID base address from MapViewOfFile
LPTSTR pInFile = NULL;

// pSize is a 32-bit offset from the address in pInFile, pointing to a DWORD
DWORD __based( pInFile ) * pSize;

// pIn is a 32-bit offset from the address in pInFile, pointing to a TCHAR
TCHAR __based( pInFile ) * pIn;

When you use `pSize` or `pIn`, the compiler automatically generates code to add the stored offset value to the current base address in `pInFile` to calculate the actual memory address to access. This makes the stored offsets portable across different mappings of the same file data.

The syntax requires the explicit *, which differs from typical Windows typedef conventions (like LPDWORD). A `typedef` could be used to hide this: typedef DWORD __based(pInFile) * BP_DWORD; BP_DWORD pSize;

Based pointers typically store 32-bit offsets. For 64-bit offsets needed for very large files/mappings, alternative techniques like storing ULONGLONG offsets and manually adding them to the 64-bit base address might be required.

Topic 79: Base Pointer Example

This topic describes an example program (`sortMM`) that builds a persistent index for a data file containing variable-sized records, using based pointers within the index file.

Problem: Sorting large files directly each time they are accessed is inefficient. We need a way to maintain a sorted index that persists between program runs.

Solution using Based Pointers and Memory Mapping:

  1. Data File (`MyFile`): Contains the original variable-sized records. Assume each record starts with a fixed-size key (e.g., 8 characters).
  2. Index File (`MyFile.idx`): A separate file created to hold the index. Each entry in the index file corresponds to a record in the data file and contains:
    • The fixed-size key of the data record.
    • A based pointer (`P` in the diagram) storing the offset of the corresponding data record within the `MyFile`. This offset is relative to the start of the `MyFile` mapping.
  3. Process:
    • Index Creation:
      • Map the original data file (`MyFile`) into memory (read-only).
      • Create and map the index file (`MyFile.idx`) into memory (read-write).
      • Scan through the mapped `MyFile`, identifying each record's key and its starting offset relative to the beginning of the `MyFile` mapping.
      • For each record, write an entry to the mapped `MyFile.idx` containing the key and the calculated offset (stored as the based pointer value).
    • Sorting:
      • Map the index file (`MyFile.idx`) into memory.
      • Use `qsort` on the mapped index file view, providing a comparison function that compares the key fields of the index records. This sorts the index entries by key.
      • The based pointers (offsets) within the index entries are rearranged along with their corresponding keys.
    • Searching/Accessing:
      • To find a record by key: Perform a binary search (or other efficient search) on the sorted, mapped index file (`MyFile.idx`).
      • Once the matching index entry is found, retrieve the based pointer (offset) value from it.
      • Map the original data file (`MyFile`) into memory.
      • Calculate the absolute virtual address of the data record by adding the retrieved offset to the base address of the `MyFile` mapping.
      • Access the data record directly at the calculated address.

This approach keeps the original data file unchanged, creates a sortable index, and uses based pointers (offsets) within the index file to ensure the links to the data file remain valid even if the mapping base addresses change in subsequent runs.

Topic 80: Dynamic Link Library (DLLs)

Dynamic Link Libraries (DLLs) are a fundamental feature of Windows, enabling code sharing and modularity. They rely heavily on memory mapping internally.

Static Linking (Conventional Approach)

The traditional method involves gathering all application source code and required library functions into a single, self-contained executable file during the linking phase of compilation. While simple, this has disadvantages:

  • Large Executables: Executables contain copies of all linked library code, increasing disk space usage and potentially physical memory requirements (though shared code pages help).
  • Update Complexity: If a shared library function needs an update (e.g., bug fix, security patch), every application statically linked against it must be recompiled and relinked.
  • Resource Usage: Multiple running programs statically linked against the same library will each have their own copy of the library code loaded into memory (unless the OS can share identical code pages).
  • Reduced Portability/Flexibility: An application might be tied to specific library versions compiled into it.

How DLLs Solve These Problems

DLLs use **dynamic linking**, where library functions are not combined into the executable at compile time. Instead, linking occurs later:

  • Implicit Linking (Load Time): The OS loader connects the application to the required DLL functions when the application starts.
  • Explicit Linking (Run Time): The application explicitly loads the DLL and retrieves function addresses using APIs like LoadLibrary and GetProcAddress.

Benefits of DLLs:

  • Smaller Executables: Application executables are smaller as they don't contain the library code itself.
  • Code Sharing: A single copy of the DLL code is loaded into physical memory by the OS and mapped into the virtual address space of all processes using it. This saves memory.
  • Easier Updates: A DLL can be updated (e.g., patched) independently. As long as the interface remains compatible, applications using it will automatically benefit from the update without needing recompilation.
  • Modularity: Encourages breaking applications into logical, reusable components.
  • Per-Process Globals: While the DLL code is shared, each process using the DLL gets its own copy of the DLL's global and static variables, ensuring data isolation between processes.

Importance of DLLs

DLLs are crucial in Windows:

  • The entire Windows API is implemented as a set of DLLs (e.g., `kernel32.dll`, `user32.dll`, `gdi32.dll`). Applications call these DLLs to interact with the OS kernel.
  • They enable code sharing between multiple processes.
  • DLL code runs within the address space of the calling process, using its resources (handles, thread stack).
  • DLLs must be written in a thread-safe manner if they are to be used by multi-threaded applications.
  • DLLs can export functions (entry points) and variables for use by other modules.
  • Technologies like COM (Component Object Model) heavily rely on DLLs.

Topic 81: Implicit Linking

Implicit Linking, also known as load-time dynamic linking, is the more common and straightforward technique for using DLLs.

Process:

  1. DLL Creation: The source code for the functions intended to be shared is compiled and linked into a DLL file (e.g., `MyLibrary.dll`). During this process, the linker also generates an associated **Import Library** file (e.g., `MyLibrary.lib`).
  2. Import Library (.LIB): This `.lib` file does *not* contain the actual function code. Instead, it contains information (stubs or placeholders) that tells the linker how to reference the functions exported by the DLL.
  3. Application Linking: When building the application executable (e.g., `MyApp.exe`) that uses the DLL functions, the developer links `MyApp.exe` against the import library (`MyLibrary.lib`). The linker uses the information in the import library to resolve the external function calls made by `MyApp.exe`, embedding references to the DLL and the functions within it into the executable.
  4. Application Loading: When `MyApp.exe` is launched, the Windows OS loader inspects the executable's import table. It sees the reference to `MyLibrary.dll`.
  5. DLL Loading & Resolution: The loader finds `MyLibrary.dll` (searching standard locations like the application directory, system directories, etc.), loads it into the process's address space (if not already loaded for another process), and then patches the calls within `MyApp.exe` to point directly to the actual addresses of the functions inside the loaded `MyLibrary.dll`.

From the application developer's perspective, once linked against the import library, calling a DLL function looks the same as calling a statically linked function. The complexity is handled by the linker and the OS loader.

Placement:

  • The import library (`.lib`) needs to be available to the linker during the application build process.
  • The DLL file (`.dll`) needs to be available to the loader at runtime, typically placed in the same directory as the application executable or in a directory included in the system's search path.

Function interfaces (exported names, calling conventions, parameters) defined in DLLs must be managed carefully to ensure compatibility between the DLL and the applications using it.

Topic 82: Exporting and Importing Interfaces

For functions within a DLL to be callable by external applications, they must be explicitly **exported** from the DLL. Similarly, the calling application must **import** the functions.

Exporting from a DLL

This can be achieved in primarily two ways:

  1. Module Definition File (.DEF): A text file linked with the DLL project that lists the functions (and variables) to be exported, potentially assigning ordinal numbers.
  2. __declspec(dllexport): A Microsoft-specific storage modifier placed directly in the function's declaration within the DLL's source code. 
    // In the DLL's source code (e.g., MyLibrary.h or MyLibrary.c)
    __declspec(dllexport) DWORD MyFunction (/* parameters */);
    Using __declspec(dllexport) instructs the compiler and linker to generate the necessary export information. This also helps generate the corresponding import library (`.LIB`) file.

Importing into an Application

When implicitly linking, the calling application needs to declare the DLL functions using the __declspec(dllimport) storage modifier. This tells the compiler that the function resides in an external DLL and helps optimize the function call mechanism.

// In the application's source code (e.g., MyApp.c, often via including MyLibrary.h)
    __declspec(dllimport) DWORD MyFunction (/* parameters */);

Convenience Macro

To simplify managing declarations for both building the DLL and using it, a common technique is to use a preprocessor macro in the DLL's header file:

// In the DLL's header file (e.g., MyLibrary.h)
    
    #ifdef MYLIBRARY_EXPORTS  // Define this only when BUILDING the DLL
        #define LIBSPEC __declspec(dllexport)
    #else                      // For applications USING the DLL
        #define LIBSPEC __declspec(dllimport)
    #endif
    
    // Optional: Handle C++ name mangling
    #ifdef __cplusplus
    extern "C" {
    #endif
    
    // Declare the exported function using the macro
    LIBSPEC DWORD MyFunction(/* parameters */);
    
    #ifdef __cplusplus
    } // extern "C"
    #endif

When compiling the DLL, define `MYLIBRARY_EXPORTS`. When compiling the application using the DLL, do not define it. The `extern "C"` block prevents C++ name mangling if the DLL is intended to be used by C or other languages.

DLL Search Order

When an application implicitly or explicitly loads a DLL without a full path, Windows searches for the DLL in a specific order (the "safe" search order is default on modern systems):

  1. The directory containing the application executable.
  2. The system directory (e.g., `C:\Windows\System32`). Use GetSystemDirectory() to find this path.
  3. The 16-bit system directory (rarely relevant now).
  4. The Windows directory (e.g., `C:\Windows`). Use GetWindowsDirectory() to find this path.
  5. The current working directory.
  6. Directories listed in the `PATH` environment variable.

(Note: This order can be modified by manifests, `SetDllDirectory`, or `AddDllDirectory`). Placing the DLL in the same directory as the application executable is the simplest approach.

Topic 83: Explicit Linking

Explicit Linking (also known as run-time dynamic linking) provides more control over when DLLs are loaded and unloaded. The application explicitly calls APIs to load the library, get addresses of functions within it, and then unload the library when done.

With explicit linking, the application does not link against an import library (`.LIB`) file during its build process. Function calls are made indirectly through function pointers obtained at runtime.

Required Functions:

  1. LoadLibrary() / LoadLibraryEx(): Loads the specified DLL into the address space of the calling process. 
    HMODULE LoadLibrary(LPCTSTR lpLibFileName);
    HMODULE LoadLibraryEx(LPCTSTR lpLibFileName, HANDLE hFile, DWORD dwFlags);
    • lpLibFileName: Name or full path of the DLL file. The `.DLL` extension is optional.
    • LoadLibraryEx provides additional `dwFlags` for controlling search paths (e.g., LOAD_LIBRARY_SEARCH_DLL_LOAD_DIR) and behavior.
    • Returns a module handle (HMODULE, which is the base address where the DLL is loaded) on success, or `NULL` on failure.
  2. GetProcAddress(): Retrieves the address of an exported function (or variable) inside the loaded DLL. 
    FARPROC GetProcAddress(HMODULE hModule, LPCSTR lpProcName);
    • hModule: The module handle returned by `LoadLibrary` or `LoadLibraryEx`.
    • lpProcName: The name of the function to retrieve (case-sensitive). This **must** be an ANSI string (`LPCSTR`), even in Unicode builds, unless retrieving by ordinal. Alternatively, you can retrieve by ordinal number using `MAKEINTRESOURCE(ordinal)`.
    • Returns a function pointer (`FARPROC`, which needs casting to the correct type) on success, or `NULL` if the function is not found.
  3. FreeLibrary(): Decrements the reference count of the loaded DLL. When the count reaches zero, the DLL is unmapped from the process address space. 
    BOOL FreeLibrary(HMODULE hModule);
    • hModule: The module handle returned by `LoadLibrary`.
    • Returns `TRUE` on success, `FALSE` on failure.

The application declares a function pointer variable of the correct type, assigns the result of `GetProcAddress` (after casting) to it, and then calls the DLL function using the function pointer.

Topic 84: Explicit Linking a file conversion function

This topic shows an example of explicitly linking and calling a function, presumably the Caesar cipher function (`cci_f`) from Topic 76, assuming it has been compiled into a DLL.

Topic 85: DLL Entry Point (`DllMain`)

When creating a DLL, you can optionally specify an entry-point function, conventionally named DllMain. This function is called by the system under specific circumstances, allowing the DLL to perform initialization or cleanup operations.

DllMain is invoked when:

  • A process loads the DLL (Process Attach).
    • Implicit linking: At process startup.
    • Explicit linking: When LoadLibrary/LoadLibraryEx is called successfully.
  • A process unloads the DLL (Process Detach).
    • Implicit linking: At process termination.
    • Explicit linking: When FreeLibrary causes the DLL's reference count to drop to zero, or during process termination if still loaded.
  • A new thread is created within a process that has already loaded the DLL (Thread Attach).
  • A thread terminates within a process while the DLL is still loaded (Thread Detach).

(Note: Calls for thread attach/detach can be disabled for performance using DisableThreadLibraryCalls()).

The LoadLibraryEx function also has flags that can suppress the call to DllMain for process attach.

`DllMain` Signature

BOOL WINAPI DllMain(
        HINSTANCE hinstDLL,     // Handle to the DLL module instance
        DWORD     fdwReason,    // Reason for calling the function
        LPVOID    lpvReserved   // Further specifies attach/detach type
    );
  • hinstDLL: The base address where the DLL is loaded in memory (same as the `HMODULE` returned by `LoadLibrary`).
  • fdwReason: Indicates why the function is being called. One of:
    • DLL_PROCESS_ATTACH: The DLL is loading into the process's virtual address space. Perform process-specific initialization here.
    • DLL_THREAD_ATTACH: A new thread is starting in the current process. Perform thread-specific initialization here.
    • DLL_THREAD_DETACH: A thread is exiting cleanly. Perform thread-specific cleanup here.
    • DLL_PROCESS_DETACH: The DLL is unloading from the process's virtual address space. Perform process-specific cleanup here.
  • lpvReserved:
    • For DLL_PROCESS_ATTACH: Non-NULL indicates a static/implicit load; NULL indicates a dynamic/explicit load via `LoadLibrary`.
    • For DLL_PROCESS_DETACH: Non-NULL indicates `FreeLibrary` was called; NULL indicates the process is terminating.
  • **Return Value:** Should return TRUE on success for DLL_PROCESS_ATTACH. If it returns FALSE, the process initialization fails (if implicitly linked) or LoadLibrary fails. Return values for other reasons are ignored.

Restrictions on `DllMain`

The code within DllMain must be very simple and fast. Because all calls to `DllMain` are serialized by the system using a global loader lock, complex operations can lead to deadlocks or performance issues.

Do NOT call functions within `DllMain` that might directly or indirectly:

  • Load other DLLs (e.g., LoadLibrary, LoadLibraryEx).
  • Perform blocking waits on synchronization objects.
  • Perform complex initialization or I/O operations.
  • Call functions in other DLLs that might violate these rules themselves.

DllMain should typically only be used for minimal, simple initialization/cleanup related to the DLL's state (e.g., initializing TLS - Thread Local Storage).

Topic 86: DLL Version Management

Complications ("DLL Hell")

The strength of DLLs (shared code, easy updates) can also lead to complications, sometimes referred to as "DLL Hell":

  • Interface Changes: If a new DLL version changes its exported functions (different parameters, removed functions), older applications linked against the previous interface will break when trying to load the new DLL.
  • Version Conflicts: An application needing features from a newer DLL might accidentally load an older version found earlier in the search path, causing unexpected behavior or crashes. Conversely, installing a new application might overwrite a shared DLL with a newer version that breaks older applications.

Version Management Strategies

  • Separate Directories: Using different installation directories for different application versions (less effective for shared system DLLs).
  • Versioned Filenames: Including version numbers directly in DLL and corresponding LIB filenames (e.g., `MyLib_1_0.dll`, `MyLib_1_0.lib`, `MyLib_2_0.dll`, `MyLib_2_0.lib`). Applications explicitly link (implicitly or explicitly) against the specific version they require.
  • Side-by-Side Assemblies (SxS): Microsoft's preferred approach. Applications declare their specific DLL dependencies (including version numbers) in an XML manifest file. The OS loader uses this manifest to find and load the correct version of the DLL, often from a shared assembly cache (`WinSxS` directory). This allows multiple versions of the same DLL to coexist on the system without conflict.

Querying DLL Information

DLLs can optionally export a function, typically named DllGetVersion, which applications can call (usually via explicit linking) to query the DLL's version information.

// Standard signature for DllGetVersion
    HRESULT CALLBACK DllGetVersion(DLLVERSIONINFO* pdvi);
    
    // Structure filled by DllGetVersion
    typedef struct _DLLVERSIONINFO {
      DWORD cbSize;          // Size of the structure (set by caller)
      DWORD dwMajorVersion;  // Major version number
      DWORD dwMinorVersion;  // Minor version number
      DWORD dwBuildNumber;   // Build number
      DWORD dwPlatformID;    // Platform identifier (e.g., DLLVER_PLATFORM_WINDOWS)
    } DLLVERSIONINFO;
  • The application allocates a `DLLVERSIONINFO` structure, sets `cbSize` to `sizeof(DLLVERSIONINFO)`, and passes a pointer to it when calling the DLL's `DllGetVersion` function.
  • The DLL fills the structure with its version details.
  • dwPlatformID could indicate compatibility (e.g., `DLLVER_PLATFORM_NT` for NT-based systems).

Topic 87: Windows Processes and Threads

Windows process management revolves around multitasking (running multiple programs concurrently), utilizing multiprocessor systems effectively, and using threads as the fundamental unit of execution.

Resources of Processes

From a programmer's view, a process is an executing program and owns resources:

  • Distinct Virtual Address Space: Each process operates within its own private virtual memory space, isolating it from other processes.
  • Code Segments: Contains the executable code for the application and any loaded DLLs. Code pages can be shared between processes using the same DLLs or executable image.
  • Data Segments: Contains global variables and static variables. Each process gets its own private copy of global variables, even those defined within shared DLLs.
  • Process Heap: The default heap used for dynamic memory allocation (see Topic 64).
  • Other Resources: Handles to system objects (files, synchronization objects, registry keys, etc.), environment variables (inherited from parent or set explicitly).
  • One or More Threads: A process must contain at least one thread to execute code.

Resources of Threads

A thread is the unit of execution scheduled by the OS kernel. It runs within the context of its owning process and shares the process's address space and most resources. However, each thread has its own unique resources:

  • Stack: Each thread has its own private stack used for function calls (parameters, return addresses), local (auto) variables, exception handling information, and interrupt handling.
  • Thread Context: A structure containing the state of the CPU registers (instruction pointer, stack pointer, general-purpose registers) specific to that thread. This context is saved/restored during context switches.
  • Thread Local Storage (TLS): An array-like mechanism allowing each thread to have its own private storage for data that needs to be unique per thread but accessible like global/static variables. Useful when library functions need to maintain state per calling thread without passing extra parameters.
  • Arguments: When created, a thread can receive a unique argument (often passed via the stack or a structure pointer).

Topic 88: Process Creation (`CreateProcess`)

The fundamental API for creating a new process in Windows is CreateProcess().

Unlike some OSes, Windows doesn't maintain explicit parent-child process structures within the kernel itself; the creating process is simply considered the parent. `CreateProcess` creates a new process with a single, primary thread.

The function has 10 parameters and returns a `BOOL` indicating success or failure. It does not return handles directly; instead, handles to the new process and its primary thread are returned via an output parameter structure (`PROCESS_INFORMATION`).

`CreateProcess` Signature

BOOL CreateProcess(
      LPCTSTR               lpApplicationName,      // Name of the module to execute (optional)
      LPTSTR                lpCommandLine,          // Command line string (mutable)
      LPSECURITY_ATTRIBUTES lpProcessAttributes,    // Process security attributes (optional)
      LPSECURITY_ATTRIBUTES lpThreadAttributes,     // Primary thread security attributes (optional)
      BOOL                  bInheritHandles,        // Should child inherit handles?
      DWORD                 dwCreationFlags,        // Process creation flags
      LPVOID                lpEnvironment,          // Environment block (optional)
      LPCTSTR               lpCurrentDirectory,     // Starting directory (optional)
      LPSTARTUPINFO         lpStartupInfo,          // STARTUPINFO structure
      LPPROCESS_INFORMATION lpProcessInformation    // Receives process/thread handles and IDs
    );

Key Parameters:

  • lpApplicationName / lpCommandLine: Specify the executable image and arguments (See Topic 90).
  • lpProcessAttributes / lpThreadAttributes: Optional security attributes for the new process and thread objects.
  • bInheritHandles: If TRUE, the new process inherits copies of any inheritable handles from the calling process (See Topic 91).
  • dwCreationFlags: Combines flags controlling process creation behavior, such as:
    • CREATE_SUSPENDED: Starts the primary thread in a suspended state. Requires a call to ResumeThread() using the thread handle from `lpProcessInformation` to start execution.
    • DETACHED_PROCESS: Creates a process without a console window. Cannot be combined with CREATE_NEW_CONSOLE.
    • CREATE_NEW_CONSOLE: Creates the process with a new console window, separate from the parent's.
    • CREATE_UNICODE_ENVIRONMENT: Indicates the environment block (`lpEnvironment`) is Unicode.
    • CREATE_NEW_PROCESS_GROUP: Makes the new process the root of a new process group (relevant for console signal handling, e.g., Ctrl+Break).
    • Additional flags control priority, window appearance, error handling, etc.
  • lpEnvironment: Optional pointer to a custom environment block for the new process. If NULL, inherits the parent's environment.
  • lpCurrentDirectory: Optional starting directory for the new process. If NULL, inherits the parent's current directory.
  • lpStartupInfo: Pointer to a STARTUPINFO structure, specifying main window appearance (title, position, size) and standard I/O handles (for redirection).
  • lpProcessInformation: Pointer to a PROCESS_INFORMATION structure that receives output from the function:
    • HANDLE hProcess: Handle to the new process.
    • HANDLE hThread: Handle to the primary thread of the new process.
    • DWORD dwProcessId: Process ID (PID) of the new process.
    • DWORD dwThreadId: Thread ID (TID) of the primary thread.

Important: The caller is responsible for closing the `hProcess` and `hThread` handles returned in `lpProcessInformation` using CloseHandle() when they are no longer needed. Closing the thread handle does not terminate the thread.

Topic 89: Why Processes need IDs and Handles

Processes in Windows are identified and managed using two primary mechanisms: Process IDs (PIDs) and Process Handles.

  • Process ID (PID):
    • A system-wide unique number assigned to a process when it is created.
    • Remains constant throughout the process's lifetime.
    • Becomes invalid once the process terminates.
    • The OS may reuse a PID for a new process after the original process has terminated.
    • Required by some management functions (e.g., certain debugging APIs, `OpenProcess`).
  • Process Handle (HANDLE):
    • A reference (obtained via `CreateProcess`, `OpenProcess`, or inheritance) to a kernel process object within the context of a specific process (the one holding the handle).
    • Required for most process management functions (e.g., `WaitForSingleObject`, `TerminateProcess`, `GetExitCodeProcess`).
    • Can have associated security attributes defining what operations the handle holder can perform on the target process.
    • A single process can have multiple handles open to it (potentially with different access rights) from various processes.
    • Handles consume system resources and **must** be closed using CloseHandle() when no longer needed to avoid resource leaks.

Processes operate independently. Although a child process initially inherits environment variables, the current directory, and potentially handles from its parent (via `CreateProcess`), subsequent changes in the parent process generally do not affect the child, and vice versa, unless explicit Inter-Process Communication (IPC) mechanisms are used.

Topic 90: Specifying the Executable Image and the Command Line

The CreateProcess API requires specifying the program to execute using either the `lpApplicationName` parameter or the `lpCommandLine` parameter, but not usually both.

  • Using `lpApplicationName` (Less Common):
    • Set `lpCommandLine` to the arguments only (excluding the program name itself).
    • `lpApplicationName` must **not** be `NULL`.
    • It should specify the full path to the executable (including the `.EXE`, `.BAT`, etc., extension). Quotes are generally not needed even for paths with spaces.
    • If only the filename is given, Windows searches relative to the current directory.
    • This method is less flexible than using `lpCommandLine`.
  • Using `lpCommandLine` (Recommended):
    • Set `lpApplicationName` to `NULL`.
    • `lpCommandLine` contains the **entire** command line, including the executable name (which might contain spaces and require quoting) followed by its arguments, all delimited by spaces. "C:\\Program Files\\MyApp\\App.exe" -option /value "Argument with spaces"
    • The first token in `lpCommandLine` (parsed according to specific quoting/space rules) is treated as the executable name.
    • If this name does not include a path, the system searches for the executable using the standard DLL search order described in Topic 82 (Application dir, System32, Windows dir, Current dir, PATH).
    • This parameter is **mutable** (`LPTSTR`, not `LPCTSTR`) because historically, applications could modify their own command line string in memory (though this is highly discouraged).

Accessing the Command Line

A child process typically receives its command-line arguments through the standard C/C++ `main` function parameters (`int argc, char *argv[]` or `int argc, wchar_t *argv[]`). Alternatively, a Windows process can retrieve its *entire*, unparsed command line string (as passed to `CreateProcess`) by calling:

LPTSTR GetCommandLine(VOID);

Because the string returned by `GetCommandLine` might be modified by the application itself or other code, it's advisable for a program to make a copy of this string if it needs to parse or rely on the original command line content.

Topic 91: Inheritable Handles

Handle inheritance is a mechanism allowing a newly created child process to receive copies of specific handles owned by its parent process. This is commonly used to redirect the child's standard input, output, or error streams by having the child inherit handles to files or pipes opened by the parent.

Two conditions must be met for a handle to be inherited:

  1. Individual Handle Inheritability: The specific handle in the parent process must have been created with the inheritable property enabled. This is typically done via the `SECURITY_ATTRIBUTES` structure passed to the handle creation function (e.g., `CreateFile`, `CreatePipe`, `CreateEvent`). 
    SECURITY_ATTRIBUTES sa;
    sa.nLength = sizeof(SECURITY_ATTRIBUTES);
    sa.lpSecurityDescriptor = NULL; // Default security
    sa.bInheritHandle = TRUE;       // Make handles created with this inheritable
    
    HANDLE hInheritableFile = CreateFile(..., &sa, ...);
    HANDLE hNonInheritableFile = CreateFile(..., NULL, ...); // Not inheritable
    A single `SECURITY_ATTRIBUTES` structure instance (with `bInheritHandle = TRUE`) can be reused when creating multiple inheritable handles.
  2. `CreateProcess` Inheritance Flag: The `bInheritHandles` parameter in the `CreateProcess` call must be set to `TRUE`. If this flag is `FALSE`, *no* handles are inherited, regardless of their individual inheritability settings. If `TRUE`, *only* those handles in the parent that were explicitly marked as inheritable are duplicated for the child process.

Topic 92: Passing Inheritable Handles

Simply making handles inheritable isn't enough; the child process needs to know the *value* of the inherited handle to use it. When a handle is inherited, the child process receives its own handle entry in its process handle table, but this entry refers to the same underlying kernel object as the parent's handle. The actual numeric value of the handle in the child process might or might not be the same as in the parent.

There are several ways to communicate handle values to a child process:

  1. Inter-Process Communication (IPC): Use mechanisms like named pipes, shared memory, sockets, or window messages after the child process starts to send the handle values from parent to child. This is flexible but more complex to set up.
  2. Command Line: Convert the handle value (which is essentially a pointer-sized number) into a string and pass it as a command-line argument to the child via `CreateProcess`. The child then parses the argument back into a handle value. This is generally discouraged as handle values can sometimes be guessed, and passing them on the command line might expose them.
  3. `STARTUPINFO` Structure (Preferred for Standard Handles): This is the standard method for redirecting standard input, output, and error.
    • The parent process creates pipes or opens files, ensuring the relevant ends have inheritable handles (e.g., the read end of the child's stdin pipe, the write end of the child's stdout pipe).
    • The parent fills a `STARTUPINFO` structure.
    • It sets the `dwFlags` member to include `STARTF_USESTDHANDLES`.
    • It sets the `hStdInput`, `hStdOutput`, and `hStdError` members of the `STARTUPINFO` structure to the inheritable handle values it wants the child to use for standard I/O.
    • It passes this configured `STARTUPINFO` structure to `CreateProcess` (with `bInheritHandles = TRUE`).
    • The child process automatically starts with its standard handles set to the values provided in `STARTUPINFO`, allowing functions like `printf` or `ReadFile(GetStdHandle(STD_INPUT_HANDLE), ...)` to work with the redirected handles seamlessly without the child needing any specific code changes.

Remember:

  • Inherited handles are distinct copies referencing the same kernel object.
  • Parent and child might access the same file object but potentially through different file pointers unless the object is shared carefully (e.g., using memory mapping or explicit synchronization).
  • Both the parent and the child process are responsible for closing their respective copies of the inherited handles when they are finished using them.

Topic 96: Waiting for a Process

Synchronization between processes, such as waiting for one process to terminate before another continues, can be achieved using Windows wait functions. These are general-purpose functions capable of waiting on various types of kernel objects (including processes, threads, events, mutexes, semaphores) that become "signaled" under certain conditions. For a process object, it becomes signaled when the process terminates.

The two primary wait functions are:

  1. WaitForSingleObject(): Waits for a single kernel object to become signaled or for a timeout interval to elapse. 
    DWORD WaitForSingleObject(
  HANDLE hHandle,        // Handle to the object to wait on
  DWORD  dwMilliseconds  // Timeout interval in milliseconds
);
  2. WaitForMultipleObjects(): Waits for one or all objects in an array of kernel objects to become signaled, or for a timeout. 
    DWORD WaitForMultipleObjects(
  DWORD        nCount,          // Number of object handles in the array
  const HANDLE *lpHandles,      // Pointer to the array of object handles
  BOOL         bWaitAll,        // TRUE to wait for all, FALSE to wait for any
  DWORD        dwMilliseconds   // Timeout interval in milliseconds
);

Parameters:

  • hHandle: The handle to the single object (e.g., a process handle obtained from `CreateProcess`) for `WaitForSingleObject`.
  • nCount: The number of handles in the `lpHandles` array. Cannot exceed `MAXIMUM_WAIT_OBJECTS` (defined in `windows.h`, typically 64).
  • lpHandles: Pointer to an array of object handles.
  • bWaitAll: If `TRUE`, the function returns only when *all* objects in `lpHandles` are signaled simultaneously. If `FALSE`, it returns when *any one* of the objects becomes signaled.
  • dwMilliseconds: The timeout interval.
    • 0: Tests the object(s) state and returns immediately (zero timeout).
    • INFINITE (0xFFFFFFFF): Waits indefinitely until the object(s) become signaled.
    • A positive value: Waits for the specified number of milliseconds.

Return Values:

The return value indicates the result of the wait:

  • WAIT_OBJECT_0: For `WaitForSingleObject`, indicates the object became signaled. For `WaitForMultipleObjects` with `bWaitAll=TRUE`, indicates all objects became signaled. For `WaitForMultipleObjects` with `bWaitAll=FALSE`, indicates the object at index 0 in `lpHandles` became signaled.
  • WAIT_OBJECT_0 + n: (For `WaitForMultipleObjects` with `bWaitAll=FALSE`) Indicates that the object at index `n` in the `lpHandles` array became signaled.
  • WAIT_ABANDONED_0 / WAIT_ABANDONED_0 + n: Indicates the wait completed because a mutex object (at index 0 or n) was abandoned (i.e., the thread owning it terminated without releasing it).
  • WAIT_TIMEOUT (0x00000102L): The timeout interval elapsed before the required object(s) became signaled.
  • WAIT_FAILED ((DWORD)0xFFFFFFFF): The function failed (e.g., invalid handle, invalid parameter). Check GetLastError().

Topic 97: Environment Block

Each Windows process has an associated Environment Block. This block stores environment strings relevant to the process, typically inherited from its parent process or the system.

  • The block contains a series of null-terminated strings.
  • Each string usually follows the format Name=Value.
  • The entire block is terminated by an additional null character.
  • A common example is the PATH environment variable: PATH=C:\Windows;C:\Windows\System32;...

Environment variables can be retrieved and modified using specific APIs:

  • GetEnvironmentVariable(): Retrieves the value of a specific environment variable. 
    DWORD GetEnvironmentVariable(
  LPCTSTR lpName,    // Variable name
  LPTSTR  lpBuffer,  // Buffer to receive the value
  DWORD   nSize      // Size of the buffer
);
    • Returns the number of characters copied to `lpBuffer` (excluding null terminator) on success.
    • Returns 0 if the variable is not found (`GetLastError` returns `ERROR_ENVVAR_NOT_FOUND`).
    • If `lpBuffer` is too small, returns the required buffer size (including null terminator).
  • SetEnvironmentVariable(): Creates, modifies, or deletes an environment variable for the current process (and subsequently created child processes, but not the parent or system). 
    BOOL SetEnvironmentVariable(
  LPCTSTR lpName,   // Variable name
  LPCTSTR lpValue   // New value (or NULL to delete)
);
    • lpName: The name of the variable. Cannot contain the '=' character.
    • lpValue: The value to set. If the variable exists, its value is overwritten. If it doesn't exist, a new variable is created. If `lpValue` is `NULL`, the variable specified by `lpName` is deleted.
    • Returns `TRUE` on success, `FALSE` on failure.

When calling CreateProcess, setting the `lpEnvironment` parameter to `NULL` causes the child process to inherit a copy of the parent's current environment block.

Access rights specified in `CreateProcess` (often `PROCESS_ALL_ACCESS` by default if attributes are NULL) control operations on the process handle. Other options exist like `PROCESS_QUERY_INFORMATION`, `PROCESS_TERMINATE`, `PROCESS_SET_INFORMATION`, `DUPLICATE_HANDLE`, `CREATE_HANDLE`. Limiting rights (e.g., only giving the parent `PROCESS_TERMINATE` rights) can enhance security.

Topic 98: A Pattern Searching Example (`grepMP`)

This topic describes an example program (`grepMP`) that leverages Windows multitasking (multiple processes) to search for a specific text pattern within multiple files concurrently.

Concept:

  1. The main `grepMP` process takes the search pattern and a list of filenames as command-line arguments.
  2. For each input file, the main process creates a separate child "grep" process (presumably executing a simpler `grep.exe` program or a function within the same executable).
  3. Each child process is responsible for searching for the pattern within its assigned input file.
  4. Standard output (`stdout`) for each child process is redirected to a unique temporary file created by the parent. This redirection is set up using inheritable handles and the `STARTUPINFO` structure passed to `CreateProcess`.
  5. The main process launches all child processes and then uses `WaitForMultipleObjects` to wait for them to complete. (Since `WaitForMultipleObjects` is limited, typically to 64 handles, the waiting might be done in batches if there are many files).
  6. After a child process terminates, the main process checks its exit code. A specific exit code (e.g., 0) indicates the child found the pattern in its file.
  7. If the pattern was found (exit code 0), the main process then "cats" (displays) the contents of the corresponding temporary output file to its own standard output. This is likely done by creating another child process to execute `cat tempfile.tmp`.
  8. Finally, the main process deletes all temporary files and cleans up process handles.

This approach parallelizes the search operation across multiple processes, potentially speeding up the search significantly on multi-core systems compared to searching files sequentially in a single process.

Implementation Sketch (`grepMP` Example):

#include "Everything.h"
#include  // For malloc/free

// Structure to hold per-process info (temp file name)
typedef struct { TCHAR tempFile[MAX_PATH]; } PROCFILE;

int _tmain (int argc, LPTSTR argv[]) {
    HANDLE hTempFile;
    SECURITY_ATTRIBUTES stdOutSA = {sizeof(SECURITY_ATTRIBUTES), NULL, TRUE}; // Inheritable handle SA
    TCHAR commandLine[MAX_PATH + 100];
    STARTUPINFO startUpSearch, startUpCat; // Separate startup info
    PROCESS_INFORMATION processInfo;
    DWORD exitCode, dwCreationFlags = 0;
    int iProc, numFiles;
    HANDLE *hProc;     /* Array of child process handles. */
    PROCFILE *procFile;  /* Array of temp file names. */

#ifdef UNICODE
    dwCreationFlags = CREATE_UNICODE_ENVIRONMENT;
#endif

    if (argc < 3) {
        ReportError (_T("Usage: grepMP pattern file1 [file2 ...]"), 1, FALSE);
        return 1;
    }
    numFiles = argc - 2;

    /* Initialize startup info structures */
    GetStartupInfo (&startUpSearch); // For grep processes
    GetStartupInfo (&startUpCat);      // For cat processes

    /* Allocate storage for process handles and temp file names. */
    procFile = (PROCFILE*)malloc (numFiles * sizeof (PROCFILE));
    hProc = (HANDLE*)malloc (numFiles * sizeof (HANDLE));
    if(procFile == NULL || hProc == NULL) { ReportError(_T("Mem alloc failed"), 1, TRUE); }

    /* Create a separate "grep" process for each file. */
    for (iProc = 0; iProc < numFiles; iProc++) {
        // Create command line: grep "pattern" "filename"
        _stprintf_s (commandLine, MAX_PATH + 100, _T ("grep \"%s\" \"%s\""), argv[1], argv[iProc + 2]);

        // Create a unique temp file name for this process's output
        if (GetTempFileName (_T("."), _T ("gtm"), 0, procFile[iProc].tempFile) == 0) {
            ReportError (_T ("Temp file creation failed."), 2, TRUE);
        }

        // Create the temp file handle for writing (inheritable)
        hTempFile = CreateFile (procFile[iProc].tempFile, GENERIC_WRITE,
                           FILE_SHARE_READ | FILE_SHARE_WRITE, &stdOutSA,
                           CREATE_ALWAYS, FILE_ATTRIBUTE_NORMAL, NULL);
        if (hTempFile == INVALID_HANDLE_VALUE) {
            ReportError (_T ("Failure opening temp file."), 3, TRUE);
        }

        // Set up STARTUPINFO for redirection
        startUpSearch.dwFlags = STARTF_USESTDHANDLES;
        startUpSearch.hStdOutput = hTempFile;
        startUpSearch.hStdError = hTempFile; // Redirect stderr too
        startUpSearch.hStdInput = GetStdHandle (STD_INPUT_HANDLE); // Inherit stdin

        /* Create the grep process */
        if (!CreateProcess (NULL, commandLine, NULL, NULL,
                              TRUE, /* Inherit handles */
                              dwCreationFlags, NULL, NULL, &startUpSearch, &processInfo)) {
            ReportError (_T ("ProcCreate failed."), 4, TRUE);
        }

        /* Close unwanted handles (parent's handle to temp file, child thread handle) */
        CloseHandle (hTempFile); // Child has its inherited copy
        CloseHandle (processInfo.hThread);

        /* Save the child process handle. */
        hProc[iProc] = processInfo.hProcess;
    } // End for loop creating processes

    /* Wait for all grep processes to complete, in batches if needed */
    for (iProc = 0; iProc < numFiles; iProc += MAXIMUM_WAIT_OBJECTS) {
        int count = min(MAXIMUM_WAIT_OBJECTS, numFiles - iProc);
        WaitForMultipleObjects (count, &hProc[iProc], TRUE, INFINITE);
    }

    /* Output results from temp files for processes that found matches */
    for (iProc = 0; iProc < numFiles; iProc++) {
         // Check exit code of the grep process
        if (GetExitCodeProcess (hProc[iProc], &exitCode) && exitCode == 0) { // Assume exit code 0 means found
            /* Pattern was detected - List results using cat */
            if (numFiles > 1) _tprintf (_T("\n--- Results from %s ---\n"), argv[iProc+2]);

            _stprintf_s (commandLine, MAX_PATH + 100, _T ("cat \"%s\""), procFile[iProc].tempFile);

            // Create cat process (inherits parent's stdout)
            if (!CreateProcess (NULL, commandLine, NULL, NULL,
                                  FALSE, /* Don't inherit handles */
                                  dwCreationFlags, NULL, NULL, &startUpCat, &processInfo)) {
                ReportError (_T ("Failure executing cat."), 0, TRUE); // Report but don't exit main loop
            } else {
                // Wait for cat process to finish
                WaitForSingleObject (processInfo.hProcess, INFINITE);
                CloseHandle (processInfo.hProcess);
                CloseHandle (processInfo.hThread);
            }
        }
        /* Close the handle to the grep process */
        CloseHandle (hProc[iProc]);
        /* Delete the temporary file */
        if (!DeleteFile (procFile[iProc].tempFile)) {
            ReportError (_T ("Cannot delete temp file."), 6, TRUE); // Report but don't exit loop
        }
    }

    // Free allocated memory
    free (procFile);
    free (hProc);

    return 0;
}

This sketch assumes a `grep.exe` and `cat.exe` exist or are implemented elsewhere. Error checking and resource cleanup need careful implementation.

Topic 99: Working in Multiprocessor Environment

Windows supports running processes concurrently on both uniprocessor and multiprocessor systems.

  • Uniprocessor System: The OS achieves concurrency by time-multiplexing the single processor. It rapidly switches between executing threads from different processes, giving the illusion of parallelism.
  • Multiprocessor System: The Windows scheduler can run threads from the same or different processes truly in parallel on separate physical processors or cores. This can lead to substantial performance gains for multi-threaded applications or when running multiple processes concurrently (like the `grepMP` example).

The performance gain on multiprocessor systems is often not perfectly linear with the number of processors due to:

  • Dependencies between processes/threads (e.g., waiting for signals, locks, or I/O).
  • Synchronization overhead (acquiring/releasing locks).
  • Communication overhead between threads/processes.
  • System overhead (scheduling, context switching).

To optimize performance, programs can be designed with an understanding of this potential parallelism.

Processor Affinity

While the scheduler normally decides which processor a thread runs on, Windows allows constraining a process or thread to run only on a specific subset of the available processors. This is done using a **processor affinity mask**.

  • Setting affinity (via `SetProcessAffinityMask` or `SetThreadAffinityMask`) can sometimes be useful for performance tuning (e.g., improving cache locality) but can also hinder performance if it prevents the scheduler from utilizing available cores effectively.
  • It allows ensuring specific threads run on specific CPUs if needed.

The ability to create independent threads within a process allows developers to explicitly design applications that can be scheduled on separate processors for parallel execution.

Topic 100: Process Execution Times

Windows provides a mechanism to determine the amount of time a process has consumed during its execution.

The function used for this purpose is GetProcessTimes():

BOOL GetProcessTimes(
  HANDLE     hProcess,        // Handle to the process
  LPFILETIME lpCreationTime,  // Receives process creation time
  LPFILETIME lpExitTime,      // Receives process exit time
  LPFILETIME lpKernelTime,    // Receives time spent in kernel mode
  LPFILETIME lpUserTime       // Receives time spent in user mode
);
  • hProcess: A handle to the process of interest. Can be a handle to the current process (`GetCurrentProcess()`) or a handle to another process (potentially terminated) for which you have appropriate access rights (e.g., `PROCESS_QUERY_INFORMATION`).
  • lpCreationTime: Pointer to a FILETIME structure that receives the time the process was created.
  • lpExitTime: Pointer to a FILETIME structure that receives the time the process exited. If the process has not terminated, the content is undefined.
  • lpKernelTime: Pointer to a FILETIME structure that receives the total amount of time all threads within the process have spent executing in kernel mode (e.g., performing system calls).
  • lpUserTime: Pointer to a FILETIME structure that receives the total amount of time all threads within the process have spent executing in user mode (application code).
  • Returns TRUE on success, FALSE on failure.

FILETIME is a 64-bit value representing the number of 100-nanosecond intervals since January 1, 1601 (UTC). You can perform arithmetic on these 64-bit values (often via the `LARGE_INTEGER` union) to calculate durations.

  • Elapsed Time ("Real Time"): Can be computed by subtracting `lpCreationTime` from `lpExitTime` (for a terminated process) or subtracting `lpCreationTime` from the current time (for a running process).
  • Total CPU Time: The sum of `lpKernelTime` and `lpUserTime` represents the total CPU time consumed by the process's threads.

A similar function, GetThreadTimes(), exists to retrieve timing information for a specific thread, requiring a thread handle.

Topic 101: Example of Process Execution Times (`timep`)

This topic describes an example program (`timep`) that executes a command provided on its command line and then prints the elapsed ("Real"), kernel ("Sys"), and user times consumed by the executed command's process.

Program Logic (`timep`):

  1. Get Command Line: Uses GetCommandLine() to retrieve the full command line string the `timep` program was invoked with.
  2. Skip Program Name: Uses a helper function (e.g., `SkipArg`) to parse the command line and get a pointer (`targv`) to the actual command/arguments that `timep` should execute (skipping `timep.exe` itself).
  3. Check Usage: Verifies that a command was actually provided.
  4. (Optional) Check OS Version: The example code includes checks using `GetVersionEx` to ensure it's running on Windows NT/2000 or later, as `GetProcessTimes` might have had different behavior or availability on older systems like Windows 9x.
  5. Get Startup Info: Initializes a `STARTUPINFO` structure using `GetStartupInfo`.
  6. Create Process: Uses CreateProcess to execute the command specified by `targv`. Crucially, it passes `TRUE` for `bInheritHandles` (though not strictly necessary for this example unless redirection was intended) and receives the process and thread handles in a `PROCESS_INFORMATION` structure (`procInfo`).
  7. Wait for Process: Uses `WaitForSingleObject` with the process handle (`hProc = procInfo.hProcess`) and `INFINITE` timeout to wait until the child process terminates.
  8. Get Process Times: Calls `GetProcessTimes` using the child process handle (`hProc`) to retrieve the creation, exit, kernel, and user times into `FILETIME` structures (likely using `LARGE_INTEGER` unions for easier arithmetic).
  9. Calculate Elapsed Time: Subtracts the creation time from the exit time (`elapsedTime.li = exitTime.li - createTime.li`).
  10. Convert Times: Uses FileTimeToSystemTime to convert the elapsed, kernel, and user `FILETIME` values into `SYSTEMTIME` structures for easier formatting.
  11. Print Times: Uses `_tprintf` to display the Real (elapsed), User, and Sys (kernel) times in HH:MM:SS.ms format, extracting the components from the `SYSTEMTIME` structures.
  12. Cleanup: Closes the handles for the child process's primary thread (`procInfo.hThread`) and the child process itself (`procInfo.hProcess`) using `CloseHandle`.
  13. Exits.

The `SkipArg` function is not a standard API call; it would be a custom helper function to parse command-line arguments, likely handling spaces and quotes.

Topic 103: Simple Job Management Shell

This topic introduces an example program implementing a basic command shell (`JobObjectShell.c` / `JobShell`) that provides simple job control features, similar to shells found in Unix/Linux environments. This shell manages background processes using Windows Job Objects.

Shell Commands:

  • jobbg [options] command [args...]: Runs the specified `command` in the background. Assigns it a job number and records its process ID in a shared job file.
  • jobs: Lists all background jobs currently managed by the shell (reading from the shared job file), showing their status (running, done).
  • kill [options] jobNumber: Terminates the process associated with the specified `jobNumber`. Can optionally send Ctrl+C or Ctrl+Break instead of terminating directly.
  • quit: Exits the job shell.

Implementation Details:

  • Shared Job File: Uses a user-specific file (e.g., determined via `GetJobMgtFileName`) to store information about active background jobs (job number, process ID, command line).
  • Concurrency Control: Uses file locking (`LockFileEx`) on the shared job file to ensure that multiple instances of the shell running concurrently do not corrupt the job list.
  • Job Objects: Creates a Windows Job Object (`CreateJobObject`) when the shell starts. All background processes launched by `jobbg` are assigned to this job object (`AssignProcessToJobObject`). This allows:
    • Managing processes as a group.
    • Applying limits (e.g., process time limits via `SetInformationJobObject` with `JOBOBJECT_BASIC_LIMIT_INFORMATION`).
    • Querying aggregate statistics (e.g., total user time via `QueryInformationJobObject` with `JobObjectBasicAccountingInformation`).
  • Process Creation (`jobbg`):
    • Uses `CreateProcess` to launch the background command.
    • Options allow creating the process with a new console (`-c`, `CREATE_NEW_CONSOLE`) or detached without a console (`-d`, `DETACHED_PROCESS`).
    • The process is initially created suspended (`CREATE_SUSPENDED`).
    • Adds the process to the job object.
    • Records the job number, process ID, and command line in the locked job file (using a helper like `GetJobNumber`).
    • Resumes the suspended thread (`ResumeThread`) only after successfully adding the job information.
    • Handles are closed appropriately.
  • Listing Jobs (`jobs`):
    • Uses a helper function (`DisplayJobs`) which locks the job file, reads each entry, checks the status of the corresponding process (using `OpenProcess` and `GetExitCodeProcess`), prints the status, and potentially cleans up entries for processes that no longer exist.
    • Also queries and displays accounting information from the job object.
  • Killing Jobs (`kill`):
    • Uses a helper function (`FindProcessId`) to look up the process ID associated with the given job number in the locked job file.
    • Opens the target process handle (`OpenProcess` with `PROCESS_TERMINATE` and potentially `PROCESS_VM_READ` for status checks).
    • If options `-c` or `-b` are given, uses `GenerateConsoleCtrlEvent` to send a `CTRL_C_EVENT` or `CTRL_BREAK_EVENT` to the process group (requires the target process to share a console or be in the same process group).
    • Otherwise, terminates the process directly using `TerminateProcess`.
    • Waits briefly (`WaitForSingleObject`) and closes the handle.
  • Time Limit: An optional command-line argument to the shell itself can set a per-process user time limit via the job object's `JOBOBJECT_BASIC_LIMIT_INFORMATION`.

This example combines process management, job objects, file I/O, file locking, and basic command parsing to create a functional job control shell.

Topic 105: Listing Background jobs (`DisplayJobs`)

This topic details the logic within the `DisplayJobs` function, used by the `jobs` command in the Job Management Shell example.

Functionality (`DisplayJobs`):

  1. Get Job File Path: Determines the path to the shared job database file (e.g., via `GetJobMgtFileName`).
  2. Open Job File: Opens the file with read-write access (`CreateFile`, `OPEN_EXISTING`).
  3. Lock File: Acquires an exclusive lock on the *entire* job file using `LockFileEx` to prevent concurrent modifications by other shell instances while reading and potentially updating.
  4. Read Loop (`__try` block):
    • Reads job records (`JM_JOB` structure) sequentially from the file using `ReadFile`.
    • For each record read:
      • If the `ProcessId` in the record is 0, skip (represents an empty/deleted slot).
      • Try to open the process handle using `OpenProcess` with sufficient rights (`PROCESS_QUERY_INFORMATION` or `PROCESS_ALL_ACCESS`) and the stored `ProcessId`.
      • If `OpenProcess` succeeds:
        • Check the process status using `GetExitCodeProcess`. If the exit code is `STILL_ACTIVE`, print the job as running.
        • If the exit code is not `STILL_ACTIVE`, print the job as "+ Done". Since the process terminated, mark this record for removal by setting `ProcessId` to 0 and writing the modified record back to the *same position* in the file (requires seeking back using `SetFilePointer`).
        • Close the process handle obtained from `OpenProcess`.
      • If `OpenProcess` fails (returns `NULL`), assume the process no longer exists in the system (or access is denied). Print the job as "Done". Mark the record for removal as above.
      • Print the job number and the command line stored in the record.
  5. Unlock File (`__finally` block): Releases the exclusive lock on the job file using `UnlockFileEx`.
  6. Close Job File: Closes the handle to the job file using `CloseHandle`.

This function ensures that the job list displayed is relatively up-to-date by checking process status and cleans up records for terminated processes within a critical section protected by file locking.

Topic 106: Finding a Process Id (`FindProcessId`)

This topic describes the helper function `FindProcessId`, used by the `kill` command in the Job Management Shell example to retrieve the process ID associated with a given job number.

Functionality (`DWORD FindProcessId (DWORD jobNumber)`):

  1. Get Job File Path: Determines the path to the shared job database file.
  2. Open Job File: Opens the file with read access (`CreateFile`, `OPEN_EXISTING`). Handles failure.
  3. Calculate Offset: Determines the byte offset within the file where the record for the specified `jobNumber` should reside (e.g., `offset = sizeof(JM_JOB) * (jobNumber - 1)` assuming job numbers start from 1).
  4. Check File Size (Optional but Recommended): Use `GetFileSizeEx` to ensure the calculated offset is within the bounds of the file to prevent reading past the end.
  5. Position File Pointer: Use `SetFilePointer` (or `SetFilePointerEx`) to move the file pointer to the calculated `offset`.
  6. Lock Record: Acquire a shared lock (read lock) on the specific record's byte range using `LockFileEx`. The `OVERLAPPED` structure's offset fields are set to the calculated `offset`, and the lock length is `sizeof(JM_JOB)`. This prevents another process from modifying this specific record while it's being read.
  7. Read Record: Read the single `JM_JOB` record from the current position using `ReadFile`.
  8. Unlock Record: Release the shared lock using `UnlockFileEx` with the same offset and size.
  9. Close Job File: Close the file handle using `CloseHandle`.
  10. Return Process ID: If the read was successful and the record corresponds to the job, return the `ProcessId` field from the `jobRecord` structure. Otherwise (file error, invalid job number, record not found), return 0.

This function provides a targeted lookup based on the job number, using file locking to ensure data consistency while reading the specific record.

Topic 107: Job Objects

Windows Job Objects provide a mechanism for managing a group of processes as a single unit. You can create a job object and then assign one or more processes to it.

Key uses and features:

  • Group Management: Allows operations to be performed on all processes within the job simultaneously (e.g., terminating all processes in the job).
  • Resource Limits: Enables setting resource limits that apply to all processes within the job collectively or individually. Examples include:
    • Per-process or job-wide user-mode execution time limits.
    • Working set size limits (minimum and maximum).
    • Limit on the total number of active processes within the job.
    • Process priority class constraints.
    • Processor affinity constraints for all processes in the job.
  • Accounting Information: Automatically maintains aggregate accounting information for all processes that have ever been associated with the job (e.g., total user time, total kernel time, total page faults, peak memory usage).
  • UI Restrictions: Can impose certain user interface restrictions (e.g., prevent access to specific window handles or clipboard formats).
  • Security: Can enforce specific security limitations.

Creating and Managing Job Objects:

  1. CreateJobObject(): Creates or opens a job object. Can be named (for sharing) or unnamed. Requires security attributes. 
    HANDLE CreateJobObject(
  LPSECURITY_ATTRIBUTES lpJobAttributes, // Security attributes (optional)
  LPCTSTR               lpName           // Name for sharing (optional)
);
  2. OpenJobObject(): Opens an existing named job object.
  3. AssignProcessToJobObject(): Assigns a process (specified by its handle) to an existing job object. 
    BOOL AssignProcessToJobObject(
  HANDLE hJob,     // Handle to the job object
  HANDLE hProcess  // Handle to the process to assign
);
    • A process can only be associated with a single job object during its lifetime.
    • Child processes are automatically assigned to the same job as the parent unless the `CREATE_BREAKAWAY_FROM_JOB` flag is used in `CreateProcess`.
  4. SetInformationJobObject(): Sets limits and controls for the processes associated with the job object. 
    BOOL SetInformationJobObject(
  HANDLE                 hJob,                      // Handle to the job object
  JOBOBJECTINFOCLASS     JobObjectInformationClass, // Type of limit/info to set
  LPVOID                 lpJobObjectInformation,    // Pointer to structure containing limit data
  DWORD                  cbJobObjectInformationLength // Size of the structure
);
    • `JobObjectInformationClass` specifies the type of information being set (e.g., `JobObjectBasicLimitInformation`, `JobObjectExtendedLimitInformation`, `JobObjectBasicUIRestrictions`).
    • `lpJobObjectInformation` points to a structure specific to the class (e.g., `JOBOBJECT_BASIC_LIMIT_INFORMATION`) containing the desired limits.
  5. QueryInformationJobObject(): Retrieves current limits or accounting information for the job object. (Similar parameters to `SetInformationJobObject`).
  6. TerminateJobObject(): Terminates all processes currently associated with the job object.
  7. CloseHandle(): Closes the handle to the job object when no longer needed.

Topic 108: Using Job Objects (Example)

This topic revisits the Job Management Shell example, highlighting its use of Job Objects specifically for limiting process execution time and obtaining aggregate time statistics.

Enhancements using Job Objects:

  1. Initialization:
    • A global `HANDLE hJobObject` is declared.
    • A `JOBOBJECT_BASIC_LIMIT_INFORMATION` structure (`basicLimits`) is declared and initialized. The `LimitFlags` member is set to include `JOB_OBJECT_LIMIT_PROCESS_TIME`.

Topic 109: Thread Overview

A thread is the basic unit of execution within a process. A process starts with one thread (the primary thread) and can create additional threads to perform tasks concurrently.

In a multi-threaded system:

  • Multiple threads can exist and execute within the context of a single process.
  • These threads share the process's resources, such as its virtual address space (code, data, heap), file handles, and other kernel objects.
  • Organizing and coordinating these threads (synchronization, communication, data sharing) presents challenges but also offers significant benefits.

Topic 110: Thread Issues

While threads simplify some programming problems (like responsive UIs or overlapping I/O with computation), they also introduce potential issues:

  • Data Sharing Conflicts: Since threads within a process share the same memory space (data segments, heap), one thread can inadvertently overwrite or corrupt data being used by another thread if access is not properly synchronized (e.g., using mutexes, critical sections).
  • Concurrency Degradation: In some cases, particularly with high contention for shared resources or excessive synchronization overhead, adding more threads can actually degrade performance rather than improve it.
  • Complexity: Converting simple, single-threaded logic to a correct and efficient multi-threaded implementation can significantly complicate program design, debugging, and testing. Issues like race conditions and deadlocks can be difficult to find and fix.
  • Overheads vs. Gains: Traditional concurrent processing using multiple single-threaded processes involves higher overhead for context switching and inter-process communication. Threads reduce this overhead but introduce the need for careful intra-process synchronization.

Despite these challenges, threads enable efficient use of multiprocessor systems and allow for asynchronous operations and responsive applications with less programming effort than managing multiple processes for the same tasks.

Topic 111: Thread Basics

Key characteristics of threads:

  • Shared Resources: Threads within the same process share the process's virtual address space (code segments, data segments, heap) and system resources (file handles, etc.).
  • Unique Resources: Each thread has its own:
    • Stack: Used for local variables, function call parameters, and return addresses. The stack isolates local function execution between threads.
    • Thread Context: CPU register state (instruction pointer, stack pointer, etc.).
    • Thread Local Storage (TLS): A mechanism providing per-thread private static or global storage. Each thread accesses its own unique instance of TLS data.
    • Thread ID: A unique identifier within the process.
  • Data Exclusivity: While address space is shared, programmers must ensure threads do not improperly interfere with shared data. Data intended to be private to a thread should typically be stored on its stack (local variables) or in TLS. Access to shared data often requires synchronization.
  • Arguments: When creating a thread, a single pointer-sized argument can usually be passed to its starting function. This argument is typically stored initially on the thread's stack or accessed via registers depending on the calling convention.

TLS provides a small array of pointers per thread, accessible only by that thread. It allows library functions or other code to maintain per-thread state without modifying function signatures to pass state pointers around, ensuring data privacy between threads using the same function.

Topic 112: Thread Management (`CreateThread`)

Like other system resources, threads are treated as kernel objects managed via handles.

The primary API to create a new thread within the calling process is CreateThread():

HANDLE CreateThread(
  LPSECURITY_ATTRIBUTES   lpThreadAttributes, // Security attributes (optional)
  SIZE_T                  dwStackSize,        // Initial stack size in bytes (0 = default)
  LPTHREAD_START_ROUTINE  lpStartAddress,     // Pointer to the thread function
  LPVOID                  lpParameter,        // Argument passed to the thread function
  DWORD                   dwCreationFlags,    // Creation flags (0 or CREATE_SUSPENDED)
  LPDWORD                 lpThreadId          // Receives the thread ID (optional)
);
  • lpThreadAttributes: Optional pointer to a `SECURITY_ATTRIBUTES` structure (rarely needed for threads).
  • dwStackSize: The initial size of the stack to reserve for the thread, in bytes. If 0, the thread receives the default stack size specified in the executable's header (typically 1 MB). The OS commits stack pages on demand as needed.
  • lpStartAddress: Pointer to the application-defined function that the new thread will begin executing. This function must match the `LPTHREAD_START_ROUTINE` signature: 
    DWORD WINAPI ThreadFunc(LPVOID lpParam);
    It accepts a single `LPVOID` parameter (which is the value passed in `lpParameter` to `CreateThread`) and returns a `DWORD` exit code when the thread terminates by returning from this function.
  • lpParameter: A single pointer-sized value passed directly as the argument to the `lpStartAddress` function. Often used to pass a pointer to a structure containing multiple arguments.
  • dwCreationFlags:
    • 0: The thread starts running immediately after creation.
    • CREATE_SUSPENDED (0x00000004): The thread is created in a suspended state and does not run until `ResumeThread()` is called with its handle.
  • lpThreadId: Optional pointer to a `DWORD` variable that receives the system-wide unique thread identifier (TID). Can be `NULL` if the TID is not needed.
  • **Return Value:** Returns a handle to the new thread object on success, or `NULL` on failure (check `GetLastError`).

While `CreateThread` exists, using the C/C++ runtime library function `_beginthreadex` (from `process.h`) is often recommended when using the runtime library, as it performs necessary initialization for runtime features within the new thread. `CreateThread` does not.

Topic 113: Exiting Thread

A thread ceases execution under several conditions:

  1. Returning from Thread Function: The thread function specified in `CreateThread` (or `_beginthreadex`) returns. The return value becomes the thread's exit code. This is the preferred method for clean termination.
  2. Calling `ExitThread()`: The thread explicitly calls the `ExitThread` function from anywhere within its code. 
    VOID ExitThread(DWORD dwExitCode);
    • dwExitCode: The exit code for the terminating thread.
    • This function terminates *only* the calling thread.
  3. Process Termination: The entire process exits (e.g., `ExitProcess` is called, the primary thread returns from `main`/`WinMain`, or the process is terminated externally). All threads within the process are abruptly terminated.
  4. Calling `TerminateThread()` (Discouraged): One thread calls `TerminateThread` on the handle of another thread in the same process. 
    BOOL TerminateThread(HANDLE hThread, DWORD dwExitCode);
    • This provides an abrupt termination.
    • **Strongly Discouraged:** The target thread is stopped immediately without any chance for cleanup. Its stack is not unwound, C++ destructors are not called, `__finally` blocks do not execute, and DLL notifications (`DLL_THREAD_DETACH`) are not sent. Resources held by the thread (locks, memory) may be leaked, potentially destabilizing the process. Use only as a last resort when a thread is completely unresponsive.

When a thread exits (via methods 1 or 2):

  • Its stack is deallocated.
  • If the thread is associated with any DLLs, `DllMain` is called with the `DLL_THREAD_DETACH` reason for each DLL.
  • The thread object itself remains in the system until the last open handle referring to it is closed using `CloseHandle()`.

The process terminates when its last running thread exits.

Retrieving Exit Code

Other threads within the same process (or other processes with appropriate access rights) can retrieve the exit code of a terminated thread using `GetExitCodeThread()`:

BOOL GetExitCodeThread(
  HANDLE  hThread,       // Handle to the thread
  LPDWORD lpdwExitCode   // Pointer to receive the exit code
);
  • hThread: Handle to the thread whose exit code is desired.
  • lpdwExitCode: Pointer to a `DWORD` variable that receives the exit code.
  • If the thread is still running, the value placed in `*lpdwExitCode` will be `STILL_ACTIVE` (259).
  • Returns `TRUE` on success, `FALSE` on failure.

Topic 114: Thread Identity

Similar to processes, threads have both handles and unique identifiers (TIDs).

  • GetCurrentThread(): Returns a **pseudo-handle** for the calling thread. This is a special constant handle value ((HANDLE)-2) that always refers to the current thread. It can be used in most functions requiring a thread handle for the current thread but cannot be inherited or duplicated for use by other threads. Use `DuplicateHandle` if a real handle to the current thread is needed by another thread/process.
  • GetCurrentThreadId(): Returns the system-wide unique Thread ID (TID) of the calling thread.
  • GetThreadId(): Returns the Thread ID (TID) associated with a given thread handle. 
    DWORD GetThreadId(HANDLE Thread);
  • OpenThread(): Opens a handle to an existing thread object using its Thread ID. Requires specifying desired access rights. 
    HANDLE OpenThread(
  DWORD dwDesiredAccess,  // e.g., THREAD_QUERY_INFORMATION, THREAD_SUSPEND_RESUME
  BOOL  bInheritHandle,   // Inheritable flag
  DWORD dwThreadId        // TID of the target thread
);

TIDs are unique system-wide while the thread exists, whereas handles are specific to a process and refer to the thread object.

Topic 115: More on Thread Management

Beyond the core functions (`CreateThread`, `ExitThread`, `GetExitCodeThread`), Windows offers additional functions for managing and querying threads, some introduced in later OS versions (XP/2003+).

  • GetProcessIdOfThread(): (Requires Windows 2003/XP SP1+) Given a thread handle, returns the process ID (PID) of the process that owns the thread. Useful when interacting with threads in other processes or mapping threads back to their parent process. 
    DWORD GetProcessIdOfThread(HANDLE Thread);
  • GetThreadIOPendingFlag(): Determines if a specified thread has any outstanding I/O requests pending. This can indicate if a thread is blocked waiting for an I/O operation to complete. The result is instantaneous and might change immediately after the call. 
    BOOL GetThreadIOPendingFlag(HANDLE hThread, PBOOL lpIOIsPending);
    • hThread: Handle to the thread.
    • lpIOIsPending: Pointer to a BOOL that receives TRUE if I/O is pending, FALSE otherwise.
    • Returns TRUE on success, FALSE on failure.

Suspending and Resuming Threads

Windows allows one thread to temporarily pause (suspend) and later resume the execution of another thread within the same process. Each thread maintains a **suspend count**.

  • A thread only runs when its suspend count is 0.
  • If the count is greater than 0, the thread is suspended (paused).
  • The CREATE_SUSPENDED flag in `CreateThread` starts a thread with a suspend count of 1.
  • SuspendThread(): Increments the target thread's suspend count. If the count becomes non-zero, the thread stops executing. 
    DWORD SuspendThread(HANDLE hThread);
  • ResumeThread(): Decrements the target thread's suspend count. If the count reaches 0, the thread becomes eligible to run again. 
    DWORD ResumeThread(HANDLE hThread);
  • Both functions return the thread's *previous* suspend count on success, or (DWORD)-1 (0xFFFFFFFF) on failure (check `GetLastError`).
  • Caution: Suspending threads can be dangerous if the target thread holds locks or resources needed by other threads, potentially leading to deadlocks. It's generally used for specific coordination tasks (like ensuring initialization is complete before a worker thread runs) rather than general-purpose pausing.

Topic 116: Waiting for Threads

Just as a process can wait for another process to terminate (Topic 96), one thread can wait for another thread to complete its execution. Since thread objects are kernel objects, the standard wait functions WaitForSingleObject and WaitForMultipleObjects can be used with thread handles.

  • A thread object becomes **signaled** when the thread terminates (either by returning from its function, calling `ExitThread`, or being terminated).
  • Once signaled due to termination, a thread object remains signaled permanently.
  • A call like WaitForSingleObject(hOtherThread, INFINITE); will block the calling thread until `hOtherThread` terminates.
  • WaitForMultipleObjects can be used to wait for one or all of several threads to terminate.
    • The limit of `MAXIMUM_WAIT_OBJECTS` (usually 64) applies. To wait for more threads, multiple calls or different strategies (like using events signaled by threads upon completion) are needed.

Waiting on thread handles is a fundamental synchronization technique, ensuring that actions dependent on a thread's completion are performed only after the thread has actually finished.

Exiting the entire process (e.g., via `ExitProcess` or the primary thread returning) also signals all thread objects within that process.

Topic 117: C Library in Threads

Using standard C library functions concurrently from multiple threads requires caution regarding **thread safety**.

Some C library functions maintain internal static state between calls. If multiple threads call such a function concurrently, they might interfere with each other's internal state, leading to corrupted results or crashes.

  • The example given is strtok(), which uses an internal static pointer to keep track of the position in the string being tokenized. If two threads call `strtok` on different strings simultaneously, they will overwrite each other's internal pointer, leading to incorrect tokenization.

To address this, Windows C/C++ compilers (like Microsoft Visual C++) provide **thread-safe versions** of the C runtime library (CRT). The multithreaded version is typically named `LIBCMT.LIB` (static linking) or `MSVCRT.DLL` (dynamic linking, used by default in most modern projects).

These thread-safe libraries:

  • Protect internal global state with locks (e.g., for `stdio` functions).
  • Provide alternative functions that avoid static state (e.g., `strtok_s` instead of `strtok`).
  • Use Thread Local Storage (TLS) where appropriate to maintain per-thread state internally.

When creating threads, using the C runtime library's thread creation functions is often preferred over the raw Windows API `CreateThread` because they perform necessary initialization for the CRT within the new thread:

  • _beginthreadex() (recommended): Creates a new thread and initializes the CRT state for it. Returns a `uintptr_t` which must be cast to `HANDLE`. Offers more control (security attributes, initial state) similar to `CreateThread`.
  • _endthreadex(): Called by a thread created with `_beginthreadex` to terminate itself and clean up CRT resources. Returning from the thread function implicitly calls this.
  • (Older functions _beginthread/_endthread exist but have limitations and are less flexible).

Using `_beginthreadex` / `_endthreadex` ensures that CRT functions (like `malloc`, `stdio`, locale functions, signal handling) work correctly within the threads you create.

Topic 118: Multithreaded Pattern Searching (`grepMT`)

This topic describes `grepMT`, a multithreaded version of the pattern searching example previously implemented using multiple processes (`grepMP`). This version uses multiple threads within a single process to achieve concurrency.

Concept:

Instead of launching a separate `grep` process for each file, the main "Boss" thread creates a separate worker thread (`ThGrep`) for each input file specified on the command line.

  1. Argument Preparation: The main thread prepares an argument structure (`GREP_THREAD_ARG`) for each worker thread. This structure contains:
    • The search pattern.
    • The input filename for that thread.
    • A unique temporary filename for that thread's output.
    • The argument count (always 4 for `grep pattern infile outfile`).
  2. Thread Creation: The main thread loops through the input files, creating a worker thread (`ThGrep`) for each using `_beginthreadex`. Handles to these threads are stored in an array (`tHandle`).
  3. Worker Thread (`ThGrep`):
    • Receives its arguments via the `GREP_THREAD_ARG` pointer.
    • Constructs a command line string like `grep "pattern" "infile" > "tempfile"`. (Or more likely, it performs the grep logic internally instead of calling `CreateProcess`).
    • Correction: The description and code imply the worker thread *itself* likely performs the grep logic, reading the input file and writing matches to its assigned temporary output file, rather than calling `CreateProcess`. It would return an exit code (e.g., 0 if found, non-zero otherwise).
  4. Waiting: The main thread waits for all worker threads to complete using `WaitForMultipleObjects` on the `tHandle` array.
  5. Processing Results: After threads complete, the main thread iterates through the results:
    • Gets the exit code of each worker thread using `GetExitCodeThread`.
    • If the exit code indicates a match was found (e.g., 0), it uses `CreateProcess` to run `cat tempfile` (or similar) to display the contents of the thread's temporary output file. It waits for this `cat` process to finish.
    • Closes the handle for the worker thread.
    • Deletes the temporary file.
  6. Cleanup: Frees any dynamically allocated memory (for argument structures, handle arrays).

This multithreaded approach avoids the overhead of creating multiple processes but requires careful management of shared resources (if any) and thread synchronization (though this specific example minimizes sharing by using separate temp files).

The structure described uses asynchronous I/O concepts metaphorically – the main thread can potentially do other work while worker threads perform file I/O and searching concurrently.

Topic 119: Boss worker and other thread models

Multithreaded programs can be designed using various architectural models, analogous to human work arrangements, to manage concurrent tasks efficiently.

  • Boss-Worker Model:
    • One "boss" thread assigns tasks to multiple "worker" threads.
    • Workers perform their assigned tasks independently.
    • Workers report results back to the boss.
    • The boss typically waits for all workers to complete before finalizing the overall result.
    • The `grepMT` example (Topic 118) fits this model, where the main thread (boss) creates worker threads for each file and waits for them.
  • Work Crew Model:
    • Multiple worker threads cooperate on a single large task.
    • Each worker performs a small piece of the overall task.
    • Workers might divide the work amongst themselves dynamically (e.g., using a shared work queue) without explicit direction from a single boss after initial setup.
    • The parallel MergeSort example (Topic 120) uses this model, where threads sort subarrays and then cooperate (implicitly via waiting) to merge results.
  • Client-Server Model:
    • Commonly used for network services or request handling.
    • A server process listens for client requests.
    • When a request arrives, the server typically spawns a dedicated thread (or assigns one from a pool) to handle that specific client's request concurrently with other clients.
    • Each client interaction runs largely independently in its own server-side thread.
  • Pipeline Model:
    • Data flows sequentially through a series of processing stages.
    • Each stage is handled by a dedicated thread.
    • Thread A performs stage 1 and passes the result to Thread B for stage 2, which passes it to Thread C for stage 3, and so on.
    • Allows different stages of processing for different data items to occur concurrently. Requires buffers and synchronization between stages.

Using these established models can simplify design, improve performance, aid debugging and maintenance, and simplify synchronization logic by providing well-defined patterns of interaction.

Topic 120: MergeSort: Exploiting Multiple Processors

This topic describes how multithreading can be used to parallelize the MergeSort algorithm, aiming for optimal performance gain on multiprocessor systems. It utilizes the Work Crew model.

Parallel MergeSort Strategy:

  1. Divide: A large array (or file mapped to memory) is divided into smaller, roughly equal subarrays. The number of subarrays typically matches the desired number of threads (often a power of two, ideally matching the number of available processor cores).
  2. Conquer (Parallel Sort): A separate worker thread is created for each subarray. Each thread independently sorts its assigned subarray using an efficient sequential algorithm (like the standard library `qsort()`).
  3. Combine (Parallel Merge): Threads merge their sorted subarrays in pairs.
    • Thread 0 sorts subarray 0. Thread 1 sorts subarray 1.
    • Thread 0 waits for Thread 1 to finish sorting.
    • Thread 0 then merges the sorted subarray 0 and sorted subarray 1.
    • Similarly, Thread 2 sorts subarray 2, Thread 3 sorts subarray 3. Thread 2 waits for Thread 3 and then merges subarrays 2 and 3.
    • This merging process continues up the hierarchy (Thread 0 might wait for Thread 2 to finish its merge, then merge the first half with the second half) until the entire array is sorted.

Implementation Considerations:

  • Thread Count: Performance is often best when the number of threads matches the number of available processor cores. Using more threads than cores can lead to increased context switching overhead. The number of records should ideally be divisible by the number of threads.
  • Synchronization: Threads need to wait for adjacent threads to complete their sorting/merging phase before they can proceed with the next level of merging. This is typically achieved using `WaitForSingleObject` on the handle of the required adjacent thread.
  • Race Conditions: To avoid race conditions where a merging thread starts before the thread it needs to merge with has even been created or started sorting, all worker threads should be created initially in a suspended state (`CREATE_SUSPENDED`). Only after all threads are created should they all be resumed (using `ResumeThread`) to run concurrently.

This approach allows the computationally intensive sorting of subarrays to occur in parallel, followed by a merging phase that can also have some degree of parallelism.

Topic 121: MergeSort: Code Structure (`SortMT`)

This details the code structure for the multithreaded MergeSort (`SortMT`) example described in Topic 120.

Key Definitions:

#include "Everything.h"
#define DATALEN 56  /* Example data length */
#define KEYLEN 8    /* Example key length */

typedef struct _RECORD {
    TCHAR key[KEYLEN];
    TCHAR data[DATALEN];
} RECORD;
#define RECSIZE sizeof (RECORD)
typedef RECORD * LPRECORD;

/* Structure passed as argument to each sort/merge thread */
typedef struct _THREADARG {
    DWORD    iTh;         /* Thread number (index): 0, 1, 2, ... */
    LPRECORD lowRecord;   /* Pointer to start of subarray for this thread */
    LPRECORD highRecord;  /* Pointer to end (one past last element) of subarray */
} THREADARG, *PTHREADARG;

// Global or static variables
static DWORD nRec;            /* Total number of records */
static HANDLE * pThreadHandle; /* Array of thread handles */

// Function Prototypes
static DWORD WINAPI SortThread (PTHREADARG pThArg); // Thread function
static int KeyCompare (LPCTSTR pRec1, LPCTSTR pRec2); // qsort comparison
static VOID MergeArrays (LPRECORD p1, DWORD nRecs); // Merges two adjacent blocks

Main Logic (`_tmain`):

  1. Parse options (e.g., `-n` for no print) and get the number of threads (`nt` or `numFiles` in OCR) and the input filename. Validate arguments.
  2. Open the input file (`CreateFile` with Read/Write).
  3. Map the entire file into memory (`CreateFileMapping`, `MapViewOfFile` into `pRecords`). Handle potential errors. Close mapping handle (`mHandle`).
  4. Calculate total records (`nRec`) and records per thread (`nRecTh`). Validate that `nRec` is divisible by `numFiles` and `numFiles` is a power of 2 (per limitations).
  5. Allocate memory for the thread argument array (`threadArg`) and thread handle array (`pThreadHandle`).
  6. Loop `numFiles` times (from `iTh = 0`):
    • Fill `threadArg[iTh]` with the thread number, `lowRecord` pointer (calculated based on previous threads), and `highRecord` pointer.
    • Create the `SortThread` using `_beginthreadex`, passing `&threadArg[iTh]` as the parameter and using the `CREATE_SUSPENDED` flag. Store the returned handle in `pThreadHandle[iTh]`.
  7. Loop `numFiles` times: Resume each thread using `ResumeThread(pThreadHandle[iTh])`.
  8. Wait for the entire sort-merge process to complete by waiting on the first thread handle (`WaitForSingleObject(pThreadHandle[0], INFINITE)`), as thread 0 is responsible for the final merge.
  9. Loop `numFiles` times: Close each thread handle using `CloseHandle(pThreadHandle[iTh])`.
  10. (Optional) Print the sorted array (`pRecords`) if `noPrint` is false. Null-terminate the mapped view carefully first.
  11. Unmap the file view (`UnmapViewOfFile(pRecords)`).
  12. (Optional based on OCR) Restore original file length if it was extended.
  13. Close the file handle (`CloseHandle(hFile)`).
  14. Free allocated memory (`threadArg`, `pThreadHandle`).

Thread Logic (`SortThread`):

  1. Receive `pThArg` containing thread number (`myNumber`), `lowRecord`, and `highRecord`.
  2. Calculate the number of records in this thread's initial subarray (`numbersInGroup`).
  3. Sort the initial subarray using `qsort(first, numbersInGroup, RECSIZE, KeyCompare)`.
  4. Enter a `while` loop for merging phases. The loop continues as long as this thread is the "left" thread of a pair to be merged (`myNumber % groupSize == 0`) and hasn't processed the whole array (`numbersInGroup < nRec`).
    • `groupSize` starts at 2 and doubles each merge phase.
    • `twoToI` (representing 2i where i is merge step) tracks the index offset of the adjacent right-hand thread/block.
    • Wait for the adjacent right-hand thread to complete its sort/merge phase: `WaitForSingleObject(pThreadHandle[myNumber + twoToI], INFINITE)`.
    • Call `MergeArrays(first, numbersInGroup)` to merge this thread's current sorted block (`first` pointer, `numbersInGroup` records) with the adjacent block (also `numbersInGroup` records, starting at `first + numbersInGroup`).
    • Double `numbersInGroup`, `groupSize`, and `twoToI` for the next merge level.
  5. Return 0 (thread exit code).

Merge Logic (`MergeArrays`):

  1. Takes a pointer `p1` to the start of the first sorted array and the number of records `nRecs` in *each* array (so total size is `2 * nRecs`).
  2. Pointer `p2` is calculated as `p1 + nRecs` (start of the second array).
  3. Allocates a temporary destination buffer (`pDestHold`) large enough to hold `2 * nRecs` records.
  4. Performs a standard two-way merge from `p1` and `p2` into `pDest`, comparing records using `KeyCompare`.
  5. Copies any remaining elements from `p1` or `p2` after one is exhausted.
  6. Copies the entire merged result from the temporary buffer (`pDestHold`) back over the original two arrays (starting at `p1Hold`).
  7. Frees the temporary buffer.

Topic 122: Introduction to Parallelism

Multiprocessing (multiple processes) and multithreading (multiple threads within a process) both enable multiple flows of execution. With the rise of multi-core and multiprocessor systems, harnessing **parallelism** (true simultaneous execution) is key to performance improvement, as relying solely on increasing CPU clock rates is no longer feasible.

Programs designed for parallelism, like the `grepMT` and `SortMT` examples, often exhibit these properties:

  • Task Decomposition: The main task is divided into smaller subtasks that can run largely independently. Worker threads or processes handle these subtasks.
  • Independence: Worker subtasks run with minimal interaction or dependency on each other during their main execution phase.
  • Result Merging: A master thread/process may combine or merge the results from completed worker subtasks.
  • Minimal Mutual Exclusion: The core work of the subtasks often doesn't require locking shared resources, reducing contention. Synchronization might only be needed for task assignment or result merging.
  • Scalability: Performance ideally scales automatically (up to a limit) as the program runs on systems with more processors, as the OS kernel assigns worker threads to available cores.
  • Serialization Equivalence: If the program is forced to run sequentially (e.g., on a single core or by waiting after each step), the final result should be identical to the parallel execution result. Serialized execution is often easier to debug.
  • Limits to Speedup: Maximum performance improvement is limited by the inherent parallelism of the algorithm (Amdahl's Law - parts that cannot be parallelized limit overall speedup), thread management overhead, and any remaining sequential computation or synchronization.

Topic 123: Thread Local Storage (TLS)

In multithreaded programs, while threads share the process's global and heap memory, sometimes a thread needs data that behaves like a global or static variable but is unique to that thread.

One way is to pass pointers to thread-specific data structures as parameters. Another mechanism provided by Windows is **Thread Local Storage (TLS)**.

TLS provides each thread with its own private array of `LPVOID` (void pointers). The system manages these arrays.

  • TLS Indexes: A process can allocate TLS indexes (slots) dynamically. Once an index is allocated, it exists for *all* threads within the process (current and future). Think of indexes as rows in the conceptual diagram.
  • Per-Thread Values: Each thread can then store a unique `LPVOID` pointer value in its private slot corresponding to that allocated index. Think of the intersection of a row (index) and a column (thread) holding a value specific to that thread for that index.
  • Access: A thread can only access its *own* TLS values. It cannot directly access the TLS values of another thread.

TLS APIs:

  • DWORD TlsAlloc(VOID);
    • Allocates a TLS index that is available across all threads in the process.
    • Returns a unique index value (a `DWORD`) on success, or `TLS_OUT_OF_INDEXES` ((DWORD)-1) on failure.
    • This is typically called once during process or library initialization.
  • BOOL TlsFree(DWORD dwTlsIndex);
    • Releases a previously allocated TLS index, making it available for reuse.
    • Should be called when the index is no longer needed by any thread (e.g., during process or DLL unload).
    • Returns `TRUE` on success, `FALSE` on failure.
  • BOOL TlsSetValue(DWORD dwTlsIndex, LPVOID lpTlsValue);
    • Stores a pointer-sized value (`lpTlsValue`) in the calling thread's slot for the specified `dwTlsIndex`.
    • Each thread calls this to store its own private data pointer (e.g., pointer to dynamically allocated memory) associated with the index.
    • Returns `TRUE` on success, `FALSE` on failure.
  • LPVOID TlsGetValue(DWORD dwTlsIndex);
    • Retrieves the pointer-sized value stored in the calling thread's slot for the specified `dwTlsIndex`.
    • Returns the stored value on success. If no value has been set for this thread at this index, returns `NULL`.
    • If the function fails (e.g., invalid index), it returns `NULL` and sets the last error; check `GetLastError()` to distinguish failure from a stored `NULL`.

Cautions: TLS provides convenient per-thread global storage but adds some overhead compared to stack variables. Ensure allocated TLS indexes are freed (`TlsFree`). Memory pointed to by TLS values must be managed appropriately (e.g., freed before the thread exits if dynamically allocated).

Topic 124: Processes and Thread priorities

In a multitasking system, the OS scheduler manages which thread runs on a processor at any given time. Windows uses a priority-based, preemptive scheduling algorithm.

  • The kernel always attempts to run the highest-priority thread that is in the **Ready** state.
  • A thread is not ready if it is Waiting (e.g., on an object), Suspended, or Blocked (e.g., waiting for I/O).

Thread priorities are determined relative to the **Priority Class** of their owning process.

Process Priority Classes:

Set initially by `CreateProcess` or modified later using `SetPriorityClass()`. Determines the base priority range for threads within the process.

  • IDLE_PRIORITY_CLASS: Lowest priority; runs only when the system is idle.
  • BELOW_NORMAL_PRIORITY_CLASS: Lower than normal.
  • NORMAL_PRIORITY_CLASS: Default for most applications.
  • ABOVE_NORMAL_PRIORITY_CLASS: Higher than normal.
  • HIGH_PRIORITY_CLASS: For time-critical tasks, but use with caution as it can starve lower-priority threads.
  • REALTIME_PRIORITY_CLASS: Highest possible priority; typically requires administrative privileges. Should only be used for very short, critical tasks that must run immediately, as it can preempt system threads.

Background Modes:

  • PROCESS_MODE_BACKGROUND_BEGIN: Lowers the process priority class temporarily (e.g., for background work).
  • PROCESS_MODE_BACKGROUND_END: Restores the original priority class.

APIs: BOOL SetPriorityClass(HANDLE hProcess, DWORD dwPriorityClass); and DWORD GetPriorityClass(HANDLE hProcess);

Thread Relative Priorities:

Within a process's priority class, individual threads can have their priority adjusted relative to the process base priority, typically within a range of +/- 2 "points".

  • THREAD_PRIORITY_IDLE: Lowest relative priority (only for `IDLE_PRIORITY_CLASS` processes).
  • THREAD_PRIORITY_LOWEST: Base - 2
  • THREAD_PRIORITY_BELOW_NORMAL: Base - 1
  • THREAD_PRIORITY_NORMAL: Base priority (default at creation).
  • THREAD_PRIORITY_ABOVE_NORMAL: Base + 1
  • THREAD_PRIORITY_HIGHEST: Base + 2
  • THREAD_PRIORITY_TIME_CRITICAL: Sets a high absolute priority (15 for most classes, 31 for `REALTIME_PRIORITY_CLASS`). Use with caution.

Background Modes:

  • THREAD_MODE_BACKGROUND_BEGIN / THREAD_MODE_BACKGROUND_END: Similar to process background modes, but applied per-thread.

APIs: BOOL SetThreadPriority(HANDLE hThread, int nPriority); and int GetThreadPriority(HANDLE hThread);

Dynamic Priority Boosts:

Thread priorities are not static. The Windows scheduler may temporarily boost a thread's priority under certain conditions (e.g., when it completes I/O, wakes from a wait, or receives window messages) to improve responsiveness. This boosting can be enabled or disabled per-thread using SetThreadPriorityBoost().

Topic 125: Thread States

A thread transitions through various states during its lifetime, managed by the Windows executive (kernel scheduler).

  • Initialized: Thread object created but not yet ready to run (e.g., if created suspended).
  • Ready: The thread is able to run but is waiting for a processor to become available. The scheduler selects the highest-priority ready thread to run.
  • Running: The thread is currently executing instructions on a processor. On a multi-processor system, multiple threads can be in the running state simultaneously.
  • Waiting (Blocked/Sleeping): The thread cannot run because it is waiting for an event to occur or an object to become signaled (e.g., waiting on `WaitForSingleObject`, blocked on I/O completion, waiting for a mutex).
  • Transition: An intermediate state when a thread is moving from waiting back to ready after the object it was waiting on becomes signaled.
  • Terminated: The thread has finished execution but the thread object still exists (because open handles to it remain). Allows other threads to retrieve its exit code. The object is destroyed when the last handle is closed.
  • Standby: A thread selected to run next on a specific processor is placed in the standby state just before running.

Transitions:

  • Initialized -> Ready: When a suspended thread is resumed or a non-suspended thread is created.
  • Ready -> Running: The scheduler selects the thread to run on a processor.
  • Running -> Waiting: Thread calls a blocking function (e.g., `WaitForSingleObject` on a non-signaled object, `Sleep`, blocking I/O).
  • Running -> Ready: Thread's time slice (quantum) expires, or a higher-priority thread becomes ready (preemption).
  • Waiting -> Ready: The object the thread was waiting on becomes signaled.
  • Running -> Terminated: Thread returns from its start function or calls `ExitThread`.
  • Any State -> Ready (if suspended): When a thread is suspended (`SuspendThread`), it moves to the Ready state but is ineligible to run. `ResumeThread` makes it eligible again.

Programmers can influence thread states using functions like `CreateThread` (with `CREATE_SUSPENDED`), `SuspendThread`, `ResumeThread`, `Sleep`, and the wait functions.

Topic 126: Mistakes while using Threads

Developing correct multithreaded programs requires careful attention to avoid common pitfalls stemming from the asynchronous nature of thread execution.

Key Principles & Common Mistakes:

  • Asynchronous Execution: Threads run independently and can be preempted or resumed at any time. Do not make assumptions about the relative order or timing of execution between threads unless explicit synchronization is used.
  • Initialization Order (Race Conditions): Ensure data required by a child thread is fully initialized by the parent thread *before* the child thread might access it. Creating threads suspended and resuming them only after initialization is complete is one technique. Failure to do this leads to race conditions.
  • Synchronization vs. Priority: Do not confuse thread priority with synchronization. High priority does not guarantee a thread will run before a lower-priority thread if the high-priority thread is blocked or waiting. Synchronization objects (mutexes, events, etc.) are needed to enforce specific execution orders or protect shared resources.
  • Testing Limitations: Testing alone is insufficient to guarantee correctness. Multithreaded bugs (like race conditions or deadlocks) may only appear intermittently depending on timing, system load, number of processors, etc. Careful design, code reviews, and understanding potential concurrency issues are crucial.
  • Stack Size: The default thread stack size (often 1MB) is usually sufficient, but deeply recursive functions or large local variables might require explicitly requesting a larger stack size via `CreateThread` or linker settings.
  • Appropriate Use: Use threads only when tasks are naturally concurrent or when overlapping computation with I/O provides a benefit. Adding threads to purely sequential tasks often just adds complexity and overhead (context switching, synchronization) without performance gains.
  • Resource Consumption: Creating a very large number of threads can consume significant virtual memory (for stacks) and increase context-switching overhead, potentially degrading performance, especially if the number of threads greatly exceeds the number of available processors.
  • Simplicity: Strive for the simplest design that correctly solves the problem. Avoid unnecessary complexity, as it often leads to subtle concurrency bugs.

Topic 127: Timed Waits

Threads often need to pause execution for a specific duration or yield control to other threads.

  • Sleep(): Causes the calling thread to relinquish the remainder of its time slice and enter the **Waiting** state for a specified interval. 
    VOID Sleep(DWORD dwMilliseconds);
    • dwMilliseconds: The time interval to wait, in milliseconds.
      • A positive value: The thread waits for approximately this duration.
      • 0: The thread yields the rest of its time slice to any other thread of equal or higher priority that is ready to run. If no such threads are ready, `Sleep(0)` returns immediately.
      • INFINITE: The thread sleeps indefinitely (generally not useful, equivalent to a permanent wait).
    • After the interval expires, the thread moves to the **Ready** state and waits to be scheduled. The actual time spent sleeping might be longer than requested due to scheduling latencies.
    • Useful for periodic tasks or introducing delays.
  • SwitchToThread(): Yields execution to another thread that is ready to run on the *same processor* as the calling thread. 
    BOOL SwitchToThread(VOID);
    • If another thread is ready on the current processor, the scheduler runs it. The calling thread becomes ready again.
    • If no other threads are ready on the current processor, `SwitchToThread` returns immediately (`FALSE`).
    • Considered a form of "cooperative yielding" among threads on the same core.
    • Returns `TRUE` if execution was yielded, `FALSE` otherwise.

Topic 128: Fibers

A **fiber** is a lightweight unit of execution, even lighter than a thread, that is scheduled manually by the application in **user mode**, rather than preemptively by the OS kernel scheduler.

Key Characteristics:

  • User-Mode Scheduling: The application explicitly controls when fibers run and switch between them using specific API calls (e.g., `SwitchToFiber`). The OS kernel is generally unaware of fibers.
  • Run within Threads: Fibers always run within the context of a thread. A single thread can manage and execute multiple fibers, but only one fiber associated with that thread runs at a time.
  • Independent Stacks: Each fiber has its own stack, allowing it to maintain its own execution context (local variables, function call state).
  • Shared Thread Context: Fibers within the same thread share the thread's resources, including its Thread Local Storage (TLS). Fibers intended to run on different threads should not rely on TLS.
  • Cooperative Multitasking: Since scheduling is manual (non-preemptive), fibers must explicitly yield control (`SwitchToFiber`) to allow other fibers to run.
  • Lightweight: Creating and switching between fibers generally has lower overhead than creating and switching between threads because it doesn't involve kernel transitions.

Fiber Uses:

  • Portability: Can help in porting applications designed around cooperative multitasking or specific user-mode scheduling needs.
  • Resource Pooling / Non-Blocking Waits: Instead of blocking a whole thread on I/O or a lock, a fiber can switch to another fiber while waiting for a resource, allowing the underlying thread to remain active handling other fibers (requires careful management, often with asynchronous I/O).
  • Flexibility: Can run on any thread (one at a time), not bound to a specific thread unless designed that way.
  • Co-routines: Useful for implementing co-routines where control is explicitly passed between related tasks.
  • Performance Claims: Some platforms or libraries utilize fibers claiming performance advantages for specific workloads by avoiding kernel scheduling overhead (e.g., high-throughput network servers).

Topic 129: Fiber APIs

Windows provides a set of APIs for creating and managing fibers:

  1. Enable Fiber Mode for a Thread:
    • A thread must first be converted to operate in "fiber mode" before it can create or switch to fibers.
    • LPVOID ConvertThreadToFiber(LPVOID lpParameter);
    • LPVOID ConvertThreadToFiberEx(LPVOID lpParameter, DWORD dwFlags);
    • This converts the calling thread into a fiber (the "primary fiber"). It returns a pointer representing the fiber address. `lpParameter` becomes the fiber data associated with this primary fiber.
  2. Create New Fibers:
    • Once a thread is in fiber mode, it can create additional fibers.
    • LPVOID CreateFiber(SIZE_T dwStackSize, LPFIBER_START_ROUTINE lpStartAddress, LPVOID lpParameter);
    • LPVOID CreateFiberEx(SIZE_T dwStackCommitSize, SIZE_T dwStackReserveSize, DWORD dwFlags, LPFIBER_START_ROUTINE lpStartAddress, LPVOID lpParameter);
    • Creates a new fiber with its own stack.
    • lpStartAddress: Pointer to the starting function for the new fiber (signature: `VOID WINAPI FiberFunc(LPVOID lpParam)`).
    • lpParameter: Argument passed to the fiber function.
    • Returns a pointer representing the new fiber's address (not a handle).
  3. Switch Execution Between Fibers:
    • Transfers control from the currently running fiber to another fiber associated with the *same thread*.
    • VOID SwitchToFiber(LPVOID lpFiber);
    • lpFiber: The address of the fiber to switch execution to.
    • The context (registers, stack pointer) of the current fiber is saved, and the context of the target fiber is restored. Execution resumes within the target fiber where it last left off (or at its start function if never run before).
  4. Delete Fibers:
    • Releases the resources (like the stack) associated with a fiber that is no longer needed.
    • VOID DeleteFiber(LPVOID lpFiber);
    • lpFiber: The address of the fiber to delete. A fiber cannot delete itself; another fiber within the same thread must delete it.
  5. Access Fiber Data:
    • Fibers can have an associated application-defined data pointer (set during conversion or creation).
    • LPVOID GetFiberData(VOID);: Returns the fiber data pointer associated with the currently running fiber.
    • LPVOID GetCurrentFiber(VOID);: Returns the fiber address of the currently running fiber.

Topic 130: Using Fibers

Fibers enable application-controlled scheduling within a thread. A typical scheme involves:

  1. Convert the primary thread to a fiber using `ConvertThreadToFiber`.
  2. Create multiple worker fibers using `CreateFiber`.
  3. Manage switching between these fibers using `SwitchToFiber` based on application logic.

Two common scheduling policies:

  • Master-Slave Scheduling:
    • One fiber (often the primary one created from the thread) acts as the master scheduler.
    • Worker fibers perform tasks and then always use `SwitchToFiber` to transfer control back to the master fiber.
    • The master fiber then decides which worker fiber to run next. (Fiber 1 in the diagram returning to Primary).
  • Peer-to-Peer Scheduling:
    • Fibers decide amongst themselves which fiber should execute next based on some policy (e.g., round-robin, priority, event completion).
    • A fiber might switch directly to another specific worker fiber. (Fibers 0 and 2 in the diagram switching between each other).

Primary then switches to Fiber 2. Fiber 2 does work and switches back to Fiber 0, which switches to Fiber 2, etc., until eventually, control returns to Primary, and the thread might exit.

Topic 131: Need for Thread Synchronization

Concurrent execution enabled by threads, while powerful, introduces the need for **synchronization** when multiple threads access shared resources (like shared memory variables).

Without synchronization, **race conditions** can occur, where the final outcome depends unpredictably on the exact timing and interleaving of thread execution, often leading to incorrect results.

Example Race Condition:

Consider two threads executing the same code to increment a shared variable `N`, using a temporary variable `M`:

Shared variable: `N` (initially 4)

Thread 1 Code:

M = N;       // M becomes 4
M = M + 1;   // M becomes 5
N = M;       // N becomes 5

Thread 2 Code:

M = N;       // M becomes ?
M = M + 1;   // M becomes ?
N = M;       // N becomes ?
  • Normal (Sequential) Execution: If Thread 1 runs completely, then Thread 2 runs, the final result is N=6 (Thread 1 sets N=5, Thread 2 reads N=5, sets N=6).
  • Concurrent Execution (Race Condition):
    1. Thread 1 executes `M = N;` (M1=4)
    2. Thread 1 executes `M = M + 1;` (M1=5)
    3. Context Switch occurs before Thread 1 executes `N = M;`
    4. Thread 2 executes `M = N;` (M2=4, because N is still 4)
    5. Thread 2 executes `M = M + 1;` (M2=5)
    6. Thread 2 executes `N = M;` (N becomes 5)
    7. Context Switch occurs back to Thread 1
    8. Thread 1 executes its remaining instruction: `N = M;` (N becomes 5, using M1's value)
    In this concurrent scenario, the final value of N is 5, which is incorrect. The update from Thread 1 was effectively lost.

Synchronization mechanisms (like Mutexes, Critical Sections, Semaphores, Interlocked operations) are required to protect the "critical section" (the sequence of operations `M=N; M=M+1; N=M;`) ensuring only one thread can execute it at a time, preventing the race condition.

Topic 133: Volatile Storage

Compiler optimizations can sometimes interfere with correct multithreaded behavior, especially when dealing with shared memory.

Latent Defects from Optimizations:

Compilers often optimize code by keeping frequently accessed variables in CPU registers for faster access, only reading from memory initially and writing back to memory later. In a multithreaded scenario:

  • Thread A reads shared variable `X` into a register.
  • Thread B modifies `X` in memory.
  • Thread A continues working with the stale value of `X` held in its register, unaware of the change made by Thread B in memory.
  • Later, Thread A might write its register value back to memory, overwriting Thread B's update.

Turning off compiler optimizations globally can severely degrade performance.

The `volatile` Qualifier (ANSI C):

ANSI C provides the volatile type qualifier to address this.

volatile DWORD sharedCounter;

Declaring a variable `volatile` instructs the compiler:

  • Every access (read or write) to the `volatile` variable must correspond to an actual memory access (loading from or storing to memory). The compiler should not optimize away these accesses or keep the variable's value solely in a register across sequence points where its value might change externally.
  • The compiler should not reorder accesses to `volatile` variables relative to other `volatile` accesses or specific sequence points, although it might still reorder them relative to non-volatile accesses.

Essentially, `volatile` tells the compiler "the value of this variable might change at any time due to external factors" (like another thread, a hardware register, or a signal handler).

Using `volatile`:

  • Use `volatile` only when necessary, as it inhibits certain optimizations and can degrade performance by forcing memory access.
  • It's typically needed if a variable is modified by multiple threads *and* other threads read that variable, *and* proper synchronization is not already ensuring visibility (though often, synchronization primitives handle memory visibility implicitly).
  • It's also used for memory-mapped hardware registers.
  • Important: `volatile` by itself does **not** guarantee atomicity or solve race conditions related to sequences like read-modify-write. It only ensures that reads and writes happen to/from memory as specified in the code sequence. You still need locks or atomic operations (like Interlocked functions) to protect critical sections involving `volatile` variables if multiple threads modify them.

Topic 134: Memory Architecture and Memory Barriers

Modern multi-core processor architectures introduce complexities related to memory visibility between different processor cores.

Cache Coherency:

  • Each processor core typically has its own private caches (e.g., L1 instruction and data caches). Cores on the same chip might share a larger cache (e.g., L2 or L3). Main memory is shared among all cores/chips.
  • When one core (e.g., Core 0) modifies a variable, the change might initially only exist in its private L1 cache.
  • For another core (e.g., Core 3) to see this change, the updated value must eventually be written back through the cache hierarchy to main memory (or a shared cache), and Core 3's cached copy (if any) must be invalidated or updated.
  • Hardware cache coherency protocols (like MESI) attempt to manage this automatically, but they don't always guarantee the *order* in which changes become visible to other processors.
  • The `volatile` qualifier ensures reads/writes access memory (or at least bypass certain register optimizations) but doesn't, by itself, guarantee cross-processor visibility or ordering.

Memory Barriers (Fences):

To ensure that memory operations happen in a predictable order and that changes made by one processor become visible to others when needed, **memory barriers** (or memory fences) are used.

  • A memory barrier is an instruction or operation that enforces an ordering constraint on memory operations preceding and succeeding it.
  • It typically ensures that all writes *before* the barrier become visible to other processors before any reads or writes *after* the barrier are executed.
  • It also often ensures that reads/writes are not speculatively reordered by the processor or compiler across the barrier.

In Windows:

  • Synchronization primitives like mutexes, critical sections, and semaphores implicitly contain memory barriers within their acquire/release operations. When you lock a mutex, there's usually an "acquire barrier"; when you unlock it, there's a "release barrier," ensuring changes made inside the lock are visible afterwards.
  • Interlocked functions (Topic 135) also act as memory barriers, ensuring atomicity and visibility for the specific operation.
  • Explicit memory barrier functions exist (e.g., `MemoryBarrier` macro, `_ReadWriteBarrier` intrinsic) but require careful understanding of memory models.

Memory barriers are essential for correct lock-free programming and ensuring proper visibility in multithreaded code on multi-core systems, although they are often handled implicitly by higher-level synchronization objects. They ensure that updates propagate correctly through the cache hierarchy and main memory.

Topic 135: Interlocked Functions

When simple atomic operations like incrementing, decrementing, or exchanging shared variables are needed, Windows provides **Interlocked Functions**. These functions guarantee that the operation completes as a single, indivisible (atomic) unit, even in the presence of multiple threads running concurrently on multiple processors.

They are generally simpler and faster than using a full synchronization object (like a critical section or mutex) for these specific operations.

Key Interlocked Functions:

  • InterlockedIncrement() / InterlockedDecrement(): Atomically increments or decrements a 32-bit signed LONG variable. 
    LONG InterlockedIncrement(LONG volatile *Addend);
LONG InterlockedDecrement(LONG volatile *Addend);
    • `Addend`: Pointer to the variable to modify. Must be aligned on a 32-bit boundary. Should be declared `volatile`.
    • Returns the *resulting* incremented/decremented value.
  • InterlockedIncrement64() / InterlockedDecrement64(): 64-bit versions for `LONGLONG` (`__int64`). Require 64-bit alignment.
  • InterlockedExchange(): Atomically sets a 32-bit variable to a new value and returns the *original* value. 
    LONG InterlockedExchange(LONG volatile *Target, LONG Value);
  • InterlockedExchangePointer(): Atomic exchange for pointer-sized variables.
  • InterlockedExchangeAdd(): Atomically adds a value to a 32-bit variable and returns the *original* value. 
    LONG InterlockedExchangeAdd(LONG volatile *Addend, LONG Value);
    • To increment by 2 atomically: `InterlockedExchangeAdd(&N, 2);`
  • InterlockedCompareExchange(): Atomically compares a variable's value with a comparand; if they match, sets the variable to a new value. Returns the *original* value. Fundamental for building lock-free data structures. 
    LONG InterlockedCompareExchange (
    LONG volatile* Destination,
    LONG          Exchange,
    LONG          Comparand
 );
  • Many other variants exist (for Add64, CompareExchange64, CompareExchangePointer, bit tests, etc.).

Important Properties:

  • Atomicity: Ensure the operation (e.g., read-increment-write) completes without interruption from other threads.
  • Memory Barrier: Interlocked functions act as full memory barriers. They guarantee that memory operations preceding the interlocked call are completed and visible before the interlocked operation, and the interlocked operation is completed and visible before subsequent memory operations execute. This ensures correct ordering and visibility across processors.
  • Performance: While faster than mutexes/CS for simple operations, they still incur overhead compared to non-atomic operations due to the required hardware locking and memory barrier effects.

Topic 136: Local and Global Storage

A critical principle for writing correct multithreaded code is distinguishing between data that is local to a thread and data that is shared globally among threads.

Incorrect Use of Global Storage:

Using a global variable to store data that is logically specific to a single thread or a single invocation of a thread function is a common error.

DWORD N; // Global variable

DWORD WINAPI ThreadFunc (TH_ARGS* pArgs) { // Assume TH_ARGS has a 'Count' member
   // ...
   N = 2 * pArgs->Count; // Store thread-specific result in global N
   // ...
   /* Use N; value is specific to this call? */
   UseValue(N);
   // ...
   return some_result;
}

// Main thread creating multiple worker threads
for(int i=0; i < MAX; ++i) {
   //... Allocate and initialize pArgs data ...
   // CreateThread(..., ThreadFunc, pArgs, ...);
   // If N was used here, e.g., N += ThreadFunc(pArgs) in a single-threaded version,
   // directly converting this to multithreaded is problematic.
}

If multiple threads execute `ThreadFunc` concurrently, they will all write to the *same* global variable `N`. The value read by `UseValue(N)` in one thread might have been overwritten by another thread immediately before, leading to incorrect behavior. The final value of `N` after all threads run would be unpredictable.

Guideline:

  • If data is specific to a single thread's execution or a single function call within that thread, store it **locally** (on the thread's stack as a local variable) or pass it via parameters.
  • If data needs to persist across function calls *within the same thread* but remain private to that thread, use **Thread Local Storage (TLS)** (Topic 123).
  • Use **global or static variables** only for data that is intentionally **shared** among multiple threads. Access to shared global/static data *must* be protected using synchronization mechanisms (interlocked functions, critical sections, mutexes, etc.) if multiple threads might modify it concurrently, or if visibility between threads is critical.

Converting single-threaded programs (which might reuse global variables sequentially) to multi-threaded programs requires careful analysis to identify and properly handle shared state.

Topic 137: How to write thread Safe Code

Code is considered **thread-safe** if it functions correctly even when executed concurrently by multiple threads. This means multiple threads can call the same function or access the same resources without causing race conditions, deadlocks, or data corruption.

Guidelines for Thread Safety:

  1. Minimize Shared State: Keep variables local whenever possible. Use thread parameters (passed-in data structures) or Thread Local Storage (TLS) for thread-specific data, rather than global variables.
  2. Protect Shared Resources: If data *must* be shared between threads (global variables, static variables, shared heap objects):
    • Protect access using appropriate synchronization primitives (Critical Sections, Mutexes, Semaphores, Events, Reader/Writer Locks). Ensure locks are acquired before accessing the shared data and released afterwards.
    • Use Interlocked functions for simple atomic operations on shared variables.
    • Use memory barriers where necessary if using lock-free techniques (advanced).
  3. Avoid Race Conditions: Be vigilant about sequences of operations that must appear atomic but aren't inherently (e.g., check-then-act, read-modify-write). Ensure initialization is complete before dependent threads run (e.g., create suspended, initialize, then resume). Use synchronization objects (like events) to signal when conditions are met.
  4. Immutable Data: If shared data is read-only after initialization, no locking is needed for read access. Consider designing with immutable objects where feasible.
  5. Reentrancy: Ensure functions do not rely on static or global state that would be corrupted if the function were called concurrently by multiple threads (or recursively by the same thread if state isn't restored). Use local variables or thread-specific storage.
  6. Isolate Process Environment Changes: Threads should generally avoid changing process-wide settings (like environment variables, current directory, standard I/O handles) unless specifically designed to do so and properly synchronized, as this affects all threads in the process.
  7. Deadlock Prevention: When acquiring multiple locks, always acquire them in the same consistent order across all threads to prevent deadlocks. Avoid holding locks for extended periods or while calling external/blocking code. Consider using `TryEnterCriticalSection` or timed waits (`WaitForSingleObject` with timeout) to detect potential deadlocks.

Topic 138: Thread Synchronization Objects

Windows provides several types of kernel and user-space objects specifically designed for thread synchronization and mutual exclusion.

Synchronization Mechanisms:

  • Wait Functions: `WaitForSingleObject` and `WaitForMultipleObjects` are the core mechanism by which threads wait for synchronization objects to become signaled.
  • File Locks: Used for synchronizing access to file regions between *processes* (and potentially threads within them). See Topic 41.
  • Interlocked Functions: For atomic operations on variables. See Topic 135.

Primary Synchronization Objects:

  1. Critical Sections:
    • Lightweight, user-mode mutex for synchronizing threads *within the same process*.
    • Fast acquisition/release when there is no contention. Spins briefly before yielding if contended.
    • Do not have handles, cannot be named or shared between processes.
    • Managed via `InitializeCriticalSection`, `DeleteCriticalSection`, `EnterCriticalSection`, `LeaveCriticalSection`, `TryEnterCriticalSection`. (See Topic 139).
  2. Mutexes (Mutual Exclusion):
    • Kernel objects, identified by handles.
    • Provide mutually exclusive access to a shared resource. Only one thread can "own" the mutex at a time.
    • Can be named and shared between processes.
    • Support timeouts in wait functions.
    • Handle thread termination ("abandonment") gracefully.
    • Slower than uncontended critical sections due to kernel transition.
    • Managed via `CreateMutex`, `OpenMutex`, `ReleaseMutex`, `WaitForSingleObject`. (See Topic 144).
  3. Semaphores:
    • Kernel objects with handles. Maintain a count between zero and a specified maximum.
    • Used to control access to a resource with a limited number of units (e.g., limiting concurrent accesses to N).
    • Waiting on a semaphore succeeds (and decrements count) if the count > 0. Releasing increments the count.
    • Can be named and shared between processes.
    • Managed via `CreateSemaphore`, `OpenSemaphore`, `ReleaseSemaphore`, `WaitForSingleObject`. (See Topic 147).
  4. Events:
    • Kernel objects with handles. Used to signal the occurrence of an event or condition.
    • Can be auto-reset (signal releases one waiting thread, then automatically resets) or manual-reset (signal releases all waiting threads, stays signaled until explicitly reset).
    • Can be named and shared between processes.
    • Managed via `CreateEvent`, `OpenEvent`, `SetEvent`, `ResetEvent`, `PulseEvent`, `WaitForSingleObject`. (See Topic 150).

Other Synchronization Primitives (Newer Windows Versions):

  • Slim Reader/Writer (SRW) Locks: Lightweight user-mode locks optimized for scenarios where reads are frequent and writes are infrequent. Allow multiple readers concurrently or one exclusive writer.
  • Condition Variables: Used in conjunction with Critical Sections or SRW locks to allow threads to wait efficiently for a specific condition to become true.
  • Waitable Timers: Kernel objects that become signaled at a specific time or periodically.
  • I/O Completion Ports: High-performance mechanism for managing asynchronous I/O operations using a pool of worker threads.

Risks: Using synchronization objects improperly can lead to deadlocks, race conditions, or starvation. Careful design is essential.

Topic 139: CRITICAL_SECTION Objects

Critical Section Concept

A **critical section** is a block of code that accesses a shared resource (like a data structure) and must not be executed concurrently by more than one thread at a time to prevent race conditions.

Windows provides the `CRITICAL_SECTION` structure and associated functions as a simple, efficient mechanism for protecting critical sections *within a single process*.

Properties:

  • User-Mode Object: Initialization and uncontended lock/unlock operations happen primarily in user mode, making them very fast compared to kernel objects like mutexes when there is no contention. If contention occurs, it may transition to kernel mode to wait.
  • No Handles: Critical sections are managed directly via a pointer to the `CRITICAL_SECTION` structure, not via system handles.
  • Intra-Process Only: Cannot be shared or used for synchronization between different processes.
  • Ownership: A critical section is "owned" by the thread that successfully calls `EnterCriticalSection` or `TryEnterCriticalSection`.

API Functions:

  • Initialization / Deletion:
    • VOID InitializeCriticalSection(LPCRITICAL_SECTION lpCriticalSection);: Initializes the structure before use.
    • VOID DeleteCriticalSection(LPCRITICAL_SECTION lpCriticalSection);: Releases system resources associated with the critical section when it's no longer needed.
  • Entering / Leaving:
    • VOID EnterCriticalSection(LPCRITICAL_SECTION lpCriticalSection);: Attempts to acquire ownership (enter) the critical section. If it's already owned by another thread, the calling thread blocks (waits) indefinitely until the owner calls `LeaveCriticalSection`.
    • VOID LeaveCriticalSection(LPCRITICAL_SECTION lpCriticalSection);: Releases ownership of the critical section, allowing one waiting thread (if any) to acquire ownership and proceed. Must be called by the thread that owns the critical section. Each `EnterCriticalSection` call must be matched by a `LeaveCriticalSection` call.
  • Recursive Acquisition: A thread that already owns a critical section can call `EnterCriticalSection` on it again without blocking. Windows maintains a recursion count. The thread must call `LeaveCriticalSection` the same number of times it called `EnterCriticalSection` before another thread can acquire it. This supports recursive functions that need to protect shared data.
  • Non-Blocking Attempt:
    • BOOL TryEnterCriticalSection(LPCRITICAL_SECTION lpCriticalSection);: Attempts to acquire ownership. If successful, returns `TRUE`. If the critical section is already owned by another thread, it returns `FALSE` immediately without blocking.

Timeout Behavior: `EnterCriticalSection` does not have a timeout parameter. A thread blocked on `EnterCriticalSection` will wait forever if the owning thread never calls `LeaveCriticalSection` (e.g., due to a deadlock or crash). This is a key difference from mutexes.

Topic 140: CRITICAL_SECTION for Protecting Shared Variables

Critical sections provide an easy and intuitive way to protect shared variables from concurrent modification by multiple threads within the same process.

Consider a server maintaining shared status counters (number of requests, responses, etc.). Since multiple worker threads might update these counters concurrently, access must be synchronized.

Example Usage Pattern:

// Global shared variables and the critical section object
CRITICAL_SECTION g_csCounterLock;
volatile DWORD g_nRequests = 0;
volatile DWORD g_nResponses = 0;
// ... other shared status variables ...

// Initialization (called once at startup)
InitializeCriticalSection(&g_csCounterLock);

// Worker Thread Function Example
DWORD WINAPI WorkerThreadFunc(LPVOID pParam) {
    // ... do some work ...

    // Update shared counters within the critical section
    EnterCriticalSection(&g_csCounterLock);
    // --- Critical Section Start ---
    g_nRequests++;
    // Maybe read/update g_nResponses or other status
    // --- Critical Section End ---
    LeaveCriticalSection(&g_csCounterLock);

    // ... maybe more work ...
    return 0;
}

// Cleanup (called once at shutdown)
DeleteCriticalSection(&g_csCounterLock);

Using EnterCriticalSection before accessing the shared counters and LeaveCriticalSection afterwards ensures that only one thread can modify `g_nRequests` and `g_nResponses` at any given time, preventing race conditions.

The OCR also mentions using an intermediate variable within the critical section (like the M=N; M=M+1; N=M; example). While technically correct if protected by a CS, it's often unnecessary. Direct modification (like `g_nRequests++`) is usually fine within the CS. However, the intermediate variable example *does* emphasize that the *entire sequence* that needs to be atomic must be inside the `Enter`/`Leave` block.

Topic 141: Protect a Variable with a Single Synchronization Object

A fundamental guideline for using synchronization objects (like Critical Sections or Mutexes) to protect shared data is:

All shared variables that need to be accessed together atomically (i.e., within the same critical section or logical operation) must be protected by the *same*, single synchronization object across all threads that access them.

Using different synchronization objects to protect different accesses to the *same* shared variable, or using different objects for related variables within the same logical operation, will fail to provide proper mutual exclusion and can lead to race conditions or incorrect results.

Incorrect Example Mentioned in OCR:

The OCR describes an incorrect scenario where two different synchronization objects (e.g., `cs1` and `cs2`) might be used to protect access to the same variable `N` in different parts of the code or by different threads.

// Example of INCORRECT synchronization
CRITICAL_SECTION cs1, cs2;
volatile LONG N = 0;

// Thread A might do:
EnterCriticalSection(&cs1);
N += 5;
LeaveCriticalSection(&cs1);

// Thread B might do:
EnterCriticalSection(&cs2); // Uses a DIFFERENT CS!
LONG temp = InterlockedDecrement(&N); // Interlocked is atomic itself...
LeaveCriticalSection(&cs2);

// Thread C might do:
EnterCriticalSection(&cs2); // Also uses cs2
N -= 5;
LeaveCriticalSection(&cs2);

This code is flawed because Thread A's modification using `cs1` is not mutually exclusive with Thread C's modification using `cs2`. They could potentially modify `N` concurrently, leading to incorrect results. Even though Thread B uses an atomic `InterlockedDecrement`, its surrounding critical section `cs2` doesn't coordinate with `cs1` used by Thread A.

Correct Approach: All threads accessing `N` (or a group of related variables that must be consistent) must use the *same* lock (e.g., always use `cs1`).

Topic 142: Producer-Consumer Problem

The **Producer-Consumer problem** is a classic concurrency problem used to illustrate synchronization challenges. It involves two types of processes or threads:

  • Producers: Generate data items (messages, tasks) and place them into a shared buffer or queue.
  • Consumers: Retrieve data items from the shared buffer/queue and process them.

The challenge lies in coordinating access to the shared buffer:

  • Producers shouldn't add data if the buffer is full.
  • Consumers shouldn't remove data if the buffer is empty.
  • Producers and consumers must not access the buffer simultaneously to avoid data corruption.

The simplistic version described here focuses on a single shared message block (not a queue) and highlights mutual exclusion and signaling:

  • A single shared `MSG_BLOCK` structure holds the data.
  • A Producer thread periodically creates a new message, calculates a checksum, and places it in the `MSG_BLOCK`.
  • A Consumer thread, when prompted by the user, reads the *most recent complete* message from the `MSG_BLOCK`, validates the checksum, and displays it.
  • Lost Messages: Consumers only read the latest message; if the producer creates several messages before the consumer reads, intermediate messages are lost.
  • Mutual Exclusion: Access to the `MSG_BLOCK` must be protected (e.g., using a Critical Section or Mutex) to prevent the consumer from reading while the producer is writing, or vice-versa.
  • Signaling: A mechanism (like an Event object) might be needed to signal the consumer when a new message is ready.
  • Invariant: An invariant property (a condition always true outside the critical section) is maintained, such as the checksum always matching the current message content when the block is marked as "ready".

Topic 143: Sample Program for Producer-Consumer Problem (`simplePC.c`)

This topic outlines the structure and logic of the `simplePC.c` example program, which implements the single-message-block producer-consumer scenario described in Topic 142 using Critical Sections for mutual exclusion.

Data Structure (`MSG_BLOCK`):

typedef struct MSG_BLOCK_TAG {
    CRITICAL_SECTION mGuard;       /* Guard the message block structure */
    DWORD            fReady;       /* 0=Not Ready/Empty, 1=Ready/Contains valid message */
    DWORD            fStop;        /* Flag to signal threads to stop */
    volatile DWORD nCons;        /* Number of messages consumed */
    volatile DWORD mSequence;    /* Sequence number of last message produced */
    DWORD            nLost;        /* Calculated number of lost messages */
    time_t           mTimestamp;   /* Timestamp of message creation */
    DWORD            mChecksum;    /* Checksum of mData */
    DWORD            mData[DATA_SIZE]; /* Message Contents */
} MSG_BLOCK;

// Global instance
MSG_BLOCK mBlock = { {0}, 0, 0, 0, 0, 0, 0, 0, {0} }; // Requires InitializeCriticalSection

Main Thread (`_tmain`):

  1. Initializes the critical section: `InitializeCriticalSection(&mBlock.mGuard);`
  2. Creates the Producer thread (`Produce`) using `_beginthreadex`.
  3. Creates the Consumer thread (`Consume`) using `_beginthreadex`.
  4. Waits for both threads to complete using `WaitForSingleObject`.
  5. Deletes the critical section: `DeleteCriticalSection(&mBlock.mGuard);`
  6. Prints final statistics (Produced, Consumed, Lost).

Producer Thread (`Produce`):

  1. Seeds random number generator.
  2. Enters a loop `while (!mBlock.fStop)`.
  3. Sleeps for a random delay.
  4. Enters the critical section: `EnterCriticalSection(&mBlock.mGuard);`
  5. Uses `__try`/`__finally` to ensure the critical section is always left.
  6. Inside `__try`: If `!mBlock.fStop`:
    • Sets `mBlock.fReady = 0;` (Marks buffer as being updated).
    • Calls `MessageFill(&mBlock);` to generate data, timestamp, and checksum.
    • Sets `mBlock.fReady = 1;` (Marks buffer as ready).
    • Atomically increments `mBlock.mSequence` using `InterlockedIncrement`.
  7. Inside `__finally`: `LeaveCriticalSection(&mBlock.mGuard);`
  8. Loop continues until `fStop` is set by the consumer.

Consumer Thread (`Consume`):

  1. Enters a loop `while (!mBlock.fStop)`.
  2. Prompts user: "Enter 'c' for Consume; 's' to stop: ".
  3. Reads user command using `_tscanf`.
  4. If command is 's': Sets `mBlock.fStop = 1;` (Signals producer and itself to stop).
  5. If command is 'c':
    • Enters the critical section: `EnterCriticalSection(&mBlock.mGuard);`
    • Uses `__try`/`__finally` to ensure the critical section is always left.
    • Inside `__try`:
      • If `mBlock.fReady == 0`: Prints "No new messages".
      • Else (`mBlock.fReady == 1`):
        • Calls `MessageDisplay(&mBlock);` to display message and validate checksum.
        • Calculates lost messages: `mBlock.nLost = mBlock.mSequence - mBlock.nCons + 1;` (Slightly off, should likely be `mSequence - nCons`).
        • Marks buffer as consumed: `mBlock.fReady = 0;`
        • Atomically increments `mBlock.nCons` using `InterlockedIncrement`.
    • Inside `__finally`: `LeaveCriticalSection(&mBlock.mGuard);`
  6. Handles illegal commands.

Helper Functions:

  • MessageFill(): Fills `mData` with random data, calculates checksum, sets timestamp. Assumes caller holds the lock.
  • MessageDisplay(): Calculates checksum of current `mData`, compares with stored checksum, prints message details and validation result. Assumes caller holds the lock.

Topic 144: Mutexes

A **Mutex** (Mutual Exclusion) is a kernel synchronization object used to enforce exclusive access to a shared resource.

Advantages over Critical Sections:

  • Named and Sharable: Mutexes can be given names, allowing them to be shared and used for synchronization between threads in *different processes*. Critical sections are restricted to a single process.
  • Have Handles: Being kernel objects, mutexes are accessed via `HANDLE`s, allowing them to be used with wait functions (`WaitForSingleObject`, `WaitForMultipleObjects`).
  • Timeouts: Wait functions used with mutexes allow specifying a timeout period, preventing indefinite blocking if the mutex cannot be acquired. Critical sections do not have timeouts.
  • Abandonment Signaling: If a thread terminates while owning a mutex (without releasing it), the mutex becomes "abandoned". The wait functions return a specific code (`WAIT_ABANDONED_0`) to waiting threads, signaling that the protected resource might be in an inconsistent state. Critical sections offer no such notification.

Disadvantages compared to Critical Sections:

  • Performance: Acquiring and releasing a mutex always involves a transition to kernel mode, making it significantly slower than acquiring/releasing an *uncontended* critical section (which stays in user mode).

Ownership and Recursion:

  • A mutex is "owned" by the thread that successfully waits on its handle.
  • Only the owning thread can call `ReleaseMutex`.
  • Like critical sections, mutexes support recursion: a thread that already owns a mutex can wait on it again successfully without blocking. A recursion count is maintained, and the thread must call `ReleaseMutex` the same number of times it acquired ownership before another thread can acquire it.

API Functions:

  • HANDLE CreateMutex(LPSECURITY_ATTRIBUTES lpMutexAttributes, BOOL bInitialOwner, LPCTSTR lpName);
    • Creates or opens a mutex.
    • `lpMutexAttributes`: Optional security attributes.
    • `bInitialOwner`: If `TRUE`, the calling thread immediately owns the mutex upon creation. Ignored if opening an existing named mutex.
    • `lpName`: Optional name for sharing between processes. If `NULL`, the mutex is unnamed. If a mutex with this name already exists, returns a handle to the existing mutex and `GetLastError` returns `ERROR_ALREADY_EXISTS`.
    • Returns a handle on success, `NULL` on failure.
  • HANDLE OpenMutex(DWORD dwDesiredAccess, BOOL bInheritHandle, LPCTSTR lpName);
    • Opens an existing named mutex created by another process.
  • BOOL ReleaseMutex(HANDLE hMutex);
    • Releases ownership of the mutex. Must be called by the owning thread.
    • Returns `TRUE` on success, `FALSE` on failure (e.g., if the calling thread doesn't own the mutex).
  • Waiting: Use `WaitForSingleObject(hMutex, dwMilliseconds)` or `WaitForMultipleObjects` to acquire ownership.

Topic 145: Mutexes, CRITICAL_SECTIONS and Deadlocks

Both Mutexes and Critical Sections are used for concurrency control, but their misuse can lead to **deadlocks**.

A deadlock occurs when two or more threads are blocked indefinitely, each waiting for a resource held by the other.

Example Deadlock Scenario:

Consider two shared lists, List A and List B, protected by separate mutexes, `mutexA` and `mutexB`.

  • Operation 1 Requirement: An element must be present in both lists or neither.
  • Operation 2 Requirement: An element present in one list cannot be in the other.

Using a single mutex for both lists would serialize all access, potentially degrading performance if updates to A and B are often independent. Using separate mutexes (`mutexA`, `mutexB`) allows concurrent updates but introduces the risk of deadlock:

  • Thread 1 (needs both lists):
    1. Acquires `mutexA`.
    2. Tries to acquire `mutexB` (blocks if Thread 2 holds it).
  • Thread 2 (needs both lists):
    1. Acquires `mutexB`.
    2. Tries to acquire `mutexA` (blocks if Thread 1 holds it).

If Thread 1 acquires `mutexA` and Thread 2 acquires `mutexB` concurrently, both threads will block forever when they try to acquire the second mutex – a classic deadlock.

Topic 146: Mutexes and CS (Deadlock Avoidance)

Techniques to avoid deadlocks when using multiple synchronization objects like Mutexes or Critical Sections:

  1. Lock Hierarchy / Ordering: Define a strict, arbitrary order for acquiring multiple locks. All threads *must* acquire the locks in this predefined order. In the previous example, if the rule is "always acquire `mutexA` before `mutexB`", then Thread 2 would block trying to get `mutexA` first, preventing the deadlock cycle. This is the most common and robust technique. Threads should also release locks, usually in the reverse order of acquisition.
  2. `WaitForMultipleObjects` (Mutexes Only): Wait for *all* required mutexes simultaneously using `WaitForMultipleObjects` with the `bWaitAll` flag set to `TRUE`. The function only returns successfully (granting ownership of all mutexes) when all of them can be acquired atomically relative to other waits. If any one cannot be acquired, the function waits until all become available. This technique avoids deadlock but can reduce concurrency as a thread needs exclusive access to *all* required resources at once. **This technique is not possible with Critical Sections** because they don't have handles and cannot be used with wait functions.
  3. `TryEnterCriticalSection` / Timed Waits: Use non-blocking attempts (`TryEnterCriticalSection`) or timed waits (`WaitForSingleObject` with a timeout for mutexes). If acquiring a second lock fails or times out, release the first lock, wait or perform other work, and then retry acquiring both locks later. This is more complex to implement correctly and can lead to livelock or fairness issues.

Maintaining a consistent lock acquisition order is generally the preferred method for deadlock avoidance.

Topic 147: Semaphores

A **Semaphore** is a kernel synchronization object that maintains a counter (an integer value) between zero and a specified maximum value. It's used to control access to a shared resource that has a limited number of units available.

Semaphore Count and State:

  • The semaphore's count represents the number of available units of the resource.
  • A semaphore is **signaled** when its count is greater than 0.
  • A semaphore is **non-signaled** when its count is 0.

Operations:

  • Waiting (`WaitForSingleObject`/`Multiple`): A thread "waits" on a semaphore to acquire a unit of the resource. If the semaphore's count is greater than 0 (signaled), the wait succeeds immediately, and the semaphore's count is **decremented** by one. If the count is 0 (non-signaled), the thread blocks until another thread releases the semaphore.
  • Releasing (`ReleaseSemaphore`): A thread "releases" the semaphore (signals it) when it is finished with a unit of the resource. This **increments** the semaphore's count by a specified amount, potentially allowing a waiting thread to proceed.

API Functions:

  • HANDLE CreateSemaphore(LPSECURITY_ATTRIBUTES lpSemaphoreAttributes, LONG lInitialCount, LONG lMaximumCount, LPCTSTR lpName);
    • Creates or opens a semaphore.
    • `lInitialCount`: The starting count (0 <= Initial <= Maximum).
    • `lMaximumCount`: The maximum value the count can reach (must be >= 1).
    • `lpName`: Optional name for inter-process sharing.
    • Returns a handle or `NULL`.
  • HANDLE CreateSemaphoreEx(...): (Newer version with more options).
  • HANDLE OpenSemaphore(...): Opens an existing named semaphore.
  • BOOL ReleaseSemaphore(HANDLE hSemaphore, LONG lReleaseCount, LPLONG lpPreviousCount);
    • Increments the semaphore's count by `lReleaseCount` (must be > 0).
    • Fails if the increment would exceed `lMaximumCount`.
    • `lpPreviousCount`: Optional pointer to receive the count *before* the release.
    • Returns `TRUE` on success, `FALSE` on failure.
    • **Important:** Any thread can call `ReleaseSemaphore`, not just one that successfully waited on it. There is no concept of ownership or abandonment like with mutexes.

Topic 148: Using Semaphores

Semaphores are versatile synchronization primitives.

Semaphore Concepts & Use Cases:

  • Resource Counting: The classic use is to model a resource pool with N available units (e.g., N database connections, N buffer slots). Initialize the semaphore with `lInitialCount = N`, `lMaximumCount = N`. Threads wait on the semaphore before using a resource unit and release the semaphore afterwards. The count tracks available units.
  • Producer-Consumer Queue: Can be used in pairs for bounded buffers:
    • `emptyCount` semaphore: Initialized to buffer size N. Producers wait on this before adding an item (decrements). Consumers release this after removing an item (increments).
    • `fillCount` semaphore: Initialized to 0. Consumers wait on this before removing an item (decrements). Producers release this after adding an item (increments).
    • A mutex or critical section is still needed to protect the actual buffer data structure during insertion/removal.
  • Thread Creation Synchronization: Can solve the problem of needing to create multiple worker threads but ensuring they don't start work until all are created (alternative to creating suspended).
    1. Create a semaphore (`hGoSignal`) initialized with `lInitialCount = 0`.
    2. The main (boss) thread creates each worker thread.
    3. Each worker thread's first action is `WaitForSingleObject(hGoSignal, INFINITE);`.
    4. After the boss thread has created all N worker threads, it calls `ReleaseSemaphore(hGoSignal, N, NULL);` to increment the count to N.
    5. This releases all N waiting worker threads simultaneously to begin their work.

Topic 149: Semaphore Limitation

While versatile, Windows semaphores have a limitation regarding atomic waits for multiple units.

Problem: Waiting for Count > 1 Atomically

How can a thread wait until the semaphore count is at least, say, 2, and then decrement the count by 2 atomically?

Calling `WaitForSingleObject` twice is **not atomic**:

/* INCORRECT attempt to wait for 2 units */
WaitForSingleObject(hSem, INFINITE); // Decrements count by 1
// *** CONTEXT SWITCH CAN HAPPEN HERE ***
// Another thread might run and decrement the count before the second wait executes.
WaitForSingleObject(hSem, INFINITE); // Might block if count became 0

If a context switch occurs between the two waits, and another thread consumes the last unit, the first thread might deadlock waiting for the second unit, even if the count was initially >= 2.

Workaround / Solution:

To make the two waits atomic, they must be protected by another synchronization mechanism, typically a Critical Section or Mutex:

/* CORRECT way using a Critical Section */
EnterCriticalSection(&csSemWait); // Protect the multi-wait sequence
WaitForSingleObject(hSem, INFINITE);
WaitForSingleObject(hSem, INFINITE);
LeaveCriticalSection(&csSemWait);

// ... Perform work requiring 2 units ...

ReleaseSemaphore(hSem, 2, NULL); // Release both units at once

This ensures that once a thread enters the critical section, it successfully acquires both units from the semaphore before another thread can interfere.

Alternative (Flawed):

Using `WaitForMultipleObjects` with an array containing two handles *to the same semaphore* fails because the function detects duplicate handles in the input array and returns `WAIT_FAILED`.

Topic 150: Events

An **Event** is a kernel synchronization object primarily used for signaling. Threads can wait for an event object to be set to the **signaled** state by another thread.

Events are useful for notifying threads that a specific condition has occurred (e.g., data is available, an operation is complete, it's time to shut down).

Types of Events:

Events are created as either manual-reset or auto-reset:

  1. Auto-Reset Event:
    • When signaled (by `SetEvent` or `PulseEvent`), it releases **exactly one** waiting thread (if any).
    • After releasing one thread, the event **automatically** resets itself to the non-signaled state.
    • If signaled when no threads are waiting, it remains signaled until one thread waits on it, releases that thread, and then immediately resets.
    • Acts like a gate that lets only one thread through each time it's opened.
  2. Manual-Reset Event:
    • When signaled (by `SetEvent`), it releases **all** currently waiting threads.
    • It **remains signaled** until explicitly reset to the non-signaled state by a call to `ResetEvent`.
    • While signaled, any thread that waits on it will pass through immediately without blocking.
    • Acts like a gate that stays open (letting everyone through) until manually closed.

API Functions:

  • HANDLE CreateEvent(LPSECURITY_ATTRIBUTES lpEventAttributes, BOOL bManualReset, BOOL bInitialState, LPCTSTR lpName);
    • Creates or opens an event object.
    • `bManualReset`: `TRUE` for manual-reset, `FALSE` for auto-reset.
    • `bInitialState`: `TRUE` to create the event initially signaled, `FALSE` for non-signaled.
    • `lpName`: Optional name for inter-process sharing.
  • HANDLE CreateEventEx(...): Newer version with more options.
  • HANDLE OpenEvent(...): Opens an existing named event.
  • BOOL SetEvent(HANDLE hEvent);
    • Sets the event to the signaled state.
    • Releases one waiting thread (auto-reset) or all waiting threads (manual-reset).
  • BOOL ResetEvent(HANDLE hEvent);
    • Sets the event to the non-signaled state. Required only for manual-reset events (auto-reset events reset automatically).
  • BOOL PulseEvent(HANDLE hEvent);
    • Sets the event to signaled, releases waiting threads (one for auto-reset, all for manual-reset), and then *immediately* resets the event to non-signaled.
    • **Caution:** `PulseEvent` is unreliable and generally discouraged. Race conditions can occur where threads start waiting *after* the pulse begins but *before* it completes, causing them to miss the signal. Use `SetEvent` and explicit synchronization (like semaphores or condition variables) instead for complex signaling.
  • Waiting: Use `WaitForSingleObject` or `WaitForMultipleObjects`.

Topic 151: Event Usage Models

Events can be used in various ways depending on whether they are manual or auto-reset and whether `SetEvent` or `PulseEvent` is used.

Signaling Function Auto-Reset Event Behavior Manual-Reset Event Behavior
SetEvent Exactly one waiting thread is released (if any). Event automatically resets. If no thread is waiting, stays signaled until one waits, then releases it and resets. All currently waiting threads are released. The event remains signaled until `ResetEvent` is called by some thread.
PulseEvent
(Use with caution)		 Releases exactly one waiting thread (if any), then automatically resets. Unreliable if threads might start waiting during the pulse. Releases all currently waiting threads, then automatically resets. Unreliable if threads might start waiting during the pulse.

Understanding Events Conceptually:

  • Auto-Reset Event: Like a turnstile or a self-closing gate. `SetEvent` opens it briefly, letting exactly one waiting thread through before closing again.
  • Manual-Reset Event: Like a regular gate or door. `SetEvent` opens it. It stays open, letting any waiting threads (and any new threads that arrive) through, until someone explicitly calls `ResetEvent` to close it. `PulseEvent` briefly opens and closes it.

Topic 152: Producer-Consumer Solution Using Events

This topic describes an alternative solution to the single-message-block producer-consumer problem, this time using a combination of a Mutex and an Event object.

Approach:

  • Shared Data (`MSG_BLOCK`): The same structure as in Topic 143, but the `mGuard` member is now a `HANDLE` for a Mutex, and an additional `HANDLE mReady` for an Event is included. 
    typedef struct MSG_BLOCK_TAG {
    HANDLE mGuard;       /* Mutex to guard the message block structure */
    HANDLE mReady;       /* "Message ready" auto-reset event */
    DWORD  fReady, fStop;/* ready state flag, stop flag */
    // ... other members like mSequence, nCons, mData, etc. ...
} MSG_BLOCK;
  • Initialization (`_tmain`):
    • Create a Mutex using `CreateMutex` (initially unowned) and store handle in `mBlock.mGuard`.
    • Create an **auto-reset** Event using `CreateEvent` (initially non-signaled, `bInitialState=FALSE`, `bManualReset=FALSE`) and store handle in `mBlock.mReady`.
    • Create Producer and Consumer threads.
    • Wait for threads to complete.
    • Close Mutex and Event handles using `CloseHandle`.
  • Producer Thread (`Produce`):
    1. Loop `while (!mBlock.fStop)`.
    2. Sleep for random delay.
    3. Wait to acquire the mutex: `WaitForSingleObject(mBlock.mGuard, INFINITE);`
    4. Use `__try`/`__finally` to ensure mutex release.
    5. Inside `__try` (if `!fStop`):
      • Set `mBlock.fReady = 0;`
      • Call `MessageFill(&mBlock);`
      • Set `mBlock.fReady = 1;`
      • Increment sequence number.
      • **Signal Consumer:** Call `SetEvent(mBlock.mReady);` to signal that a message is ready.
    6. Inside `__finally`: `ReleaseMutex(mBlock.mGuard);`
  • Consumer Thread (`Consume`):
    1. Loop `while (!ShutDown)`.
    2. Prompt user ('c' or 's').
    3. If 's':
      • Acquire mutex `mGuard`.
      • Set `mBlock.fStop = 1; ShutDown = 1;`
      • Release mutex `mGuard`.
    4. If 'c':
      • **Wait for Signal:** Wait for the event indicating a message is ready: `WaitForSingleObject(mBlock.mReady, INFINITE);`. Since it's auto-reset, this wait consumes the signal.
      • **Acquire Mutex:** Wait to acquire the mutex: `WaitForSingleObject(mBlock.mGuard, INFINITE);`
      • Use `__try`/`__finally` to ensure mutex release.
      • Inside `__try`:
        • Check `if (!mBlock.fReady) _leave;` (This handles a potential race condition - although unlikely here - where the event was signaled but the producer hasn't finished/released mutex yet, or a spurious wakeup. Ensures message isn't processed twice).
        • Call `MessageDisplay(&mBlock);`
        • Update `nCons`, `nLost`.
        • Set `mBlock.fReady = 0;`
      • Inside `__finally`: `ReleaseMutex(mBlock.mGuard);`

In this version, the Mutex (`mGuard`) ensures exclusive access to the data structure, while the auto-reset Event (`mReady`) provides efficient signaling, waking up the consumer only when a new message has actually been produced and is ready.
